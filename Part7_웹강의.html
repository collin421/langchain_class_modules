LangChain 에이전트 마스터 · Part 7

도구(Tool) 정의

에이전트의 힘은 도구에서 나옴. 도구가 없으면 판단만 할 뿐 행동을 못 함. 핵심은 놀랍도록 단순함 — 함수의 이름·docstring·타입 힌트가 그대로 모델에게 주는 사용 설명서가 됨.

🟡 중급 · 7/18 @tool 예상 40분
7-1 · 7-2

함수를 도구로 — @tool

모델(LLM)은 뛰어난 두뇌지만 몸이 없음 — 직접 검색하거나 계산할 수 없음. 도구는 그 두뇌에 다는 손과 발임. 평범한 함수 위에 @tool 한 줄을 붙이면 도구가 됨.

인터랙티브 · 함수에서 명세가 추출되는 과정

코드의 하이라이트된 부분을 누르면, 그것이 모델에게 전달되는 명세의 어디로 가는지 보여줌.

@tool
def get_stock(product: str) -> str:
  """상품명을 받아 현재 재고 수량을 돌려준다."""
  # ... 내부 코드 (모델은 보지 않음)
↓ @tool 이 자동 추출
nameget_stock
설명상품명을 받아 현재 재고 수량을 돌려준다.
입력product (str)
위 코드의 색칠된 부분을 눌러보세요.
📌 참고  모델은 함수 내부 코드를 보지 않음. 오직 이름·설명·입력 명세만 봄. 반환값은 모델이 읽어야 하니 보통 문자열로 둠.
3줄 요약
  1. 도구는 모델(두뇌)에 다는 손과 발임.
  2. @tool 한 줄이 함수를 도구로 만듦.
  3. 이름·docstring·타입이 자동 추출돼 모델용 명세가 됨.
7-3 · 핵심

docstring이 도구 선택을 좌우한다

이 파트에서 가장 중요한 사실 — 모델은 docstring만 읽고 도구를 고름. docstring은 공구함에 붙은 라벨임. 라벨이 명확하면 제대로, 모호하면 엉뚱한 공구를 집음.

인터랙티브 · 같은 함수, 다른 docstring

내부 코드는 동일한 환율 도구임. docstring만 바꿔, 모델이 "1달러는 몇 원?" 질문에 이 도구를 고르는지 봄.

❓ 사용자: "1달러는 몇 원이야?"
📌 참고  docstring에 "언제 쓰는지"를 적는 것이 특히 강력함. "환율을 조회한다"보다 "해외 가격을 원화로 환산할 때 사용한다"가 모델 판단을 크게 도움. docstring은 모델에게 보내는 프롬프트의 일부임.
3줄 요약
  1. 모델은 docstring·시그니처만 보고 도구를 고름(내부 코드 아님).
  2. "무엇을·언제" 쓰는지 명확히 적는 것이 핵심임.
  3. 도구 오작동의 1순위 원인은 부실한 docstring임.
선택의 원리

여러 도구 중 올바른 것 고르기

도구가 여럿이면 모델은 각 docstring을 보고 질문에 맞는 것을 고름. 질문을 바꿔가며 모델이 어떤 도구를 집는지 봄.

인터랙티브 · 질문 → 도구 라우팅

질문을 눌러, 모델이 세 도구 중 무엇을 고르는지 관찰함.

get_stock
재고 조회
get_exchange_rate
환율 조회
calculator
계산
3줄 요약
  1. 도구가 여럿이면 모델이 docstring을 보고 상황에 맞게 고름.
  2. 도구가 필요 없으면 아무것도 안 고르고 바로 답함.
  3. 도구 설명이 겹치거나 모호하면 잘못 고름.
7-4

args_schema — 입력 구조를 정밀하게

인수·타입 힌트도 명세의 일부임. 복잡하거나 검증이 필요하면 Pydantic으로 입력 스키마를 정의함(args_schema). 인수 설명·제약(gt=0 등)까지 모델에 전달됨.

인터랙티브 · 입력 검증 체험

search_products(category: str, max_price: int > 0). 값을 넣어 스키마 검증이 어떻게 작동하는지 봄.

category
max_price
📌 참고  인수가 1~2개로 단순하면 타입 힌트 + docstring Args로 충분함. 인수가 많거나 검증(범위·필수)이 필요하면 args_schema(Pydantic)를 씀 — Part 4 구조화 출력과 같은 발상임.
3줄 요약
  1. 인수·타입 힌트도 모델에 전달되는 명세임.
  2. 단순하면 타입 힌트+Args, 복잡하면 args_schema.
  3. 설명·제약을 명확히 하면 호출 형식 오류가 줄어듦.
7-5

좋은 도구 설계 — 헛돌지 않게

에이전트가 헛도는(엉뚱한 호출·무한 반복) 원인의 다수는 도구 설계 문제임. 두 도구를 체크리스트로 진단해봄.

인터랙티브 · 도구 설계 진단
📌 참고  도구가 많아지면(10개+) 모델이 고르기 어려워짐. 그땐 묶거나, 단계를 나누거나, 여러 전문 에이전트로 분리함(멀티에이전트·Part 15). 되돌릴 수 없는 행동(삭제·발송·결제)은 human-in-the-loop(Part 14)로 보호함.
3줄 요약
  1. 좋은 도구는 단일 기능·명확한 이름·'언제' 명시·친화적 반환값임.
  2. 에이전트가 헛도는 원인의 다수는 도구 설계 문제임.
  3. 도구는 적고 명확하게 — 많아지면 분리·멀티에이전트를 고려함.
