> **이 파트의 목표**

---

## 8-1. RAG란 무엇인가 — 찾아본 다음 답하기

### 개념

**RAG(Retrieval-Augmented Generation, 검색 증강 생성)**는 이름 그대로 **"검색으로 보강한 생성"**임. 모델이 그냥 기억으로 답하는 대신, **① 질문과 관련된 문서를 검색하고 → ② 그 문서를 근거로 답을 생성**하는 2단계 방식임.

모델은 두 가지를 모름.
- **사적인 지식**: 사내 매뉴얼, 우리 회사 상품 정책, 고객 DB
- **최신 정보**: 학습 시점 이후의 사건, 오늘의 재고

RAG는 이 빈틈을 "답하기 전에 관련 자료를 찾아 손에 쥐여주는" 방식으로 메움.

### 비유

RAG는 **오픈북 시험**과 같음. 학생(모델)이 모든 걸 외울 필요 없이, 문제를 보고 **교과서에서 관련 페이지를 펼쳐(검색)** 그 내용을 근거로 답을 씀(생성). 외운 것에만 의존하면 모르는 건 지어내지만(환각), 책을 펴서 확인하면 정확한 근거로 답할 수 있음.

### 원리

RAG의 흐름은 이렇게 나뉨.

```
[준비 단계 — 미리 한 번]
문서 → 잘게 자르기(청킹) → 의미 벡터로 변환(임베딩) → 벡터스토어에 저장

[질문 단계 — 매번]
질문 → 의미 벡터로 변환 → 벡터스토어에서 "의미가 가까운 조각" 검색(retriever)
     → 찾은 조각 + 질문을 프롬프트에 넣어 → 모델이 근거 기반으로 답
```

핵심은 **"질문과 의미가 가까운 문서 조각을 찾는다"**임. 단순 키워드 일치가 아니라 **의미**로 찾는 것이 RAG의 힘이며, 그 비결이 임베딩(8-3)임.

### 📌 참고
> RAG는 모델을 재학습(파인튜닝)하지 않고도 새 지식을 줄 수 있다는 점이 큰 장점임. 문서만 갈아 끼우면 답의 근거가 바뀜. 그래서 "최신 자료 반영"이 잦은 실무에서 RAG가 표준처럼 쓰임.

### 실무 포인트
- 모델이 "모르는 것을 지어내는" 환각을 줄이는 가장 현실적인 방법이 RAG임.
- 리리마켓이라면 "반품 정책", "상품 소재 정보", "사이즈 가이드" 문서를 RAG로 붙여, 고객 문의에 정확히 답하는 봇을 만들 수 있음.

### 3줄 요약
1. RAG는 "관련 문서를 검색한 뒤 그것을 근거로 답하는" 2단계 방식임.
2. 모델이 모르는 사적·최신 지식의 빈틈을 메우고 환각을 줄임.
3. 핵심은 키워드가 아니라 "의미가 가까운" 문서를 찾는 것임.

## 8-2. 문서 로더 & 청킹 — 불러와서 자르기

### 개념

RAG의 첫 단계는 문서를 **불러오고(로딩)**, 검색하기 좋은 크기로 **자르는(청킹)** 것임. 문서가 너무 길면 통째로는 다루기 어렵고, 너무 잘게 자르면 맥락이 끊김. 적당한 크기의 **조각(chunk)**으로 나눔.

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

text = "리리마켓 반품 정책 ... (긴 문서) ..."

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 한 조각의 최대 글자 수
    chunk_overlap=50,      # 조각 간 겹치는 글자 수
)
chunks = splitter.split_text(text)
```

### 문법 핵심

| 인수 | 의미 |
|---|---|
| `chunk_size` | 한 조각의 최대 크기(글자 수). 보통 300~1000 |
| `chunk_overlap` | 이웃한 조각이 겹치는 양. 문맥이 경계에서 끊기는 것을 완화 |
| `.split_text(text)` | 문자열을 조각 리스트로 자름 |
| `.split_documents(docs)` | 로더가 만든 Document 객체들을 자름 |

### 비유

긴 문서를 청킹하는 것은 **두꺼운 책에 색인용 포스트잇을 붙여 나누는 것**과 같음. 너무 크게 나누면(한 챕터씩) 원하는 내용을 콕 집기 어렵고, 너무 잘게 나누면(한 문장씩) 앞뒤 맥락이 사라짐. `chunk_overlap`은 **포스트잇을 살짝 겹쳐 붙여** 경계의 문장이 잘려도 이웃 조각에 남게 하는 장치임.

### 원리

`RecursiveCharacterTextSplitter`는 문단 → 문장 → 단어 순으로 **자연스러운 경계를 우선**해 자름. 그래서 문장 중간을 함부로 끊지 않음. `chunk_overlap`을 두면 한 조각의 끝부분이 다음 조각의 시작에 반복돼, "정책의 핵심 문장이 경계에 걸려 검색에서 누락되는" 문제를 줄임.

### 📌 참고
> 실무에서 PDF·웹페이지·CSV 등은 전용 **문서 로더**(예: `PyPDFLoader`, `WebBaseLoader`)로 불러옴. 로더는 문서를 `Document` 객체(본문 + 출처 같은 메타데이터)로 만들어 줌. 이 강의 실습에서는 단순화를 위해 문자열을 직접 다룸.

### 실무 포인트
- `chunk_size`는 "한 조각에 하나의 완결된 의미"가 담기게 잡는 것이 이상적임. 문서 성격에 따라 조절함.
- 표·코드처럼 구조가 중요한 문서는 일반 청킹이 깨뜨릴 수 있어, 형식에 맞는 분할 전략을 별도로 고려함.

### 3줄 요약
1. RAG 첫 단계는 문서를 불러와 적당한 조각으로 자르는 것임.
2. `chunk_size`는 조각 크기, `chunk_overlap`은 경계 맥락 보존 장치임.
3. `RecursiveCharacterTextSplitter`는 자연스러운 경계를 우선해 자름.

## 8-3. 임베딩 — 텍스트를 의미의 좌표로

### 개념

**임베딩(Embedding)**은 텍스트를 **숫자 벡터(좌표)**로 바꾸는 것임. 핵심 성질은 단 하나 — **의미가 비슷한 텍스트는 가까운 좌표에, 다른 텍스트는 먼 좌표에** 놓임. 이 덕분에 "의미로 검색"이 가능해짐.

```python
from langchain_google_genai import GoogleGenerativeAIEmbeddings

embeddings = GoogleGenerativeAIEmbeddings(model="gemini-embedding-001")

vec = embeddings.embed_query("반품은 어떻게 하나요?")
print(len(vec))   # 예: 768개의 숫자로 된 벡터
```

### 비유

임베딩은 **도서관의 주제별 배치**와 같음. 사서는 책을 제목의 글자가 아니라 **내용(의미)**으로 분류해 비슷한 주제끼리 가까운 책장에 꽂음. "반품 방법" 책과 "교환 절차" 책은 글자는 달라도 의미가 가까우니 옆에 놓임. 그래서 "환불하고 싶어요"라고 물으면, 글자가 정확히 일치하지 않아도 그 근처 책장에서 답을 찾을 수 있음.

### 원리

임베딩 모델은 텍스트를 **고차원 공간의 한 점(벡터)**으로 변환함. 학습을 통해 "의미가 가까운 문장은 가까운 점에" 놓이도록 만들어져 있음. 두 텍스트의 가까움은 보통 **코사인 유사도**(벡터가 이루는 각도)로 잼 — 각도가 작을수록 의미가 비슷함.

핵심은 **"반품"과 "환불"처럼 글자가 달라도 의미가 통하면 가까운 좌표**가 된다는 점임. 키워드 검색(글자 일치)이 못 하는 일을 임베딩이 함.

### 📌 참고
> 같은 임베딩 모델을 **문서 저장 때와 질문 검색 때 모두** 써야 함. 좌표계가 같아야 비교가 되기 때문임. 모델을 바꾸면 저장된 벡터도 다시 만들어야 함. (Gemini의 정식 임베딩 모델은 `gemini-embedding-001`이며, 최신 미리보기 `gemini-embedding-2-preview`는 멀티모달·가변 차원을 지원함.)

### 실무 포인트
- "검색이 엉뚱한 문서를 가져온다"면 임베딩 모델·청킹 전략을 점검함. RAG 품질의 상당 부분이 여기서 갈림.
- 한국어 문서라면 한국어를 잘 다루는 임베딩 모델을 쓰는 것이 중요함.

### 3줄 요약
1. 임베딩은 텍스트를 의미의 좌표(벡터)로 바꾸는 것임.
2. 의미가 비슷하면 가까운 좌표 → 글자가 달라도 의미로 검색 가능함.
3. 저장과 검색에 같은 임베딩 모델을 써야 비교가 됨.

## 8-4. 벡터스토어 & retriever — 저장하고 찾기

### 개념

조각들을 임베딩해 만든 벡터를 보관하는 곳이 **벡터스토어(Vector Store)**임. 그리고 질문이 오면 거기서 **"질문과 가까운 조각"을 꺼내 오는** 장치가 **retriever(검색기)**임.

```python
from langchain_core.vectorstores import InMemoryVectorStore

# 1) 조각들을 임베딩해 벡터스토어에 저장
store = InMemoryVectorStore.from_texts(chunks, embedding=embeddings)

# 2) retriever 만들기
retriever = store.as_retriever(search_kwargs={"k": 3})   # 가장 가까운 3개

# 3) 질문으로 검색
docs = retriever.invoke("반품은 어떻게 하나요?")
for d in docs:
    print(d.page_content[:60])
```

### 문법 핵심

| 요소 | 의미 |
|---|---|
| `InMemoryVectorStore.from_texts(...)` | 텍스트들을 임베딩해 메모리에 저장 |
| `.as_retriever(search_kwargs={"k": 3})` | 가장 가까운 `k`개를 찾는 retriever 생성 |
| `retriever.invoke(질문)` | 질문과 가까운 조각들(Document)을 반환 |

### 비유

벡터스토어는 **주제별로 정리된 거대한 서가**이고, retriever는 **그 서가를 뒤져 관련 책 몇 권을 뽑아 오는 사서**임. "반품 관련 자료 3개만 가져와"라고 하면(`k=3`), 사서가 의미상 가장 가까운 조각 3개를 골라 손에 쥐여줌.

### 원리

retriever는 질문을 **같은 임베딩 모델로 벡터화**한 뒤, 벡터스토어 안의 모든 조각 벡터와 **유사도(거리)를 비교**해 가장 가까운 `k`개를 돌려줌. retriever 역시 Part 3의 **Runnable**임 — `.invoke()`를 가짐. 그래서 다음 절에서 체인의 한 블록으로 끼울 수 있음.

### 📌 참고
> `InMemoryVectorStore`는 학습·소규모용으로, 프로그램이 끝나면 사라짐. 실무에서는 FAISS, Chroma, Pinecone 같은 **영속적 벡터스토어**를 씀(저장이 유지되고 대규모를 빠르게 검색). 인터페이스(`as_retriever`, `invoke`)는 거의 같아, 배운 개념이 그대로 옮겨짐.

### 실무 포인트
- `k`(가져올 개수)는 너무 작으면 근거 누락, 너무 크면 프롬프트가 길어지고 노이즈가 섞임. 보통 3~5에서 시작해 조절함.
- retriever가 Runnable이라는 점이 다음 절(RAG 체인)의 핵심 연결고리임.

### 3줄 요약
1. 벡터스토어는 조각 벡터를 보관하고, retriever는 가까운 조각을 꺼내옴.
2. `.as_retriever(k=...)`로 만들고 `.invoke(질문)`로 검색함.
3. retriever도 Runnable이라 다음 절에서 체인에 끼움.

## 8-5. RAG 체인 조립 — 검색된 문서 + 질문을 프롬프트에

### 개념

이제 모든 조각이 모임. 초급 Part 3에서 배운 `RunnableParallel`/`RunnablePassthrough`로, **"질문은 그대로 두고, 동시에 그 질문으로 문서도 검색"**해 둘 다 프롬프트에 넣는 RAG 체인을 만듦. (Part 3에서 예고했던 바로 그 패턴임.)

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnablePassthrough
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template(
    "다음 문서를 근거로만 답해줘. 모르면 모른다고 해.\n\n"
    "[문서]\n{context}\n\n[질문]\n{question}"
)

def format_docs(docs):
    return "\n\n".join(d.page_content for d in docs)

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | model
    | StrOutputParser()
)

print(rag_chain.invoke("반품은 며칠 안에 해야 하나요?"))
```

### 비유

이 체인은 **유능한 상담원**과 같음. 질문이 오면 한 손으로는 질문을 듣고(`RunnablePassthrough`), 다른 손으로는 관련 규정집을 펼침(`retriever`). 그 둘을 책상에 함께 놓고(`prompt`), 규정을 근거로 답함(`model`). "규정에 없으면 모른다고 한다"는 지시가 환각을 막음.

### 원리 — 데이터 흐름

`rag_chain.invoke("반품은 며칠?")`이 실행되면,

1. 입력 질문이 **두 갈래**로 흐름(`RunnableParallel`의 딕셔너리 형태).
   - `context`: 질문이 `retriever`로 가 관련 문서를 찾고, `format_docs`로 문자열화됨.
   - `question`: `RunnablePassthrough`로 질문이 **그대로** 전달됨.
2. 두 결과가 `{context, question}`로 모여 `prompt`의 빈칸을 채움.
3. 채워진 프롬프트가 `model`로 가 답을 생성하고, `StrOutputParser`가 문자열로 다듬음.

즉 RAG 체인은 **새 마법이 아니라**, 초급에서 배운 LCEL 도구들(파이프·병렬·통과·파서)에 retriever 하나를 끼운 것임. 초급이 중급으로 그대로 이어짐.

### 📌 참고
> "문서를 근거로만 답하고, 없으면 모른다고 하라"는 프롬프트 지시가 RAG 품질에 매우 중요함. 이게 없으면 모델이 문서를 무시하고 지어낼 수 있음. 또한 답의 출처(어느 문서에서 나왔는지)를 함께 보여주면 신뢰도가 올라감.

### 실무 포인트
- RAG 체인 자체는 **정해진 흐름(체인)**임. 여기에 "검색할지 말지 판단"이 더해지면 에이전트가 됨 — 즉 이 RAG 체인을 **에이전트의 도구로 등록**하면, 에이전트가 필요할 때만 문서를 찾아보게 만들 수 있음(Part 9에서 연결).
- 답이 부정확하면 ① 청킹 ② 임베딩 ③ `k` ④ 프롬프트 순으로 점검하는 습관이 유용함.

### 3줄 요약
1. RAG 체인은 "질문 그대로(Passthrough) + 동시에 문서 검색(retriever)"을 프롬프트에 넣음.
2. 초급의 LCEL 도구(파이프·병렬·통과·파서)에 retriever를 끼운 것임.
3. "문서 근거로만, 없으면 모른다고" 지시가 환각을 막는 핵심임.

## Part 8 마무리 — 다음으로

Part 8에서는 에이전트에게 지식을 주는 RAG의 전 과정을 익혔음. 정리하면:

- RAG = 검색한 뒤 답하기(오픈북 시험), 사적·최신 지식의 빈틈을 메움 (8-1)
- 문서 로더 & 청킹 — 불러와 적당한 조각으로 자르기 (8-2)
- 임베딩 — 텍스트를 의미의 좌표로, 의미로 검색 (8-3)
- 벡터스토어 & retriever — 저장하고 가까운 조각 찾기 (8-4)
- RAG 체인 — LCEL로 "문서 + 질문"을 프롬프트에 (8-5)

여기까지 우리는 ① 에이전트가 무엇인지(6) ② 도구를 만드는 법(7) ③ 문서에서 답을 찾는 RAG(8)를 갖췄음. 이제 이 셋을 합칠 차례임. **Part 9**에서는 드디어 1.0의 핵심 함수 **`create_agent`로 진짜 에이전트를 제대로 만듦** — 도구(7)와 RAG(8)를 손에 쥔 에이전트가 LangGraph 런타임 위에서 도구 호출 루프를 돌며 스스로 일하는 모습을 완성함.

> 함께 제공되는 **Part 8 실습 노트북**에서는 리리마켓 정책 문서를 직접 청킹·임베딩해 `InMemoryVectorStore`에 저장하고, retriever로 의미 검색이 작동하는 것을 확인한 뒤, LCEL로 RAG 체인을 조립해 "문서를 근거로 답하는" 봇을 Gemini로 완성함. 키워드 검색과 의미 검색의 차이도 직접 비교함.
