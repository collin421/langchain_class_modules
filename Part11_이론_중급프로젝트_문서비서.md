# Part 11. 중급 프로젝트 — 문서 작성 비서 에이전트

> **이 파트의 목표 (중급 졸업)**
> Part 6~10에서 배운 셋 — **도구(7) · RAG(8) · 메모리(10)** — 을 하나로 합쳐, 실제로 일하는 에이전트를 처음부터 끝까지 만듦. 주제는 **문서 자동화**임: 보고서 개요를 짜고, 섹션 초안을 쓰고, PPT 구성안을 만들고, 참고 자료에서 근거를 가져오며, "아까 그 보고서에 결론 추가"처럼 이어 작업하는 비서임. 새 개념이 아니라 통합·실전이며, 마지막에 "이 에이전트가 버거워하는 일"을 마주하며 고급(LangGraph)으로 가는 문을 엶.

---

## 11-1. 무엇을 만들 것인가 — 프로젝트 설계

### 개념

만들 것은 **문서 작성 비서 에이전트**임. 사용자가 "○○ 주제로 보고서 만들어줘"라고 하면, 에이전트가 스스로 적절한 도구를 골라 개요·초안·요약·슬라이드 구성을 만들어 주고, 대화를 이어가며 문서를 다듬어 줌.

| 항목 | 내용 |
|---|---|
| **입력** | 자연어 요청 (예: "생성형 AI 도입 보고서 개요 짜줘") |
| **출력** | 개요·섹션 초안·슬라이드 구성·요약 (필요 시 참고 자료 근거 포함) |
| **쓰는 것** | 도구(7) + RAG(8) + 메모리(10) — 중급 전부 |

### 비유

이 에이전트는 **유능한 기획 비서**와 같음. "보고서 하나 만들어 주세요"라고 하면, 비서가 알아서 목차를 잡고(도구), 자료실에서 근거를 찾아 오고(RAG), "아까 그 목차에 사례를 더 넣어 주세요"라는 후속 지시를 기억하며(메모리) 문서를 완성해 감. 우리가 단계를 일일이 지시하지 않아도 됨.

### 원리 — 설계가 먼저

Part 5에서처럼 **입력·출력을 먼저 못 박고**, 그 사이를 잇는 에이전트를 구성함. 이번엔 출력이 "정해진 한 형식"이 아니라 "요청에 따라 달라지는 문서 작업"이므로, **여러 도구를 갖춰 에이전트가 상황에 맞게 고르게** 하는 것이 핵심임.

```
사용자 요청(자연어)
   │
   ▼
[ create_agent ]  ── 문서 작업 도구들 + RAG 도구 + 메모리
   │  (요청에 맞는 도구를 스스로 골라 실행)
   ▼
문서 산출물(개요·초안·슬라이드·요약)  + 이어지는 대화 기억
```

### 📌 참고
> 여기서 "문서 자동화"는 에이전트가 **문서의 내용(텍스트·구조)을 생성**하는 것에 초점을 둠. 그 결과를 실제 .docx·.pptx 파일로 렌더링하는 것은 별개의 후처리 단계임(에이전트가 만든 구조화된 결과를 파일 생성 코드에 넘기면 됨). 이 파트는 LangChain 에이전트의 오케스트레이션에 집중함.

### 실무 포인트
- "문서 자동화"의 핵심 가치는 반복적인 초안·구성 작업을 에이전트에 위임하고, 사람은 검토·다듬기에 집중하는 것임.
- 입력은 자유로운 자연어, 출력은 재사용 가능한 구조(개요·슬라이드 목록 등)로 잡으면 활용도가 높음.

### 3줄 요약
1. 만들 것은 요청에 따라 문서를 만들어 주는 작성 비서 에이전트임.
2. 출력이 요청마다 달라지므로 여러 도구를 갖춰 에이전트가 고르게 함.
3. 에이전트는 문서 내용을 생성하고, 파일 렌더링은 별개 후처리임.

## 11-2. 도구 설계 — 문서 작업 도구들

### 개념

문서 작업을 잘게 나눠 **각각을 도구로** 만듦(Part 7의 "단일 기능" 원칙). 에이전트는 요청에 맞는 도구를 골라 씀.

```python
from langchain_core.tools import tool

@tool
def make_outline(topic: str) -> str:
    """주제를 받아 보고서 목차(개요)를 만든다. 보고서·문서 구조가 필요할 때 사용한다."""
    ...

@tool
def draft_section(section_title: str, key_points: str) -> str:
    """섹션 제목과 핵심 요점을 받아 본문 초안을 작성한다. 특정 섹션 글이 필요할 때 사용한다."""
    ...

@tool
def make_slide_plan(topic: str, num_slides: int) -> str:
    """주제와 장수를 받아 PPT 슬라이드 구성안을 만든다. 발표 자료 구성이 필요할 때 사용한다."""
    ...

@tool
def summarize_text(text: str) -> str:
    """긴 텍스트를 핵심만 요약한다. 내용을 줄여야 할 때 사용한다."""
    ...
```

### 비유

이 도구들은 **기획 비서의 업무 매뉴얼**과 같음. "목차 잡기", "섹션 쓰기", "슬라이드 구성하기", "요약하기"가 각각 정해진 업무로 나뉘어 있어, 비서가 요청에 맞는 업무를 골라 수행함. 만능 도구 하나보다, 명확한 단일 업무 여러 개가 비서를 헷갈리지 않게 함.

### 원리

각 도구의 **docstring("~할 때 사용한다")**이 에이전트의 도구 선택을 좌우함(Part 7). "보고서 만들어줘"엔 `make_outline`을, "이걸 발표 자료로"엔 `make_slide_plan`을 고르도록, 설명에 용도를 명확히 적음. 도구 내부는 보통 LLM을 한 번 더 호출하거나 정해진 형식으로 결과를 구성함 — 즉 **도구가 작은 체인을 품을 수도** 있음(Part 9의 "체인을 도구로").

### 📌 참고
> 도구가 구조화된 결과(예: 슬라이드 목록을 리스트로)를 돌려주게 하면, 그 결과를 곧바로 파일 생성·다음 단계에 쓰기 좋음(Part 4의 구조화 출력과 연결). 다만 에이전트가 읽고 판단해야 하므로, 사람이 읽을 수 있는 형태도 함께 고려함.

### 실무 포인트
- 문서 작업을 "목차 / 초안 / 구성 / 요약"처럼 의미 단위로 쪼개면 에이전트가 정확히 라우팅함.
- 각 도구가 일관된 형식(예: 마크다운)으로 결과를 내면, 이어지는 작업·렌더링이 매끄러움.

### 3줄 요약
1. 문서 작업을 단일 기능 도구들로 나눔(목차·초안·구성·요약).
2. docstring의 "언제 쓰는지"가 에이전트의 도구 선택을 좌우함.
3. 도구가 작은 체인을 품어 구조화된 결과를 낼 수 있음.

## 11-3. RAG로 근거 붙이기 — 사실에 기반한 문서

### 개념

문서가 신뢰받으려면 **근거**가 있어야 함. Part 8의 RAG를 도구로 붙이면, 에이전트가 참고 자료(사내 자료·통계·정책 등)에서 사실을 찾아 문서에 반영함.

```python
@tool
def search_reference(query: str) -> str:
    """참고 자료에서 관련 사실·수치·근거를 검색한다.
    보고서에 근거나 데이터가 필요할 때 사용한다."""
    return reference_rag_chain.invoke(query)   # Part 8 RAG 체인
```

### 비유

`search_reference`는 비서 책상의 **자료실 출입증**과 같음. 비서는 모든 내용을 지어내지 않고, 근거가 필요한 대목에서 자료실(RAG)에 들어가 사실을 확인해 문서에 인용함. "추측으로 쓴 보고서"와 "근거로 쓴 보고서"의 차이가 여기서 갈림.

### 원리

에이전트는 요청을 보고 "이 부분엔 근거가 필요하다"고 판단하면 `search_reference`를 호출해 자료를 가져온 뒤, 그 자료를 바탕으로 `draft_section` 등 다른 도구로 문서를 작성함. 즉 **여러 도구가 협력**함 — 검색 도구로 근거를 모으고, 작성 도구로 글을 씀(Part 9의 복합 질문, 여러 도구 순차 호출). 이것이 "사실에 기반한 자동 문서"의 핵심 구조임.

### 📌 참고
> 근거의 출처를 함께 표기하게 하면(예: "[자료: 2025 보고서]") 문서 신뢰도가 올라감. RAG 도구의 반환값에 출처를 담아 두면 에이전트가 인용하며 작성하게 할 수 있음.

### 실무 포인트
- 참고 자료 영역이 여럿이면 RAG 도구를 영역별로 나눔(예: `search_market_data`, `search_internal_policy`).
- "근거 없이 지어낸 내용"을 막으려면, system_prompt에 "수치·사실은 반드시 search_reference로 확인하라"고 명시함.

### 3줄 요약
1. RAG를 도구로 붙이면 에이전트가 참고 자료에서 근거를 가져옴.
2. 검색 도구(근거 수집)와 작성 도구(글쓰기)가 협력함.
3. system_prompt에 "사실은 검색으로 확인하라"를 명시해 추측을 막음.

## 11-4. 메모리로 이어 작업하기 — 멀티턴 문서 편집

### 개념

문서 작업은 한 번에 끝나지 않음. "개요 만들고 → 한 섹션 더 추가하고 → 그걸 슬라이드로"처럼 **이어지는 작업**임. Part 10의 메모리(체크포인터 + thread_id)를 붙이면, 에이전트가 앞서 만든 문서를 기억하며 다듬어 줌.

```python
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model=model,
    tools=[make_outline, draft_section, make_slide_plan, summarize_text, search_reference],
    system_prompt="너는 유능한 문서 작성 비서야. ...",
    checkpointer=InMemorySaver(),
)
cfg = {"configurable": {"thread_id": "report-1"}}

agent.invoke({"messages": [("user", "생성형 AI 도입 보고서 개요 짜줘")]}, config=cfg)
agent.invoke({"messages": [("user", "거기에 '도입 리스크' 섹션 추가해줘")]}, config=cfg)  # "거기" = 앞 개요
agent.invoke({"messages": [("user", "이 보고서를 5장 PPT로 구성해줘")]}, config=cfg)        # "이 보고서" = 지금까지 작업
```

### 비유

메모리가 붙은 비서는 **프로젝트를 며칠에 걸쳐 함께 진행하는 동료**와 같음. 어제 만든 목차를 기억하고, "그 목차에 한 장 더"라는 말을 알아들으며, 작업을 누적해 감. 매번 처음부터 설명할 필요가 없음.

### 원리

같은 `thread_id`로 호출하면, 체크포인터가 이전 대화(만든 개요·섹션 등)를 자동으로 불러와 맥락으로 제공함(Part 10). 그래서 "거기에", "이 보고서를" 같은 지시가 통함. 에이전트는 누적된 문서 상태를 보고 후속 도구를 골라 실행함. **도구(작업) + RAG(근거) + 메모리(누적)가 한 흐름에서 함께 작동**하는 것이 이 프로젝트의 완성형임.

### 📌 참고
> 문서 작업이 길어지면 누적 messages가 커짐(Part 10). 긴 작업은 핵심 산출물만 상태에 남기고 중간 대화를 요약·정리하는 전략을 함께 쓰면 좋음.

### 실무 포인트
- 문서별로 thread_id를 부여하면(예: `report-{id}`), 여러 문서 작업을 섞이지 않게 동시에 진행할 수 있음(Part 10의 사용자 분리와 같은 원리).
- "방금 그거 더 짧게", "아까 개요 다시 보여줘" 같은 자연스러운 편집 지시가 메모리로 가능해짐.

### 3줄 요약
1. 메모리(체크포인터 + thread_id)로 이어지는 문서 편집이 가능해짐.
2. "거기에", "이 보고서를" 같은 후속 지시가 통함.
3. 도구·RAG·메모리가 한 흐름에서 함께 작동하는 완성형임.

## 11-5. 중급 졸업 — 그리고 에이전트의 한계

### 개념

축하함. 도구·RAG·메모리를 모두 갖춘 에이전트를 완성했음. 이 에이전트는 강력하지만, 한 가지 본질적 특징이 있음 — **흐름을 모델의 자유 판단에 맡김.** 그래서 "정해진 절차를 반드시 따라야 하는" 작업에서는 버거워함.

### 비유 — 자유로운 비서 vs 정해진 결재 라인

지금의 에이전트는 **알아서 일하는 비서**임. 대부분의 일은 잘하지만, 회사의 **결재 라인**처럼 "초안 → 팀장 검토 → 수정 → 임원 승인 → 발행" 같은 **반드시 거쳐야 하는 정해진 절차**가 있는 일에는 맞지 않음. 비서가 중간 검토를 건너뛰거나 순서를 바꿔 버릴 수 있기 때문임.

### 원리 — 이런 요구라면?

다음 요구를 보자.

> "보고서를 작성하되, **초안을 만든 뒤 반드시 사람의 검토를 받고**, 승인되면 발행하고, **반려되면 수정해서 다시 검토**받는 절차를 따라줘."

지금의 에이전트로는 이걸 안정적으로 보장하기 어려움. 왜냐하면 이건 **정해진 워크플로우(분기·반복·사람 개입)를 명시적으로 제어**해야 하는 일이기 때문임.

- "초안 → 검토 → 승인/반려"라는 **고정된 단계**
- 반려 시 "수정 → 재검토"로 돌아가는 **반복(루프)**
- 발행 전 **반드시 사람이 개입(승인)**

`create_agent`는 모델이 도구를 자유롭게 고르는 구조라, 이런 **명시적 절차 제어**에는 한계가 있음. 여기서 필요한 것이 **LangGraph로 흐름을 직접 그리는 것** — 노드와 엣지로 단계를 정의하고, 조건 분기와 반복, 사람 개입 지점을 우리가 설계함. 이것이 고급(Part 12~)의 문임.

### 📌 참고
> 에이전트(자유 판단)와 그래프(명시적 제어)는 대립이 아니라 **계층**임. 큰 절차는 LangGraph로 짜고, 각 단계 안에서 자유 판단이 필요한 부분만 에이전트에 맡기는 식으로 결합함. 오늘 만든 에이전트가 버려지는 게 아니라, 그래프의 한 노드가 될 수 있음.

### 실무 포인트
- 새 요구가 오면 자문할 것: "흐름을 모델 판단에 맡겨도 되나, 아니면 절차를 내가 보장해야 하나?" 후자면 LangGraph임.
- 규정 준수·승인·감사 추적이 중요한 도메인일수록 명시적 흐름 제어(LangGraph)의 가치가 큼.

### 3줄 요약
1. 완성한 에이전트는 흐름을 모델의 자유 판단에 맡김.
2. "정해진 절차·반복·사람 개입"이 필요한 일은 보장하기 어려움.
3. 그때 필요한 것이 LangGraph로 흐름을 직접 그리는 것 — 고급의 시작임.

## Part 11 마무리 — 중급의 끝, 고급의 문 앞에서

Part 11에서는 중급의 모든 것을 합쳐 작동하는 문서 작성 비서를 완성했음. 정리하면:

- 문서 자동화 비서 에이전트의 설계 (11-1)
- 문서 작업을 단일 기능 도구들로 (11-2)
- RAG로 근거를 붙여 사실 기반 문서 (11-3)
- 메모리로 이어지는 멀티턴 편집 (11-4)
- 에이전트의 한계 → LangGraph의 필요성 (11-5)

**중급 전체 회고**: 체인 vs 에이전트(6) → 도구(7) → RAG(8) → create_agent 결합(9) → 메모리(10) → 통합 프로젝트(11)를 거치며, **도구로 행동하고, 자료로 근거를 대고, 대화를 기억하는** 진짜 에이전트를 손에 쥐었음. 그러나 이 모든 흐름은 **모델의 자유 판단**에 맡겨져 있었음.

**고급(Part 12~)에서는** 그 흐름의 운전대를 우리가 직접 쥠. **Part 12**는 "왜 LangGraph인가"로, 오늘 11-5에서 마주한 그 한계 — 루프·조건 분기·장기 상태·사람 개입 — 를 정면으로 다루며, `create_agent`의 편리함을 넘어 흐름을 직접 설계해야 하는 순간이 언제인지를 손에 잡히게 익힘.

> 함께 제공되는 **Part 11 실습 노트북**에서는 문서 작업 도구들 + 참고자료 RAG + 메모리를 합쳐 **문서 작성 비서 에이전트**를 처음부터 끝까지 만듦. "보고서 개요 → 섹션 추가 → 슬라이드 구성"으로 이어지는 멀티턴 작업을 직접 돌려보고, 근거가 필요한 대목에서 에이전트가 RAG를 쓰는 것을 확인한 뒤, 마지막에 "검토·승인 절차가 필요한 요구"를 던져 에이전트의 한계를 직접 마주하며 고급으로 넘어감.
