## 9-1. create_agent — 1.0 에이전트의 정식 생성

### 개념

1.0에서 에이전트를 만드는 정식 함수는 **`create_agent`**임. 모델·도구·시스템 프롬프트 세 가지만 넘기면, **LangGraph 런타임 위에서 도구 호출 루프를 도는** 에이전트가 만들어짐.

```python
from langchain.agents import create_agent

agent = create_agent(
    model=model,
    tools=[tool1, tool2],
    system_prompt="너는 리리마켓의 친절한 상담원이야.",
)
```

### 비유

`create_agent`는 **신입 직원에게 업무 환경을 세팅해 주는 것**과 같음. "이런 일을 맡길 거야"(`system_prompt`)라고 역할을 정해주고, "이 도구들을 쓰면 돼"(`tools`)라며 장비를 쥐여주면, 직원(에이전트)이 스스로 판단해 일하기 시작함. 매 작업마다 일일이 지시하지 않아도 됨.

### 원리 — 안에서 도는 루프

`create_agent`로 만든 에이전트를 `.invoke()`하면, 내부에서 다음이 반복됨(Part 6의 ReAct 루프를 LangGraph가 구현한 것).

1. **agent 노드**: 모델을 호출함(system_prompt + 지금까지의 messages).
2. 모델 응답에 **`tool_calls`가 있으면** → **tools 노드**가 그 도구를 실행해 결과를 `ToolMessage`로 추가함.
3. 다시 agent 노드로 돌아가 모델을 호출함.
4. **`tool_calls`가 없을 때까지** 2~3을 반복하고, 없어지면 전체 `messages`를 반환함.

즉 우리는 루프를 직접 코딩하지 않음 — `create_agent`가 이 "생각→행동→관찰" 사이클을 자동으로 돌려줌.

### 📌 참고
> `model`에는 우리가 만든 모델 객체(`ChatGoogleGenerativeAI(...)`)를 넣거나, `"google_genai:gemini-3-flash"` 같은 문자열 식별자를 넣을 수도 있음. 도구가 빈 리스트면 도구 호출 루프 없이 모델 노드만 있는, 사실상 체인 같은 에이전트가 됨.

### 실무 포인트
- 가장 일반적인 형태는 **static 에이전트** — 생성 시 모델·도구·프롬프트가 고정됨. 안정적이고 디버깅이 쉬워 실무 기본임.
- 에이전트 객체는 한 번 만들어 재사용하고, 요청마다 `.invoke({"messages": [...]})`로 호출함.

### 3줄 요약
1. `create_agent(model, tools, system_prompt)`이 1.0 에이전트의 정식 생성임.
2. 내부는 LangGraph가 "모델 호출 → 도구 실행 → 다시 호출"을 자동 반복함.
3. tool_calls가 없어질 때까지 루프를 돌고 전체 messages를 반환함.

## 9-2. 도구를 쥐여주기 — 여러 도구의 조합

### 개념

에이전트의 능력은 **쥐여준 도구의 조합**으로 정해짐. Part 7에서 만든 도구들을 리스트로 넘기면, 에이전트가 상황에 맞는 도구를 골라 씀.

```python
tools = [get_stock, get_exchange_rate, calculator, track_delivery]
agent = create_agent(model=model, tools=tools, system_prompt="...")

agent.invoke({"messages": [("user", "트렌치 코트 재고 있고, 50달러는 원화로 얼마야?")]})
# → 재고 도구 + 환율 도구를 차례로(혹은 함께) 골라 씀
```

### 비유

도구 조합은 **공구 벨트에 채워주는 연장들**과 같음. 망치만 채워주면 못만 박고, 드라이버·줄자·수평계까지 채워주면 다양한 작업을 함. 다만 연장이 너무 많으면(Part 7) 무엇을 쓸지 헷갈리므로, **꼭 필요한 도구를 명확한 설명과 함께** 주는 것이 중요함.

### 원리

에이전트는 각 도구의 **이름·docstring·입력 명세**(Part 7)를 모두 본 뒤, 사용자 요청에 맞는 도구를 고름. 여러 도구가 필요한 복합 질문이면, 루프를 돌며 **도구를 순차로 여러 번** 호출할 수도 있음("재고 확인 → 환율 계산"처럼). 어떤 도구를 몇 번 쓸지는 전적으로 모델의 판단임.

### 📌 참고
> 도구가 서로 역할이 겹치거나 설명이 모호하면 에이전트가 헷갈림(Part 7의 docstring 원칙이 여기서 직결됨). 도구가 많아지면(10개+) 묶거나 멀티에이전트로 분리하는 것을 고려함(Part 15).

### 실무 포인트
- 도구 리스트는 "이 에이전트가 할 수 있는 일의 목록"임. 명세를 보고 무엇을 할 수 있는 에이전트인지 한눈에 드러나야 함.
- 복합 질문에서 도구 호출 순서가 꼬이면, 각 도구의 docstring에 "언제 쓰는지"를 더 명확히 하면 대개 나아짐.

### 3줄 요약
1. 에이전트의 능력은 쥐여준 도구 조합으로 정해짐.
2. 복합 질문이면 루프를 돌며 도구를 여러 번 순차 호출할 수 있음.
3. 도구는 꼭 필요한 것을 명확한 설명과 함께 — Part 7 원칙이 직결됨.

## 9-3. RAG를 도구로 — 필요할 때만 문서 검색

### 개념

Part 8에서 만든 RAG 체인은 그 자체로 "항상 문서를 검색하는" 흐름이었음. 그런데 에이전트에 붙이면, **"필요할 때만 문서를 찾아보게"** 만들 수 있음. 방법은 간단함 — **RAG를 `@tool`로 감싸** 도구 하나로 등록하는 것임.

```python
from langchain_core.tools import tool

@tool
def search_policy(question: str) -> str:
    """리리마켓의 반품·교환·환불·배송 정책을 검색해 답한다.
    고객이 정책·규정을 물을 때 사용한다."""
    return rag_chain.invoke(question)   # Part 8에서 만든 RAG 체인

agent = create_agent(
    model=model,
    tools=[get_stock, search_policy],   # 일반 도구 + RAG 도구
    system_prompt="너는 리리마켓 상담원이야.",
)
```

### 비유

RAG를 도구로 만드는 것은 **상담원 책상에 "규정집"을 한 권 놓아두는 것**과 같음. 상담원은 모든 질문에 규정집을 펴지 않음 — "재고 있어요?"엔 재고 시스템을 보고, "반품 며칠 안에요?"처럼 정책을 물을 때만 규정집(RAG 도구)을 펼침. **언제 펼지는 상담원(에이전트)의 판단**임.

### 원리

`search_policy`의 docstring("정책·규정을 물을 때 사용한다")을 본 에이전트는, 정책 질문이 들어왔을 때만 이 도구를 호출함. 일반 질문엔 다른 도구를 쓰거나 바로 답함. 즉 **체인(항상 검색)을 도구로 감싸면, "검색할지 말지"라는 동적 판단이 에이전트의 몫이 됨** — Part 6에서 말한 "체인을 에이전트의 도구로 쓴다"가 여기서 실현됨.

### 📌 참고
> 이 패턴이 강력한 이유: 정해진 변환(RAG 체인)의 안정성은 그대로 유지하면서, "언제 그걸 쓸지"의 유연함만 에이전트에 맡김. 잘 설계된 시스템은 이렇게 **체인과 에이전트를 계층적으로 조합**함.

### 실무 포인트
- 여러 지식 영역이 있으면 RAG 도구를 영역별로 나눠 등록함(예: `search_policy`, `search_product_info`). 에이전트가 질문에 맞는 영역을 고름.
- RAG 도구의 반환값에 "근거가 된 문서"를 함께 담으면, 에이전트가 출처를 밝히며 답하게 할 수 있음.

### 3줄 요약
1. RAG 체인을 `@tool`로 감싸면 "필요할 때만 검색하는" 도구가 됨.
2. 검색할지 말지의 판단이 에이전트의 몫이 됨(체인을 도구로).
3. 체인의 안정성 + 에이전트의 유연함을 계층적으로 조합하는 핵심 패턴임.

## 9-4. system_prompt — 에이전트에게 행동 지침을 주기

### 개념

**`system_prompt`**는 에이전트의 **역할·성격·규칙·도구 사용 가이드**를 정하는 지시문임. 모델 호출 시 messages 맨 앞에 `SystemMessage`로 붙어, 에이전트의 모든 판단에 영향을 줌(Part 1의 SystemMessage가 에이전트 단위로 커진 것).

```python
agent = create_agent(
    model=model,
    tools=[get_stock, search_policy],
    system_prompt=(
        "너는 리리마켓의 친절한 상담원이야. 20~40대 여성 고객을 응대해.\n"
        "- 정책·규정 질문은 반드시 search_policy 도구로 확인하고 답해.\n"
        "- 재고 질문은 get_stock 도구를 써.\n"
        "- 모르면 추측하지 말고 '확인이 어렵다'고 안내해.\n"
        "- 답은 친절하고 간결하게."
    ),
)
```

### 비유

`system_prompt`는 **신입 상담원에게 주는 근무 지침서**임. "우리 고객은 누구인지, 어떤 말투로 응대하는지, 어떤 상황에 어떤 도구를 쓰는지, 모를 땐 어떻게 하는지"를 적어 줌. 좋은 지침서가 있으면 상담원이 일관되고 안전하게 일함.

### 원리

`system_prompt`는 매 모델 호출의 맨 앞에 놓여, 에이전트가 도구를 고르고 답을 구성하는 **모든 판단의 기준**이 됨. 특히 **도구 사용 가이드**("정책 질문은 search_policy를 써")를 명시하면, 에이전트가 올바른 도구를 더 정확히 고름 — docstring(도구 쪽 설명)과 system_prompt(에이전트 쪽 지침)가 함께 작동함.

### 📌 참고
> system_prompt는 str 또는 `SystemMessage`로 줄 수 있음. 도구가 많거나 행동 규칙이 복잡할수록, system_prompt에 "어떤 상황에 어떤 도구를 쓰라"는 가이드를 적어 두면 에이전트가 헷갈리지 않음. 다만 너무 길고 모순된 지침은 오히려 혼란을 줌.

### 실무 포인트
- 에이전트의 톤·금지사항·안전 규칙(예: "결제·삭제는 반드시 확인 후")을 system_prompt에 못 박음. 이는 고급의 human-in-the-loop(Part 14)와도 연결됨.
- 에이전트가 도구를 안 쓰거나 잘못 쓰면, 도구 docstring과 함께 system_prompt의 도구 가이드를 점검함.

### 3줄 요약
1. `system_prompt`는 에이전트의 역할·규칙·도구 사용 가이드를 정함.
2. 매 모델 호출 맨 앞에 붙어 모든 판단의 기준이 됨.
3. docstring(도구)과 system_prompt(에이전트)가 함께 도구 선택을 좌우함.

## 9-5. 에이전트 실행과 결과 읽기

### 개념

에이전트는 `{"messages": [...]}` 형태로 입력받고, 실행이 끝나면 **전체 messages 리스트**를 돌려줌. 최종 답은 마지막 메시지이지만, 중간 messages를 보면 **에이전트가 어떤 도구를 왜 불렀는지** 추적할 수 있음(디버깅의 핵심).

```python
result = agent.invoke({"messages": [("user", "반품 며칠 안에 해야 해?")]})

# 최종 답
print(result["messages"][-1].content)

# 전체 흐름 추적 (디버깅)
for m in result["messages"]:
    if getattr(m, "tool_calls", None):
        for tc in m.tool_calls:
            print(f"🔧 도구 호출: {tc['name']}({tc['args']})")
    elif type(m).__name__ == "ToolMessage":
        print(f"👁 도구 결과: {m.content[:60]}")
    elif m.content:
        print(f"💬 {type(m).__name__}: {m.content[:60]}")
```

### 비유

`messages` 리스트는 **상담 통화 녹취록**과 같음. 최종 답만 보면 "결론"만 알지만, 녹취록 전체를 보면 "상담원이 도중에 어떤 자료를 찾아봤고(도구 호출), 무엇을 확인했는지(도구 결과)"가 다 드러남. 문제가 생겼을 때 이 녹취록을 되짚어 원인을 찾음.

### 원리

에이전트의 입력·출력 단위는 **messages**임. 입력으로 대화 메시지들을 주면, 에이전트는 거기에 자신의 생각(AIMessage)·도구 호출(tool_calls)·도구 결과(ToolMessage)를 차례로 **덧붙여** 전체 리스트로 돌려줌. 그래서 messages를 순서대로 읽으면 ReAct 루프의 전 과정이 보임. `stream(...)`을 쓰면 이 과정을 실시간으로 받을 수도 있음.

### 📌 참고
> "에이전트가 왜 이렇게 답했지?"를 추적하는 일은 실무에서 매우 잦음. 이 messages 추적이 기본 디버깅 도구이며, 더 본격적인 관측·추적은 고급의 **LangSmith**(Part 16)에서 다룸. 지금은 "messages를 펼쳐 보면 흐름이 다 보인다"를 익히면 됨.

### 실무 포인트
- 입력 형식 `{"messages": [("user", "...")]}`을 기억할 것. 이전 대화를 함께 넣으면 멀티턴 맥락이 유지됨(메모리·Part 10).
- 에이전트가 이상하게 동작하면, 먼저 messages를 출력해 "어떤 도구를 어떤 인수로 불렀는지"부터 확인함.

### 3줄 요약
1. 에이전트는 `{"messages": [...]}`로 입출력하며, 결과는 전체 messages 리스트임.
2. 최종 답은 마지막 메시지, 중간 messages로 도구 호출 흐름을 추적함.
3. messages 추적이 기본 디버깅 도구이며, 본격 관측은 Part 16에서 다룸.

## Part 9 마무리 — 다음으로

Part 9에서는 중급의 모든 조각을 하나로 합쳐 작동하는 에이전트를 완성했음. 정리하면:

- `create_agent(model, tools, system_prompt)` — 1.0의 정식 생성, 내부는 LangGraph 루프 (9-1)
- 여러 도구의 조합이 에이전트의 능력 (9-2)
- RAG를 `@tool`로 감싸 "필요할 때만 검색" — 체인을 도구로 (9-3)
- `system_prompt`로 역할·규칙·도구 가이드를 부여 (9-4)
- messages로 입출력하고 흐름을 추적 (9-5)

이제 우리는 도구와 RAG를 쥔 에이전트를 만들 수 있음. 하지만 한 가지가 빠졌음 — **에이전트가 대화를 기억하지 못함.** 지금 만든 에이전트는 매 호출이 독립적이라, "아까 그 상품" 같은 맥락을 이어가지 못함(Part 1에서 본 그 한계). **Part 10**에서는 에이전트에 **메모리와 상태**를 더해, 여러 차례의 대화를 기억하며 이어가게 만듦 — 그리고 그 기억을 LangGraph 런타임이 어떻게 상태로 관리하는지, 고급(Part 12~)으로 가는 다리를 놓음.

> 함께 제공되는 **Part 9 실습 노트북**에서는 Part 7의 도구들과 Part 8의 RAG를 합쳐, 실제로 작동하는 **리리마켓 상담 에이전트**를 만듦. `create_agent`로 도구·RAG·system_prompt를 결합하고, 복합 질문에 에이전트가 여러 도구를 골라 쓰는 과정을 messages로 추적하며, "정책 질문엔 RAG 도구만 쓰는" 판단까지 Gemini로 확인함.
