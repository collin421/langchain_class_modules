# Part 6. 체인 vs 에이전트

> **이 파트의 목표 (중급의 시작)**
> Part 5 마지막에서 "체인으로는 풀 수 없는 요구"를 마주했음. 이제 그 경계를 정면으로 다룸. 체인과 에이전트의 결정적 차이는 단 하나 — **흐름을 누가 제어하는가**임. 코드가 길을 정하면 체인, 모델이 길을 정하면 에이전트임. 이 한 문장의 차이가 중급 전체를 관통함. 이 파트는 아직 도구를 깊이 다루지 않음(그건 Part 7) — 대신 "사고의 전환"을 확실히 함.

---

## 6-1. 다시, 체인이란 — 정해진 길

### 개념

초급에서 만든 모든 것은 **체인(chain)**이었음. 체인은 `프롬프트 | 모델 | 파서`처럼 **블록이 미리 정해진 순서로 한 줄로 연결된** 흐름임. 입력이 무엇이든, 똑같은 블록을 똑같은 순서로 통과함.

```python
chain = prompt | model | parser
chain.invoke({"product": "니트"})   # 항상 prompt → model → parser
```

### 비유

체인은 **공장의 컨베이어 벨트**임(Part 3). 벨트 위 부품은 1번 공정 → 2번 공정 → 3번 공정을 정해진 순서로 지남. 부품이 "나는 2번을 건너뛰고 싶어"라고 할 수 없음 — 길은 **설계자(개발자)가 미리** 깔아 두었기 때문임.

### 원리 — 흐름의 주인은 코드다

체인에서 **다음에 무엇을 할지 정하는 것은 코드(파이프 구조)**임. 모델은 그저 자기 차례에 텍스트를 생성하는 한 공정일 뿐, 전체 흐름에는 관여하지 않음. 그래서 체인은:

- **예측 가능**: 언제나 같은 경로 → 디버깅·테스트가 쉬움
- **빠르고 저렴**: 불필요한 판단 단계가 없음
- **안전**: 정해진 일만 함

### 📌 참고
> 체인이 "단순하다"는 뜻은 아님. 분기 없이 한 방향으로 흐르되, 그 안에 여러 블록·병렬(`RunnableParallel`)이 들어갈 수 있음. 핵심은 "경로가 실행 전에 이미 고정되어 있다"는 점임.

### 실무 포인트
- 입력에 관계없이 처리 순서가 항상 같다면 → 체인이 정답임.
- 요약, 번역, 분류, 정형 추출, 카드 생성처럼 "정해진 변환"은 대부분 체인으로 충분함.

### 3줄 요약
1. 체인은 블록이 미리 정해진 순서로 연결된 흐름임.
2. 흐름의 주인은 코드 — 모델은 한 공정일 뿐임.
3. 예측 가능·빠름·안전이 장점이며, 정해진 변환에 적합함.

## 6-2. 에이전트란 — 스스로 길을 고르는 흐름

### 개념

**에이전트(agent)**는 정해진 길을 따르지 않음. 대신 **모델이 매 순간 "다음에 무엇을 할지"를 스스로 판단**함. 목표를 주면, 에이전트는 가진 **도구(tool)** 중 무엇을 쓸지, 몇 번 쓸지, 언제 멈출지를 직접 결정함.

```python
from langchain.agents import create_agent

agent = create_agent(model, tools=[search_tool, calculator])
agent.invoke({"messages": [("user", "이 상품의 경쟁사 가격을 찾아 비교해줘")]})
# → 모델이 "검색 도구를 써야겠다"고 스스로 판단하고 호출함
```

### 비유

에이전트는 **목적지만 듣고 알아서 운전하는 기사**임(Part 0의 "내비게이션을 든 운전자"). "강남역으로 가주세요"라고만 하면, 기사는 길이 막히면 우회하고, 신호를 보고 멈추고, 도착하면 스스로 끝냄. 어느 길로 갈지 일일이 지시하지 않아도 됨 — **판단은 기사(모델)의 몫**임.

### 원리 — 흐름의 주인은 모델이다

에이전트에서 **다음 행동을 정하는 것은 모델**임. 코드는 "이런 도구들이 있다"고 알려주고 목표만 던질 뿐, 그 도구를 언제 어떻게 쓸지는 모델이 결정함. 그래서 에이전트는:

- **유연함**: 입력에 따라 다른 경로를 감
- **도구 사용**: 검색·계산·DB 조회 등 외부 행동이 가능함
- **반복·분기**: 필요하면 도구를 여러 번 쓰거나 중간에 방향을 바꿈

### 📌 참고
> 1.0의 `create_agent`로 만든 에이전트는 내부적으로 **LangGraph 런타임 위에서** 돎(Part 0). 즉 "모델이 판단하고 도구를 호출하는 루프"를 LangGraph가 상태를 들고 돌려줌. 지금은 그 사실만 기억하면 됨 — 고급(Part 12~)에서 이 엔진을 직접 열어 봄.

### 실무 포인트
- 에이전트의 능력은 **도구의 품질**에 크게 좌우됨. 어떤 도구를, 어떤 설명과 함께 주느냐가 핵심임(Part 7).
- "static 에이전트"(생성 시 구성이 고정)가 가장 일반적이고 안정적임. 실행 중 구성을 바꾸는 dynamic은 고급 주제임.

### 3줄 요약
1. 에이전트는 모델이 매 순간 다음 행동을 스스로 판단하는 흐름임.
2. 도구를 주면, 무엇을·언제·몇 번 쓸지 모델이 결정함.
3. `create_agent`로 만들며 내부는 LangGraph 런타임 위에서 돎.

## 6-3. 결정적 차이 — 흐름 제어의 주체

### 개념

체인과 에이전트의 차이는 결국 **단 하나의 질문**으로 갈림.

> **"다음에 무엇을 할지, 누가 정하는가?"**

| | 체인 | 에이전트 |
|---|---|---|
| 흐름을 정하는 주체 | **코드**(미리 고정) | **모델**(실행 중 판단) |
| 경로 | 항상 동일 | 입력마다 다를 수 있음 |
| 도구 사용 | 없음(정해진 블록만) | 모델이 골라 호출 |
| 예측 가능성 | 높음 | 낮음(유연한 대신) |
| 속도·비용 | 낮음·저렴 | 높음·비쌈(판단 반복) |
| 적합한 일 | 정해진 변환 | 동적 판단·외부 행동 |

### 비유

같은 "요리"라도,

- **체인 = 레시피 카드**: 1번 썰고 → 2번 볶고 → 3번 담는다. 누가 만들어도 같은 순서.
- **에이전트 = 숙련된 셰프**: 냉장고를 열어보고(관찰), 재료에 따라 메뉴를 정하고(판단), 간을 보며 조절함(반복). 목표("맛있는 한 끼")만 주면 과정은 스스로 구성함.

### 원리

이 차이가 모든 실무 결정의 기준이 됨. **흐름이 실행 전에 확정되면 체인, 실행 중에 모델이 결정해야 하면 에이전트.** 더 똑똑한 게 에이전트가 아니라, **더 유연한** 것이 에이전트임. 유연함에는 대가(속도·비용·예측 어려움)가 따름.

### 📌 참고
> 둘은 배타적이지 않음. 실무에서는 에이전트 안에서 체인을 도구처럼 쓰거나(예: "마케팅 카드 생성 체인"을 에이전트의 도구로 등록), 큰 흐름은 그래프(LangGraph)로 짜고 각 노드에 체인을 넣는 식으로 **섞어 씀**. 이건 고급의 주제임.

### 실무 포인트
- 첫 질문은 항상 "정해진 길인가, 동적 판단인가?"임.
- 애매하면 체인으로 시작해, 진짜로 동적 판단이 필요해질 때 에이전트로 올림. 과도한 에이전트화는 비용·불안정의 원인임.

### 3줄 요약
1. 결정적 차이는 "다음 행동을 코드가 정하나, 모델이 정하나"임.
2. 에이전트는 더 똑똑한 게 아니라 더 유연한 것이며, 대가가 따름.
3. 실무 판단: 정해진 길→체인, 동적 판단→에이전트.

## 6-4. 에이전트의 동작 루프 — 생각·행동·관찰

### 개념

에이전트가 "스스로 판단한다"는 것은 구체적으로 다음 **루프(loop)**를 돈다는 뜻임. 이 루프는 답을 낼 준비가 될 때까지 반복됨.

```
   ┌─────────────────────────────────────┐
   │  1. 생각(Reasoning): 지금 무엇이 필요한가?  │
   │  2. 행동(Action): 도구를 호출함            │
   │  3. 관찰(Observation): 도구 결과를 받음     │
   │  4. 판단: 충분한가? → 부족하면 1로 되돌아감   │
   └─────────────────────────────────────┘
              ↓ (충분하면)
           최종 답변
```

### 비유

이 루프는 **탐정의 수사**와 같음. 단서를 보고(관찰) "다음엔 이 사람을 조사해야겠다"고 생각하고(추론), 실제로 조사하고(행동), 그 결과로 또 다음을 정함. 사건이 풀릴 때까지 이 과정을 반복하다가, 확신이 서면 결론을 발표함.

### 원리 — 한 번의 호출이 아니다

체인의 `.invoke()`는 보통 모델을 **한 번** 부름. 하지만 에이전트의 `.invoke()`는 내부에서 모델을 **여러 번** 부를 수 있음.

1. 사용자: "경쟁사 가격을 찾아 비교해줘"
2. 모델(생각): "검색 도구가 필요하군" → 검색 도구 **호출**(행동)
3. 도구 **결과**: "경쟁사 A는 6만원, B는 5.5만원"(관찰)
4. 모델(판단): "이제 비교할 수 있다" → 최종 답변 생성

이 루프를 LangGraph 런타임이 상태(지금까지의 대화·관찰)를 들고 돌려줌. 그래서 에이전트는 도구를 한 번도, 여러 번도, 혹은 아예 안 쓸 수도 있음 — **전적으로 모델의 판단**임.

### 📌 참고
> 이 "생각→행동→관찰" 패턴을 흔히 **ReAct**(Reasoning + Acting)라 부름. 1.0의 `create_agent`가 이 루프를 자동으로 돌려주므로, 우리가 루프를 직접 코딩할 필요는 없음. 다만 안에서 이런 일이 벌어진다는 걸 알면 디버깅이 쉬워짐.

### 실무 포인트
- 루프가 너무 많이 돌면 비용·지연이 커짐. 그래서 보통 최대 반복 횟수에 상한을 둠(폭주 방지).
- 에이전트가 "엉뚱한 도구를 자꾸 부른다"면, 도구의 **설명(description)**이 불명확한 경우가 많음(Part 7에서 다룸).

### 3줄 요약
1. 에이전트는 생각→행동(도구 호출)→관찰→판단의 루프를 반복함.
2. 한 번의 `.invoke()`로 모델을 여러 번 부를 수 있음.
3. 이 ReAct 루프를 LangGraph 런타임이 자동으로 돌려줌.

## 6-5. 언제 무엇을 쓸까 — 경계 판단과 남용 경고

### 개념

가장 중요한 실무 감각 — **에이전트는 "최신·최고"가 아니라 "필요할 때 쓰는 도구"**임. 다음 기준으로 판단함.

**체인이면 충분한 신호**
- 처리 순서가 입력과 무관하게 항상 같음
- 외부 행동(검색·계산·조회)이 필요 없음
- 결과가 예측 가능해야 함(테스트·규정 준수)

**에이전트가 필요한 신호**
- 입력에 따라 경로가 달라져야 함
- 도구(검색·DB·API·계산)를 모델이 골라 써야 함
- 도구 결과에 따라 다음 행동이 바뀌어야 함(반복·분기)

### 비유

망치(에이전트)를 들면 모든 게 못으로 보임. 하지만 나사(정해진 변환)는 드라이버(체인)로 돌려야 함. **도구가 화려하다고 모든 일에 쓰는 것은 오히려 미숙함의 신호**임.

### 원리 — 에이전트 남용의 대가

에이전트를 불필요하게 쓰면 이런 비용이 따름.

- **속도·비용**: 판단 루프마다 모델 호출이 추가됨
- **예측 불가**: 같은 입력에도 경로가 달라질 수 있어 테스트가 어려움
- **디버깅 난이도**: "왜 이 도구를 불렀는지" 추적이 필요함(그래서 Part 16의 관측이 중요)

그래서 정석은 **"체인으로 시작해, 진짜로 동적 판단이 필요해지는 지점에서만 에이전트로 올린다"**임.

### 📌 참고
> 실무에서 흔한 좋은 설계: 큰 일을 잘게 쪼개, **정해진 부분은 체인으로, 판단이 필요한 부분만 에이전트로** 맡김. 예를 들어 "카드 생성"은 체인, "검색할지 말지 판단"만 에이전트가 하고 그 안에서 카드 체인을 도구로 호출하는 식임.

### 실무 포인트
- 채용·면접에서도 "왜 여기에 에이전트를 썼는가"를 설명할 수 있어야 함. "동적 판단이 필요해서"가 답이 되어야지, "최신이라서"는 위험 신호임.
- 비용 민감한 서비스라면, 에이전트 경로를 최소화하고 가능한 한 체인으로 처리하는 설계가 유리함.

### 3줄 요약
1. 에이전트는 "필요할 때 쓰는 도구"이지 항상 더 나은 선택이 아님.
2. 남용하면 속도·비용·예측 불가·디버깅 부담이 따름.
3. 정석: 체인으로 시작해, 동적 판단이 필요한 지점에서만 에이전트로 올림.

## Part 6 마무리 — 다음으로

Part 6에서는 중급의 사고 전환을 완료했음. 정리하면:

- 체인 = 정해진 길, 흐름의 주인은 코드 (6-1)
- 에이전트 = 스스로 고르는 길, 흐름의 주인은 모델 (6-2)
- 결정적 차이 = "다음 행동을 누가 정하는가" (6-3)
- 에이전트의 동작 루프 = 생각→행동→관찰→반복(ReAct) (6-4)
- 경계 판단과 남용 경고 (6-5)

이제 우리는 에이전트가 **무엇인지·왜·언제** 쓰는지 알게 됐음. 하지만 에이전트의 힘은 **도구(tool)**에서 나옴 — 도구가 없으면 에이전트는 판단만 할 뿐 아무 행동도 못 함. **Part 7**에서는 에이전트의 손과 발인 **도구를 직접 정의**하는 법을 배움. `@tool` 데코레이터, 함수 시그니처와 docstring이 왜 곧 도구의 명세가 되는지 — 즉 "모델이 어떻게 이 도구를 이해하고 고르는지"를 손에 잡히게 익힘.

> 함께 제공되는 **Part 6 실습 노트북**에서는 같은 요구를 체인과 에이전트로 각각 처리해 동작 차이를 눈으로 비교하고, `create_agent`로 첫 에이전트를 만들어 "생각→행동→관찰" 루프가 실제로 도는 과정을 관찰함. (도구는 맛보기 수준이며, 본격적인 도구 정의는 Part 7에서 다룸.)
