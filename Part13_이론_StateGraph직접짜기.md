## 13-1. 상태(State) 정의 — 그래프가 들고 도는 것

### 개념

그래프의 출발점은 **상태(state) 정의**임. 상태는 그래프가 처음부터 끝까지 **들고 다니며 갱신하는 데이터**임(Part 10의 그 상태). `TypedDict`로 "어떤 정보를 담을지"를 먼저 설계함.

```python
from typing import TypedDict, Annotated
import operator

class State(TypedDict):
    topic: str                              # 보고서 주제
    draft: str                              # 현재 초안
    review: str                             # 검토 의견
    attempts: int                           # 수정 시도 횟수
    history: Annotated[list, operator.add]  # 누적되는 기록 (리듀서)
```

### 비유

상태는 **여러 부서를 거치는 결재 서류 봉투**와 같음. 봉투 안에는 "주제, 초안, 검토 의견, 시도 횟수" 칸이 있고, 각 부서(노드)를 지날 때마다 해당 칸을 채워 넣음. 봉투 자체가 단계 사이를 이동하며 정보를 모아 감 — 이것이 그래프의 상태임.

### 원리 — 덮어쓰기 vs 누적(리듀서)

핵심 구분 하나. 노드가 상태의 한 키를 갱신할 때, 기본은 **덮어쓰기**임 — `draft`에 새 값을 주면 이전 초안이 사라짐. 그런데 "기록을 계속 쌓고 싶다"면 덮어쓰면 안 됨. 이때 **리듀서(reducer)**를 씀.

- `Annotated[list, operator.add]` → 새 값을 기존 리스트에 **이어붙임**(누적)
- 리듀서 없는 `str`, `int` → 새 값으로 **덮어씀**

리듀서는 "이 칸은 어떻게 갱신할지"를 정하는 규칙임. 메시지 누적엔 `add_messages`라는 전용 리듀서를 씀(에이전트의 messages가 쌓이던 것이 바로 이것).

### 📌 참고
> 상태는 **작고 명확하게** 유지하는 것이 좋음. 단계 사이에 꼭 전달해야 하는 정보만 담고, 임시 값은 함수 안에서 처리함. 상태가 비대해지면 그래프가 이해·디버깅하기 어려워짐.

### 실무 포인트
- "이 그래프는 무엇을 기억해야 하는가?"가 상태 설계의 첫 질문임. 그 답이 곧 State의 필드가 됨.
- 누적이 필요한 것(기록·메시지·결과 목록)만 리듀서를 붙이고, 나머지는 덮어쓰기로 둠.

### 3줄 요약
1. 상태는 그래프가 들고 다니며 갱신하는 데이터(TypedDict로 정의)임.
2. 기본은 덮어쓰기, 누적이 필요하면 리듀서(`Annotated[list, operator.add]`)를 씀.
3. 상태는 작고 명확하게 — 단계 간 꼭 필요한 정보만 담음.

## 13-2. 노드 만들기 — 상태를 받아 갱신하기

### 개념

**노드(node)**는 "하나의 작업 단계"임. 구현은 단순함 — **상태를 받아, 갱신할 부분을 딕셔너리로 돌려주는 함수**임.

```python
def draft_node(state: State) -> dict:
    topic = state["topic"]
    new_draft = f"{topic}에 대한 보고서 초안..."   # 실제론 모델 호출
    return {"draft": new_draft, "history": ["초안 작성"]}

def review_node(state: State) -> dict:
    # 현재 초안을 검토 (실제론 모델/규칙)
    ok = len(state["draft"]) > 20
    return {"review": "통과" if ok else "보강 필요", "history": ["검토"]}
```

### 비유

노드는 **결재 봉투를 받는 각 부서 담당자**임. 담당자는 봉투(상태)를 받아 자기 일을 하고, **자기가 채운 칸만** 적어 봉투를 넘김. 다른 칸은 건드리지 않음 — 그래서 노드는 "내가 갱신할 부분만" 딕셔너리로 돌려주면 됨.

### 원리 — 부분 갱신을 돌려준다

노드 함수의 규칙.

1. **입력**: 현재 상태 전체(`state`)를 받음. 필요한 값을 `state["키"]`로 읽음.
2. **출력**: **갱신할 키만** 담은 딕셔너리를 돌려줌. 안 돌려준 키는 그대로 유지됨.
3. 돌려준 값은 상태에 반영됨 — 리듀서가 있으면 누적, 없으면 덮어쓰기.

노드는 상태를 직접 수정(mutate)하지 않고 **새 부분 갱신을 반환**함. 이 "순수 함수" 방식이 그래프를 예측 가능하고 테스트하기 쉽게 만듦.

### 📌 참고
> 노드 안에서 무엇을 하든 자유임 — 모델 호출, 도구 실행, 계산, API 호출, 심지어 `create_agent`로 만든 에이전트를 부를 수도 있음(Part 12의 "노드 안에 에이전트"). 노드는 "상태→부분 갱신" 규칙만 지키면 됨.

### 실무 포인트
- 노드 하나는 한 가지 일만 하게(단일 책임) 짜면 그래프가 읽기 쉬움.
- 노드가 돌려주는 딕셔너리의 키는 반드시 State에 정의된 키여야 함.

### 3줄 요약
1. 노드는 "상태를 받아 갱신할 부분을 딕셔너리로 돌려주는" 함수임.
2. 갱신할 키만 반환하면 나머지는 유지됨(부분 갱신).
3. 상태를 직접 수정하지 않는 순수 함수라 예측·테스트가 쉬움.

## 13-3. 엣지로 잇기 — 흐름 연결하기

### 개념

노드를 만들었으면 **엣지(edge)**로 이어 흐름을 만듦. `StateGraph`에 노드를 등록하고, `add_edge`로 "이 노드 다음 저 노드"를 연결함. 시작과 끝은 특별한 노드 `START`·`END`로 표시함.

```python
from langgraph.graph import StateGraph, START, END

builder = StateGraph(State)

builder.add_node("draft", draft_node)      # 노드 등록
builder.add_node("review", review_node)

builder.add_edge(START, "draft")           # 시작 → 초안
builder.add_edge("draft", "review")        # 초안 → 검토
builder.add_edge("review", END)            # 검토 → 끝
```

### 비유

엣지는 **결재 봉투의 이동 경로를 그린 화살표**임. "접수(START) → 기획팀(draft) → 검토팀(review) → 완료(END)"처럼 봉투가 어느 부서로 갈지를 화살표로 못 박음. 이 경로는 우리가 그린 그대로 고정됨 — 봉투가 마음대로 부서를 건너뛰지 못함.

### 원리

- `add_node("이름", 함수)`: 노드를 그래프에 등록. "이름"으로 엣지에서 참조함.
- `add_edge(A, B)`: A가 끝나면 **항상** B로 감(고정 연결).
- `START`: 그래프의 진입점. `add_edge(START, 첫노드)`로 시작을 정함.
- `END`: 그래프의 종료점. 여기 닿으면 실행이 끝남.

`add_edge`로 이은 흐름은 **무조건** 그 순서로 흐름. "초안 다음 반드시 검토"가 보장됨 — 이것이 Part 12에서 말한 "흐름 보장"의 실체임. 에이전트와 달리 모델이 순서를 건너뛸 수 없음.

### 📌 참고
> 여기까지는 한 방향으로만 흐르는 단순 그래프임. 하지만 진짜 힘은 다음 절의 **조건 분기**와 **루프**에서 나옴 — "검토 결과에 따라 길을 나누고, 필요하면 되돌아가는" 흐름임.

### 실무 포인트
- `add_edge`는 "반드시 이 순서"를 보장하는 도구임. 순서가 고정돼야 하는 단계에 씀.
- 노드 이름은 명확하게(문자열) — 엣지에서 그 이름으로 연결하므로 오타에 주의함.

### 3줄 요약
1. `add_node`로 노드 등록, `add_edge`로 "A 다음 B"를 고정 연결함.
2. `START`/`END`가 진입점·종료점임.
3. `add_edge`로 이은 흐름은 무조건 그 순서 — 흐름 보장의 실체임.

## 13-4. 조건 분기 — 상태에 따라 길 나누기

### 개념

흐름이 항상 일직선은 아님. "검토를 통과하면 발행, 못 하면 수정"처럼 **상태에 따라 다른 길**로 가야 함(Part 12의 ②조건 분기). 이때 `add_conditional_edges`를 씀.

```python
def route_after_review(state: State) -> str:
    # 상태를 보고 다음 노드 이름을 돌려줌
    if state["review"] == "통과":
        return "publish"
    else:
        return "revise"

builder.add_conditional_edges(
    "review",                                  # 이 노드 다음에
    route_after_review,                        # 이 함수로 판단해서
    {"publish": "publish", "revise": "revise"} # 반환값 → 갈 노드
)
```

### 비유

조건 분기는 **검토팀이 봉투에 찍는 도장**과 같음. 검토팀이 "통과" 도장을 찍으면 봉투는 발행팀으로, "반려" 도장을 찍으면 수정팀으로 감. **어느 길로 갈지를 상태(도장)가 결정**함 — 그리고 그 판단 규칙(`route` 함수)은 우리가 명시적으로 정함.

### 원리

`add_conditional_edges`는 세 가지를 받음.

1. **출발 노드**: 이 노드가 끝난 뒤 분기를 판단함.
2. **라우팅 함수**: 상태를 받아 **다음 노드를 가리키는 값**(보통 문자열)을 돌려줌.
3. **경로 맵**: 라우팅 함수의 반환값 → 실제 갈 노드 이름.

핵심은 **라우팅 판단을 코드(우리)가 명시적으로 한다**는 점임. 에이전트는 분기를 모델 기분에 맡기지만, 그래프는 `route` 함수의 규칙대로 **반드시** 갈라짐. 이것이 "분기 보장"임.

### 📌 참고
> 라우팅 함수가 `END`를 돌려주면 그래프가 거기서 끝남. 또 조건 분기의 목적지가 **이전 노드**가 되면 루프(반복)가 됨 — 다음 절의 핵심임.

### 실무 포인트
- 라우팅 함수는 상태만 보고 결정하는 단순 함수로 유지함(부수효과 없이).
- 분기 조건이 복잡하면, 그 판단에 필요한 정보를 미리 상태에 담아 두면 라우팅이 깔끔해짐.

### 3줄 요약
1. `add_conditional_edges`로 상태에 따라 다른 노드로 분기함.
2. 라우팅 함수가 상태를 보고 다음 노드를 결정(코드가 명시적으로).
3. 에이전트와 달리 규칙대로 반드시 갈라짐 — 분기 보장임.

## 13-5. 컴파일·실행, 그리고 루프

### 개념

노드와 엣지를 다 그렸으면 **`compile()`**로 실행 가능한 그래프를 만들고, `invoke`로 실행함. 그리고 조건 분기의 목적지를 이전 노드로 두면 **루프(반복)**가 완성됨(Part 12의 ①반복).

```python
# 수정 노드: 시도 횟수를 늘리고 초안을 고침
def revise_node(state: State) -> dict:
    return {"draft": state["draft"] + " (보강됨)",
            "attempts": state["attempts"] + 1,
            "history": ["수정"]}

def publish_node(state: State) -> dict:
    return {"history": ["발행 완료"]}

builder.add_node("revise", revise_node)
builder.add_node("publish", publish_node)

builder.add_edge("revise", "review")    # 수정 → 다시 검토 (루프!)
builder.add_edge("publish", END)

graph = builder.compile()               # 실행 가능한 그래프로

result = graph.invoke({"topic": "생성형 AI 도입", "draft": "", "review": "",
                       "attempts": 0, "history": []})
print(result["history"])   # 흐름의 흔적: ['초안 작성','검토','수정','검토','발행 완료']
```

### 비유

`compile()`은 **그린 순서도를 "작동하는 기계"로 굳히는 것**임. 그때까지는 설계도(빌더)일 뿐이고, 컴파일해야 실제로 봉투가 흐르기 시작함. 그리고 "수정 → 다시 검토" 화살표가 **순환 고리**를 만들어, 봉투가 통과 도장을 받을 때까지 검토-수정을 빙빙 돎.

### 원리 — 루프와 안전장치

`add_edge("revise", "review")`처럼 **뒤 노드에서 앞 노드로** 엣지를 그으면 순환(루프)이 됨. 흐름은 "검토 → (반려) → 수정 → 검토 → (반려) → 수정 → … → (통과) → 발행"으로, 조건이 만족될 때까지 반복함.

다만 루프는 **무한 반복 위험**이 있음(검토가 계속 반려하면 영원히 돎). 그래서 두 가지 안전장치를 둠.

1. **상태로 횟수 제한**: `attempts`를 세어, 라우팅 함수에서 "N회 넘으면 강제 종료(END)"로 분기.
2. **recursion_limit**: 그래프 실행에 최대 단계 수 상한을 둬 폭주를 막음.

`compile()`은 이때 그래프의 연결을 검사함 — 끊긴 노드나 잘못된 엣지가 있으면 오류를 냄(미리 잡아줌).

### 📌 참고
> `compile(checkpointer=InMemorySaver())`처럼 체크포인터를 넣으면, 직접 만든 그래프에도 메모리가 붙음(Part 10). 즉 우리가 만든 그래프도 thread별 상태 저장·재개가 가능해짐 — Part 14의 사람 개입(중간에 멈췄다 재개)이 여기서 이어짐.

### 실무 포인트
- 루프가 있는 그래프는 **반드시 종료 조건**(횟수 제한 등)을 함께 설계함. 안 그러면 무한 반복·비용 폭주.
- `graph.get_graph().draw_mermaid()` 등으로 그래프 구조를 그림으로 확인하면 설계 검증에 좋음.

### 3줄 요약
1. `compile()`로 실행 그래프를 만들고 `invoke`로 실행함.
2. 뒤→앞 엣지로 루프(반복)를 만들며, 종료 조건을 반드시 둠.
3. 컴파일은 연결을 검사하고, 체크포인터로 메모리도 붙일 수 있음.

- 조건 분기 — `add_conditional_edges`로 상태 따라 길 나누기 (13-4)
- 컴파일·실행·루프 — `compile()`, 반복과 종료 조건 (13-5)

이제 우리는 Part 12의 네 신호 중 **①반복 ②조건 분기 ③장기 상태**를 직접 구현할 수 있음. 남은 하나가 **④사람 개입**임 — "발행 전 반드시 사람의 승인을 받는" 것처럼, 그래프가 **중간에 멈춰 사람을 기다렸다가, 사람의 입력을 받아 이어가는** 흐름임. **Part 14**에서는 이 **Human-in-the-loop(사람 개입)**를 다룸. Part 13에서 만든 체크포인터(상태 저장·재개) 위에서, 그래프를 특정 지점에 멈추고(`interrupt`) 사람의 결정을 받아 재개하는 법을 익힘.

> 함께 제공되는 **Part 13 실습 노트북**에서는 "초안 → 검토 → (통과 시 발행 / 반려 시 수정 후 재검토)" 그래프를 처음부터 직접 짜서 실행함. 상태를 정의하고, 네 노드를 만들고, 조건 분기로 통과/반려를 나누고, 수정→검토 루프와 횟수 제한 안전장치까지 넣어, Part 12에서 "에이전트로는 보장 안 됐던" 그 절차가 그래프로는 보장되는 것을 직접 확인함.
