# Part 0. 왜 LangChain인가 / 1.0 구조 한눈에 보기

> **이 파트의 목표**
> 코드를 한 줄도 쓰기 전에 "우리가 앞으로 무엇을, 왜, 어떤 순서로 만들 것인가"의 지도를 머릿속에 그리는 단계임. 이 지도가 있어야 초급의 LCEL, 중급의 `create_agent`, 고급의 LangGraph가 따로 노는 지식이 아니라 **하나의 연결된 길**로 보임.

---

## 0-1. LLM을 직접 부르면 생기는 문제

### 개념

LLM(대규모 언어 모델, Large Language Model)은 그 자체로는 "질문을 넣으면 글자를 뱉는 함수" 하나에 불과함. 예를 들어 Gemini API를 직접 호출하면 다음과 같은 일을 매번 손으로 해야 함.

- 프롬프트 문자열을 직접 이어 붙임 (`"너는 번역가야. 다음을 번역해: " + text`)
- 모델이 돌려준 응답에서 필요한 부분만 직접 잘라냄
- 모델을 OpenAI에서 Gemini로 바꾸면 호출 코드 전체를 다시 씀
- 이전 대화를 기억시키려면 대화 기록을 직접 리스트로 쌓아 관리함
- 외부 검색·계산 같은 "행동"을 시키려면 그 흐름을 전부 직접 제어함

### 비유

LLM 직접 호출은 **전기 부품을 낱개로 사서 직접 납땜하는 것**과 같음. 저항, 트랜지스터, 전선을 하나하나 연결하면 작동은 하지만, 부품 하나를 바꾸려면 회로 전체를 다시 손봐야 함.

LangChain은 그 부품들을 **표준 규격의 블록**으로 만들어 둔 것임. 블록끼리는 똑같은 모양의 단자로 연결되므로, 모델 블록 하나만 쏙 빼서 다른 모델로 갈아 끼워도 나머지 회로는 그대로 돌아감.

### 원리

LangChain의 모든 부품은 **Runnable**이라는 하나의 공통 인터페이스(약속)를 따름. "입력을 받아 출력을 내는 모든 것은 `invoke()`라는 똑같은 메서드를 가진다"는 약속임. 이 약속 덕분에 프롬프트·모델·파서를 파이프(`|`)로 줄줄이 이을 수 있음. 이 연결 문법이 곧 **LCEL(LangChain Expression Language)**이며, 초급의 핵심임.

### 📌 참고
> `Runnable`은 "실행 가능한 것"이라는 뜻의 추상 개념임. 지금은 "LangChain 부품들이 공유하는 공통 규격" 정도로만 이해하면 충분함. 실제 동작은 Part 3(LCEL 파이프)에서 손으로 다룸.

### 실무 포인트
- 프로토타입 한 번 만들고 끝낼 거라면 API 직접 호출이 더 빠를 수 있음.
- 그러나 **모델 교체·대화 기억·도구 사용·운영 모니터링**이 한 번이라도 필요해지는 순간, 직접 호출 방식은 빠르게 무너짐. LangChain은 바로 그 지점을 메우려고 만들어진 프레임워크임.

### 3줄 요약
1. LLM은 그 자체로는 "글자를 뱉는 함수" 하나일 뿐임.
2. 직접 호출하면 프롬프트·파싱·모델 교체·기억·행동 제어를 전부 손으로 해야 함.
3. LangChain은 이 반복 작업을 표준 블록으로 묶어, 갈아 끼우고 이어 붙일 수 있게 만든 도구임.

## 0-2. LangChain 1.0 — 무엇이 달라졌는가

### 개념

LangChain은 2025년 10월 **1.0 버전**으로 올라서며 구조가 크게 정리됨. 이 강의는 **1.0 기준**으로 진행함. 1.0의 핵심 메시지는 한 문장으로 요약됨.

> **"LangChain은 에이전트를 빠르게 조립하는 층, LangGraph는 그 에이전트를 안정적으로 돌리는 엔진이다."**

### 비유

LangChain 1.0의 구조는 **자동차**에 비유할 수 있음.

| 자동차 부품 | LangChain 대응 | 역할 |
|---|---|---|
| 운전석·핸들 | LangChain (LCEL, `create_agent`) | 사람이 다루기 쉬운 조작부 |
| 엔진·변속기 | LangGraph 런타임 | 실제로 힘을 내고 제어하는 내부 동력 |
| 계기판·블랙박스 | LangSmith | 무슨 일이 벌어지는지 보여주는 관측 장치 |

운전자(개발자)는 핸들만 잡으면 됨. 하지만 차가 어떻게 굴러가는지(엔진=LangGraph)를 알면, 나중에 튜닝(고급 제어)을 할 수 있음.

### 원리

1.0에서 가장 중요한 변화는 이것임.

> **모든 LangChain 에이전트는 이제 LangGraph 런타임 위에서 실행됨.**

즉 중급에서 배울 `create_agent()` 함수는 겉보기엔 LangChain 함수지만, 그 내부에서는 LangGraph가 상태를 들고 도구 호출 루프를 돌림. 그래서 초급(LCEL) → 중급(`create_agent`) → 고급(LangGraph 직접 작성)은 **단절된 세 기술이 아니라, 같은 엔진을 점점 더 깊이 들여다보는 과정**임.

### 📌 참고
> 예전 자료에서 자주 보이는 `AgentExecutor` + `create_react_agent` 방식은 1.0 이전(0.x)의 구형 패턴임. 이 강의는 1.0의 `create_agent`를 표준으로 삼음. 옛 코드를 만나면 "버전이 다르구나" 하고 구분할 수 있어야 함.

### 실무 포인트
- 1.0부터는 별도 마이그레이션 없이 에이전트가 LangGraph 위에서 돌아, **장기 상태 유지와 중단·재개**가 기본으로 따라옴.
- 운영 현장에서 실제 병목은 오케스트레이션 프레임워크가 아니라 **응답 품질과 보안**인 경우가 많음. 도구 선택보다 "어떻게 신뢰할 수 있게 만드는가"가 고급의 진짜 주제임.

### 3줄 요약
1. 이 강의는 2025년 10월 출시된 LangChain **1.0**을 기준으로 함.
2. 1.0의 핵심은 "모든 에이전트가 LangGraph 런타임 위에서 돈다"는 것임.
3. 따라서 초급·중급·고급은 같은 엔진을 점점 깊이 보는 하나의 길임.

## 0-3. 세 개의 층 — LCEL · 에이전트 · LangGraph

### 개념

앞으로 배울 모든 것은 결국 다음 세 층 위에 있음. 이 강의의 초급/중급/고급이 정확히 이 세 층과 일대일로 대응함.

| 층 | 무엇인가 | 한 문장 정의 | 이 강의 단계 |
|---|---|---|---|
| **LCEL** | 연결 문법 | 부품을 파이프(`|`)로 이어 "정해진 길"을 만드는 표현 언어 | 🟢 초급 |
| **에이전트** | 판단하는 흐름 | 도구를 보고 **스스로 다음 행동을 고르는** 실행 단위 (`create_agent`) | 🟡 중급 |
| **LangGraph** | 제어 가능한 그래프 | 분기·반복·사람 개입·여러 에이전트를 **직접 설계**하는 엔진 | 🔴 고급 |

### 비유

세 층의 차이는 **길 안내 방식**의 차이임.

- **LCEL = 직진 도로**: 출발지에서 도착지까지 길이 하나로 정해져 있음. 갈림길이 없으니 빠르고 예측 가능함.
- **에이전트 = 내비게이션을 든 운전자**: "목적지만 정해주면, 상황을 보고 어디서 꺾을지 스스로 정함." 도구가 길이고, 모델이 운전자임.
- **LangGraph = 도시 전체 교통망을 직접 설계하는 사람**: 어디에 신호등(조건 분기)을 두고, 어디서 유턴(반복)을 허용하고, 어디서 사람이 개입(human-in-the-loop)할지를 손으로 그림.

### 원리

핵심은 **"정해진 길"과 "동적 판단"의 경계**임.

- 입력이 들어오면 항상 똑같은 순서로 처리되면 충분한가? → **LCEL(체인)**이면 됨.
- 입력에 따라 "검색을 할지, 계산을 할지, 바로 답할지"를 매번 달리 골라야 하는가? → **에이전트**가 필요함.
- 그 판단이 여러 번 반복되거나, 도중에 사람의 승인을 받아야 하거나, 여러 전문 에이전트가 협업해야 하는가? → **LangGraph**로 직접 그려야 함.

### 📌 참고
> 실무의 일반 규칙: **LCEL로 시작하라.** 그러다 "반복", "조건에 따른 분기", "여러 단계에 걸친 상태 유지"라는 벽에 부딪히면 그때 LangGraph로 넘어가면 됨. 처음부터 LangGraph를 쓰는 것은 동네 마트 갈 때 트럭을 모는 것과 같음.

### 실무 포인트
- 입문 단계 채용에서는 LCEL 기반 RAG 정도만 요구하는 경우가 많음.
- 중급 엔지니어는 LangGraph를 도구함에 추가하는 단계임.
- 시니어의 진짜 역량은 "둘 다 다루는 것"을 넘어 **"언제 어느 것도 쓰지 않고 직접 구현하는 게 나은지" 판단하는 것**임.

### 3줄 요약
1. LCEL(정해진 길) → 에이전트(판단) → LangGraph(직접 설계)의 세 층이 곧 초·중·고급임.
2. 경계는 "정해진 흐름이면 충분한가, 동적 판단이 필요한가"임.
3. 항상 LCEL로 시작해, 벽에 부딪히면 LangGraph로 올라가는 것이 정석임.

## 0-4. 이 강의를 다 들으면 만들 수 있는 것

### 개념

이 커리큘럼의 종착점은 **"스스로 판단하고, 필요하면 사람의 승인을 받으며, 여러 전문가가 협업하고, 무슨 일을 했는지 기록으로 남기는 에이전트"**임. Part 0은 그 큰 그림의 지도임.

### 전체 여정 한눈에 보기

```
🟢 초급 ── 재료 익히기
   LLM 호출 → 프롬프트 → LCEL 파이프 → 출력 파서 → 첫 체인
        │  (정해진 길을 만드는 법)
        ▼
🟡 중급 ── 판단하는 에이전트
   체인 vs 에이전트 → 도구 정의 → RAG → create_agent → 메모리
        │  (스스로 도구를 고르는 단일 에이전트)
        ▼
🔴 고급 ── 제어하는 에이전트
   LangGraph → StateGraph → 사람 개입 → 멀티에이전트 → 관측 → 배포
        │  (분기·반복·협업·운영까지 직접 설계)
        ▼
   ✅ 최종: 멀티에이전트 + HITL + 관측을 갖춘 완성형 에이전트
```

### 실무 포인트
- 각 파트는 **이론 문서(개념·문법·동작 원리)**와 **실습 노트북(직접 돌려보는 코드)** 두 벌로 구성됨.
- 실습은 **Gemini API**를 사용함. 무료 등급으로도 이 강의의 거의 모든 실습을 돌릴 수 있음(상세는 Part 1 환경 설정에서 다룸).
- 한 파트라도 건너뛰면 다음 파트의 비유와 코드가 허공에 뜸. 순서대로 누적해 가는 것이 이 강의의 설계 의도임.

### 3줄 요약
1. 종착점은 판단·승인·협업·기록을 갖춘 완성형 에이전트임.
2. 길은 초급(재료) → 중급(판단) → 고급(제어)으로 누적됨.
3. 각 파트는 이론 문서 + Gemini 실습 노트북 두 벌로 진행됨.

## Part 0 마무리 — 다음으로

Part 0에서는 코드 없이 **지도**만 그렸음. 정리하면:

- LLM 직접 호출의 한계 → LangChain이 푸는 문제 (0-1)
- LangChain 1.0의 핵심: 에이전트는 LangGraph 위에서 돈다 (0-2)
- 세 개의 층 = 초·중·고급 (0-3)
- 종착점과 전체 여정 (0-4)

이제 **Part 1**에서는 이 지도 위에 첫 발을 디딤. 개발 환경을 만들고, Gemini API 키를 발급받고, LangChain으로 모델에 첫 질문을 던져 응답을 받아봄. 즉 "글자를 뱉는 함수"를 LangChain의 표준 블록으로 처음 손에 쥐는 단계임.

> 함께 제공되는 **Part 0 실습 노트북**에서는 환경을 갖추고, 설치가 제대로 됐는지 확인하고, "Hello, LangChain" 수준의 첫 호출로 전체 파이프라인이 살아 있는지 점검함.
