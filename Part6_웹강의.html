LangChain 에이전트 마스터 · Part 6 · 중급 시작

체인 vs 에이전트

초급에서 만든 모든 것은 "정해진 길"인 체인이었음. 이제 그 경계를 정면으로 다룸. 둘의 결정적 차이는 단 하나 — 흐름을 누가 제어하는가. 코드가 정하면 체인, 모델이 정하면 에이전트임.

🟡 중급 · 6/18 개념 전환 예상 35분
6-1 · 6-2

같은 요구, 다른 처리

체인은 컨베이어 벨트(정해진 순서), 에이전트는 목적지만 듣고 알아서 가는 기사임. 같은 질문을 던졌을 때 둘이 어떻게 다르게 움직이는지 나란히 봄.

인터랙티브 · 나란히 실행

"실행"을 누르면 같은 질문("경쟁사 가격을 찾아 비교해줘")을 체인과 에이전트가 각자 처리하는 과정이 단계별로 점등됨.

❓ "이 상품의 경쟁사 가격을 찾아 비교해줘"
🛣️ 체인
입력 → 프롬프트
→ 모델 (암산/추측)
→ 파서
😕 "검색을 못 해 추측함"
🧭 에이전트
입력 → 모델 (판단)
🔧 "검색 도구 쓰자" 호출
👁 결과 관찰 → 다시 판단
😎 "정확히 비교 완료"
3줄 요약
  1. 체인은 정해진 블록을 정해진 순서로 통과함.
  2. 에이전트는 모델이 도구를 골라 행동하고 결과를 관찰함.
  3. 같은 입력도 둘은 완전히 다른 경로로 처리함.
6-3

결정적 차이 — 흐름의 주인

모든 차이는 한 질문으로 갈림 — "다음에 무엇을 할지, 누가 정하는가?" 흐름의 운전대를 코드가 쥐면 체인, 모델이 쥐면 에이전트임.

인터랙티브 · 운전대는 누구 손에
📌 참고  에이전트가 더 "똑똑한" 게 아니라 더 유연한 것임. 유연함에는 대가(속도·비용·예측 어려움)가 따름. 둘은 배타적이지 않아 실무에선 섞어 씀(에이전트가 체인을 도구로 쓰는 식).
3줄 요약
  1. 결정적 차이는 "다음 행동을 코드가 정하나, 모델이 정하나"임.
  2. 에이전트는 더 똑똑한 게 아니라 더 유연한 것임.
  3. 정해진 길→체인, 동적 판단→에이전트.
6-4

에이전트의 동작 루프 — ReAct

"스스로 판단한다"는 것은 구체적으로 생각 → 행동(도구 호출) → 관찰 → 판단의 루프를 도는 것임. 답이 준비될 때까지 반복함 — 탐정의 수사처럼.

인터랙티브 · 루프가 도는 과정

"(123 + 877)을 5로 나누면?" 질문에 에이전트가 도는 루프를 한 단계씩 따라가 봄. 체인이라면 모델 호출이 1회뿐이지만, 에이전트는 여러 번임.

📌 참고  이 "생각→행동→관찰" 패턴을 ReAct(Reasoning + Acting)라 부름. create_agent가 이 루프를 자동으로 돌려주므로 직접 코딩할 필요 없음. 루프가 너무 많이 돌면 비용·지연이 커져 보통 최대 횟수에 상한을 둠.
3줄 요약
  1. 에이전트는 생각→행동→관찰→판단의 루프를 반복함.
  2. 한 번의 .invoke()로 모델을 여러 번 부를 수 있음.
  3. 이 ReAct 루프를 LangGraph 런타임이 자동으로 돌려줌.
6-5

남용 경고 — 망치와 못

에이전트는 "최신·최고"가 아니라 필요할 때 쓰는 도구임. 단순 변환에 에이전트를 쓰면 속도·비용·예측 불가의 대가를 치름. 망치를 들면 모든 게 못으로 보이지만, 나사는 드라이버로 돌려야 함.

인터랙티브 · 같은 일, 다른 비용

"상품명 → 마케팅 카드 생성"(정해진 변환)을 체인과 에이전트로 각각 처리할 때의 비용·속도·예측성을 비교함.

3줄 요약
  1. 에이전트는 항상 더 나은 선택이 아니라 필요할 때 쓰는 도구임.
  2. 남용하면 속도·비용·예측 불가·디버깅 부담이 따름.
  3. 정석: 체인으로 시작해, 동적 판단이 필요한 지점에서만 올림.

도구(Tool) 정의

에이전트의 힘은 도구에서 나옴. 도구가 없으면 판단만 할 뿐 행동을 못 함. @tool 데코레이터, 함수 시그니처·docstring이 곧 도구 명세가 되는 원리 — 즉 모델이 도구를 어떻게 이해하고 고르는지를 배움.
