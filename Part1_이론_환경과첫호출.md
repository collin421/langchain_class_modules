## 1-1. ChatModel — LangChain의 심장

### 개념

LangChain에서 LLM을 다루는 기본 블록은 **ChatModel**임. 이름 그대로 "채팅(대화)하는 모델"이며, 우리가 ChatGPT나 Gemini 앱에서 메시지를 주고받는 그 방식을 코드로 옮긴 것임.

Gemini를 쓸 때 이 블록의 구체적인 이름은 **`ChatGoogleGenerativeAI`**임. `langchain-google-genai` 패키지 안에 들어 있음.

```python
from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(model="gemini-3-flash")
```

### 비유

ChatModel은 **전화기**와 같음. 전화기 자체는 누구와 통화할지(모델 종류), 통화 품질을 어떻게 설정할지(인수)만 정해두는 장치임. 실제 대화는 전화기를 만든 뒤 "통화 버튼"(`.invoke()`)을 눌러야 시작됨.

즉 `ChatGoogleGenerativeAI(...)`로 **전화기를 장만**하고, `.invoke()`로 **전화를 거는** 두 단계로 나뉨.

### 원리

`ChatGoogleGenerativeAI`로 만든 객체는 Part 0에서 말한 **Runnable 규격**을 따름. 그래서 이 블록은 `.invoke()`, `.stream()`, `.batch()`라는 공통 메서드를 가짐. 모델을 OpenAI나 Anthropic으로 바꿔도 이 메서드 이름은 그대로임 — 이것이 LangChain이 "블록을 갈아 끼울 수 있게" 만드는 핵심 장치임.

### 📌 참고 — 어떤 모델 이름을 써야 하는가
> 무료 등급에서는 **Flash 계열**만 쓸 수 있음(Pro는 2026년부터 유료). 신규 프로젝트의 권장 기본값은 **`gemini-3-flash`**임(무료 등급 일 1,500회). 이 강의 실습은 전부 Flash 계열로 충분함. 모델 이름은 빠르게 바뀌므로, 실행이 안 되면 [Google AI Studio](https://aistudio.google.com)에서 현재 사용 가능한 모델 이름을 확인하면 됨.

### 실무 포인트
- 모델 객체는 **한 번 만들어 재사용**함. 호출할 때마다 새로 만들 필요 없음.
- "복잡한 추론은 Pro, 빠르고 대량 처리는 Flash"가 일반 규칙이지만, 대부분의 작업은 **Flash + 좋은 프롬프트**로 해결됨. Pro를 먼저 꺼내 들지 않는 습관이 비용을 아낌.

### 3줄 요약
1. LangChain에서 LLM을 다루는 기본 블록은 ChatModel이며, Gemini는 `ChatGoogleGenerativeAI`임.
2. 블록을 만드는 것과 호출하는 것은 분리됨 (전화기 장만 → 통화 버튼).
3. 무료 등급 기본 모델은 `gemini-3-flash`(Flash 계열)임.

## 1-2. 메시지의 세 종류 — System · Human · AI

### 개념

ChatModel과 대화하는 단위는 단순 문자열이 아니라 **메시지(Message)**임. 메시지에는 **역할(role)**이 붙음. LangChain에서 가장 많이 쓰는 세 종류는 다음과 같음.

| 메시지 | 클래스 | 역할 | 누가 말하는가 |
|---|---|---|---|
| 시스템 | `SystemMessage` | 모델의 정체성·규칙을 설정함 | 개발자(무대 뒤) |
| 사람 | `HumanMessage` | 실제 질문·요청 | 사용자 |
| AI | `AIMessage` | 모델의 답변 | 모델 |

```python
from langchain_core.messages import SystemMessage, HumanMessage

messages = [
    SystemMessage(content="너는 친절한 한국어 쇼핑 도우미야."),
    HumanMessage(content="니트 코디 추천해줘."),
]
llm.invoke(messages)
```

### 비유

연극 무대를 떠올리면 쉬움.

- **SystemMessage = 배우에게 주는 役 지시문**: "너는 친절한 집사야"라고 무대 뒤에서 미리 정해줌. 관객(사용자)에게는 보이지 않지만 배우의 모든 대사 톤을 결정함.
- **HumanMessage = 관객의 질문**: 무대 위에서 실제로 던지는 말임.
- **AIMessage = 배우의 대사**: 役 지시문에 맞춰 배우가 내놓는 답임.

같은 질문이라도 役 지시문(SystemMessage)이 "퉁명스러운 평론가"냐 "다정한 친구"냐에 따라 답의 분위기가 완전히 달라짐.

### 원리

모델은 이 메시지들을 **순서대로 읽어** 다음에 올 `AIMessage`를 생성함. `SystemMessage`는 보통 맨 앞에 한 번 놓여 전체 대화의 규칙을 잡고, 그 뒤로 `HumanMessage`와 `AIMessage`가 번갈아 쌓임.

문자열 하나만 넘겨도 됨 — `llm.invoke("안녕")`은 내부적으로 `HumanMessage(content="안녕")` 하나로 변환됨. 즉 **간단할 땐 문자열, 역할이 필요할 땐 메시지 리스트**를 씀.

### 📌 참고
> `SystemMessage`, `HumanMessage`, `AIMessage`는 모두 `langchain_core.messages`에서 가져옴. `content`가 실제 텍스트가 담기는 핵심 인수임.

### 실무 포인트
- 모델의 말투·역할·금지사항을 정하는 일은 거의 전부 **SystemMessage**에서 이뤄짐. 프롬프트 엔지니어링의 절반이 여기 있음.
- 리리마켓 같은 서비스 챗봇이라면 SystemMessage에 "브랜드 톤, 답변 길이, 추천 가능한 상품 범위"를 못 박아 두는 식으로 활용함.

### 3줄 요약
1. ChatModel은 역할(role)이 붙은 메시지 단위로 대화함.
2. 세 종류는 SystemMessage(규칙)·HumanMessage(질문)·AIMessage(답)임.
3. 모델의 정체성과 말투는 대부분 SystemMessage에서 결정됨.

## 1-3. 대화는 메시지의 리스트다

### 개념

한 번의 질문-답변이 아니라 **여러 차례 이어지는 대화**는, 메시지들을 **리스트(list)에 차곡차곡 쌓는** 방식으로 표현됨. 모델은 매 호출마다 그 리스트 전체를 다시 읽음.

```python
messages = [
    SystemMessage(content="너는 친절한 쇼핑 도우미야."),
    HumanMessage(content="니트 추천해줘."),
    AIMessage(content="베이지 케이블 니트는 어떠세요?"),
    HumanMessage(content="그거 말고 좀 더 캐주얼한 걸로."),  # 이전 맥락을 이어받음
]
```

### 비유

대화 리스트는 **채팅 앱의 말풍선 기록**과 같음. 새 메시지를 보낼 때 앱이 위쪽 대화 내용을 다시 화면에 띄우듯, 모델도 매번 이전 말풍선들을 전부 다시 읽고 답함.

### 원리 — 중요한 사실

> **모델은 기억력이 없음.** 매 호출은 독립적임.

"그거 말고"가 통하는 이유는 모델이 기억해서가 아니라, **우리가 이전 대화를 리스트에 담아 매번 다시 보내주기 때문임.** 만약 마지막 `HumanMessage` 하나만 보내면 모델은 "그거"가 무엇인지 전혀 알 수 없음.

이 "매번 전체를 다시 보낸다"는 번거로움을 자동으로 처리해주는 장치가 나중에 배울 **메모리(Part 10)**이며, 그 메모리는 다시 LangGraph 런타임이 상태로 관리해줌(Part 0에서 말한 그 엔진임).

### 📌 참고
> 대화가 길어질수록 매번 보내는 메시지가 늘어나 **토큰(비용)**도 늘어남. 그래서 실무에서는 오래된 메시지를 요약하거나 잘라내는 전략을 씀. 지금은 "대화 = 리스트, 모델은 매번 전체를 다시 읽는다"만 확실히 이해하면 됨.

### 실무 포인트
- 멀티턴 대화를 직접 다룰 땐, 모델의 답(`AIMessage`)도 빠짐없이 리스트에 추가해야 맥락이 유지됨.
- 맥락이 안 이어지는 버그의 대부분은 "이전 메시지를 리스트에 안 넣어서"임.

### 3줄 요약
1. 멀티턴 대화는 메시지들을 리스트에 쌓아 표현함.
2. 모델은 기억력이 없고, 매 호출마다 리스트 전체를 다시 읽음.
3. 맥락 유지는 우리가 이전 메시지를 계속 담아 보내기 때문에 가능함.

## 1-4. 모델을 조율하는 손잡이 — 주요 인수

### 개념

`ChatGoogleGenerativeAI(...)`를 만들 때 넣는 인수들은 모델의 행동을 조율하는 **손잡이(dial)**임. 핵심만 추리면 다음과 같음.

| 인수 | 의미 | 값의 예 | 효과 |
|---|---|---|---|
| `model` | 사용할 모델 이름 | `"gemini-3-flash"` | 어떤 두뇌를 쓸지 |
| `temperature` | 답변의 무작위성·창의성 | `0` ~ `1` (이상도 가능) | 낮으면 일관, 높으면 다양 |
| `max_tokens` | 최대 출력 길이 | `512` | 답변 길이 상한 |
| `max_retries` | 실패 시 자동 재시도 횟수 | `2` | 네트워크 오류 대응 |
| `timeout` | 응답 대기 한계(초) | `30` | 너무 오래 걸리면 중단 |

```python
llm = ChatGoogleGenerativeAI(
    model="gemini-3-flash",
    temperature=0.7,
    max_tokens=512,
    max_retries=2,
)
```

### 비유

`temperature`는 **요리사의 자유도**와 같음.

- `temperature=0` → "레시피를 글자 그대로 지키는 요리사." 같은 재료면 항상 똑같은 요리가 나옴. 사실 정리·분류·추출처럼 **정답이 정해진 작업**에 적합함.
- `temperature=1` → "그날 기분대로 변주하는 요리사." 매번 다른 결과가 나옴. 카피라이팅·아이디어 브레인스토밍처럼 **다양성이 필요한 작업**에 적합함.

### 원리

LLM은 다음에 올 단어를 **확률 분포**로 계산함. `temperature`는 이 분포를 얼마나 뾰족하게(낮은 값) 혹은 평평하게(높은 값) 만들지를 조절함. 낮으면 가장 확률 높은 단어만 고르므로 결과가 일정하고, 높으면 덜 확률적인 단어도 선택될 여지가 생겨 결과가 다양해짐.

### 📌 참고
> 일부 최신 Gemini 모델(3 계열 등)은 내부적으로 `temperature` 기본값을 1.0으로 두기도 함. 무작정 0으로 낮추면 오히려 성능이 떨어지는 경우가 있으니, 작업 성격에 맞춰 조절하는 감각이 중요함.

### 실무 포인트
- 데이터 추출·분류·JSON 생성 → `temperature=0`에 가깝게.
- 마케팅 문구·아이디어 → `0.7~1.0`.
- `max_tokens`를 너무 작게 잡으면 답이 중간에 잘리므로, 답변 길이를 보며 조절함.

### 3줄 요약
1. 모델 인수는 행동을 조율하는 손잡이임 (`model`, `temperature`, `max_tokens` 등).
2. `temperature`는 일관성↔다양성을 조절하며, 작업 성격에 맞춰 정함.
3. 추출·분류는 낮게, 창작은 높게가 기본 감각임.

## 1-5. 호출 메서드 — invoke · stream · batch

### 개념

모델을 실제로 실행하는 메서드는 세 가지가 핵심임. 모두 Runnable 규격이라 **모든 LangChain 블록이 동일하게** 가지고 있음.

| 메서드 | 동작 | 언제 쓰는가 |
|---|---|---|
| `.invoke(x)` | 입력 하나 → 결과 하나를 **한 번에** 받음 | 가장 기본. 단일 요청 |
| `.stream(x)` | 결과를 **조각조각 실시간**으로 받음 | 챗봇처럼 타이핑되듯 보여줄 때 |
| `.batch([x1, x2])` | 여러 입력을 **한꺼번에** 처리함 | 대량 처리(여러 문서 요약 등) |

```python
# 한 번에 받기
res = llm.invoke("안녕하세요")
print(res.content)

# 실시간 조각으로 받기
for chunk in llm.stream("긴 이야기를 들려줘"):
    print(chunk.content, end="", flush=True)

# 여러 개 한꺼번에
results = llm.batch(["고양이 소개", "강아지 소개"])
```

### 비유

- `.invoke()` = **택배**: 주문하면 완성된 상자가 한 번에 도착함.
- `.stream()` = **유튜브 라이브 자막**: 말이 끝나길 기다리지 않고 나오는 대로 글자가 흘러나옴.
- `.batch()` = **공동구매**: 주문 여러 개를 묶어 한 번에 처리함.

### 원리

`.invoke()`가 돌려주는 것은 단순 문자열이 아니라 **`AIMessage` 객체**임. 답변 본문은 `.content`에, 토큰 사용량 등 부가 정보는 `.usage_metadata` 같은 속성에 담겨 옴.

```python
res = llm.invoke("안녕")
print(res.content)          # 답변 텍스트
print(res.usage_metadata)   # 토큰 사용량 (입력/출력/합계)
```

`.stream()`은 완성된 `AIMessage` 대신 **조각(chunk)**들을 차례로 내보냄. 각 조각의 `.content`를 이어 붙이면 전체 답이 됨.

### 📌 참고
> 세 메서드 모두 앞에 `a`를 붙인 비동기 버전(`.ainvoke()`, `.astream()`, `.abatch()`)이 있음. 웹 서버처럼 동시에 많은 요청을 처리해야 할 때 씀. 지금 단계에서는 동기 버전 세 개만 확실히 익히면 충분함.

### 실무 포인트
- 사용자에게 보여주는 챗봇 UI는 거의 항상 `.stream()`을 씀 — 응답을 기다리는 체감 시간이 크게 줄기 때문임.
- 문서 수백 건을 처리할 땐 `.invoke()`를 반복문으로 도는 것보다 `.batch()`가 빠르고 효율적임.

### 3줄 요약
1. 핵심 실행 메서드는 `.invoke()`(한 번에)·`.stream()`(실시간)·`.batch()`(대량)임.
2. `.invoke()`는 `AIMessage` 객체를 돌려주며, 본문은 `.content`에 있음.
3. 챗봇 UI는 stream, 대량 처리는 batch가 정석임.

## Part 1 마무리 — 다음으로

Part 1에서는 LLM과 대화하는 모든 기초 문법을 손에 쥐었음. 정리하면:

- ChatModel(`ChatGoogleGenerativeAI`)이 기본 블록이며 "전화기 장만 → 통화 버튼"의 두 단계임 (1-1)
- 대화 단위는 역할이 붙은 메시지 — System / Human / AI (1-2)
- 멀티턴 대화는 메시지 리스트이고, 모델은 기억력이 없어 매번 전체를 다시 읽음 (1-3)
- `temperature` 등 인수로 모델 행동을 조율함 (1-4)
- `.invoke / .stream / .batch`로 실행하며, 응답은 `AIMessage` 객체임 (1-5)

여기까지는 "질문을 던지면 답이 온다"는 한 방향 호출임. 하지만 진짜 활용은 **질문을 정교하게 설계하는 것**에서 시작됨. **Part 2**에서는 매번 똑같은 문장을 손으로 쓰는 대신, 빈칸이 있는 **프롬프트 템플릿(PromptTemplate)**으로 질문을 찍어내는 법을 배움. 이것이 LCEL 파이프(`|`)로 가는 직전 단계임.

> 함께 제공되는 **Part 1 실습 노트북**에서는 메시지 세 종류를 직접 조립해 호출하고, `temperature`를 바꿔 결과 차이를 눈으로 확인하며, 멀티턴 대화와 `.stream()`까지 손으로 돌려봄.
