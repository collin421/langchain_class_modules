# Part 10. 메모리와 상태

> **이 파트의 목표**
> Part 9에서 도구와 RAG를 쥔 에이전트를 완성했지만, 한 가지 빈틈이 있었음 — **대화를 기억하지 못함.** "아까 그 상품"이라고 하면 무엇인지 모름. 이 파트에서는 에이전트에 **메모리(기억)**를 더해 멀티턴 대화를 이어가게 만듦. 그리고 그 메모리가 사실 **LangGraph의 상태(state)**라는 것을 밝히며 — 고급(Part 12~)에서 그 상태를 직접 다루는 길로 가는 다리를 놓음.

## 10-1. 왜 메모리가 필요한가 — 에이전트도 매번 잊는다

### 개념

Part 1에서 "모델은 기억력이 없고, 매 호출이 독립적"이라고 배웠음. **에이전트도 마찬가지임.** Part 9에서 만든 에이전트를 두 번 부르면, 두 번째 호출은 첫 번째를 전혀 모름.

```python
agent.invoke({"messages": [("user", "트렌치 코트 재고 있어?")]})
agent.invoke({"messages": [("user", "그거 언제 입고돼?")]})
# → 두 번째 호출은 "그거"가 무엇인지 모름. 첫 대화를 기억 못 함.
```

### 비유

기억이 없는 에이전트는 **매번 처음 보는 손님처럼 응대하는 상담원**과 같음. 방금 "트렌치 코트"를 물어본 손님이 "그거 언제 들어와요?"라고 이어 물어도, 상담원이 1초 만에 앞 대화를 잊어 "무엇을 말씀하시나요?"라고 되묻는 상황임. 자연스러운 대화가 불가능함.

### 원리

Part 1에서 멀티턴은 "이전 메시지를 리스트에 담아 매번 다시 보내서" 가능했음. 에이전트도 같은 원리지만, 매번 전체 대화를 직접 챙겨 보내는 건 번거로움. **메모리(체크포인터)**는 이 "이전 대화를 저장했다가 다시 불러오는" 일을 **자동으로** 해주는 장치임. 우리는 그저 "이건 같은 대화다"라는 표식(thread_id)만 알려주면 됨.

### 📌 참고
> 여기서 다루는 것은 **단기 메모리(short-term memory)** — 한 대화(thread) 안에서의 맥락 유지임. 사용자의 장기 선호("이 고객은 베이지를 좋아함")를 세션을 넘어 기억하는 **장기 메모리**는 별도 개념으로, 더 복잡한 저장소를 씀. 지금은 단기 메모리에 집중함.

### 실무 포인트
- 멀티턴 챗봇·상담 에이전트라면 메모리는 사실상 필수임. 없으면 매 발화가 단발성이 됨.
- "앞에서 말한 걸 기억 못 한다"는 버그의 원인은 대개 메모리(체크포인터)를 안 붙였거나 thread_id를 안 맞춘 것임.

### 3줄 요약
1. 에이전트도 매 호출이 독립적이라 기본적으로 대화를 기억 못 함.
2. 메모리는 "이전 대화를 저장했다 다시 불러오는" 일을 자동화함.
3. 단기 메모리(한 대화 내 맥락)가 이 파트의 주제임.

## 10-2. 체크포인터 — 대화 상태를 저장하는 장치

### 개념

에이전트에 메모리를 주는 장치가 **체크포인터(checkpointer)**임. 가장 단순한 것은 `InMemorySaver`로, 대화 상태를 메모리에 저장함. `create_agent`의 `checkpointer` 인수에 넣으면 끝임.

```python
from langchain.agents import create_agent
from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model=model,
    tools=[get_stock],
    checkpointer=InMemorySaver(),   # ← 메모리 장착
)
```

### 비유

체크포인터는 **상담원 옆에 놓인 상담 기록 노트**임. 손님과 나눈 대화를 그때그때 노트에 적어 두었다가, 같은 손님이 다시 말을 걸면 노트를 펼쳐 "아, 아까 트렌치 코트 물어보셨죠"라고 맥락을 이어감. `InMemorySaver`는 그 노트를 책상 위(메모리)에 두는 것 — 가게 문을 닫으면(프로그램 종료) 노트가 사라짐.

### 원리

체크포인터는 에이전트가 호출될 때마다 **그 시점의 상태(지금까지의 messages 등)를 저장(checkpoint)**하고, 다음 호출 때 그 상태를 **다시 불러와** 이어붙임. 그래서 에이전트는 매번 "백지에서 시작"하는 대신 "저장된 대화 위에서 이어"감. 이 저장·복원이 자동으로 일어나므로, 우리가 이전 메시지를 직접 챙길 필요가 없음.

### 📌 참고
> `InMemorySaver`는 학습·프로토타입용으로, 프로그램이 끝나면 기억이 사라짐. 실무에서는 `SqliteSaver`, `PostgresSaver` 같은 **영속적 체크포인터**를 써서 서버가 재시작돼도 대화가 유지되게 함. 인터페이스는 거의 같아, 배운 개념이 그대로 옮겨짐.

### 실무 포인트
- 개발·테스트는 `InMemorySaver`, 운영은 DB 기반 체크포인터로 바꾸는 것이 일반적임.
- 체크포인터를 붙이는 것만으로는 부족함 — 다음 절의 `thread_id`로 "어느 대화인지"를 알려줘야 비로소 기억이 작동함.

### 3줄 요약
1. 체크포인터는 에이전트에 메모리를 주는 장치임.
2. `InMemorySaver()`를 `create_agent`의 `checkpointer`에 넣으면 장착됨.
3. 호출 시점의 상태를 저장했다 다시 불러와 대화를 이어줌.

## 10-3. thread_id — 대화를 구분하는 열쇠

### 개념

체크포인터를 붙였으면, 호출할 때 **이 대화가 어느 대화인지**를 `thread_id`로 알려줘야 함. 같은 `thread_id`면 같은 대화로 기억이 이어지고, 다른 `thread_id`면 별개의 대화로 분리됨.

```python
config = {"configurable": {"thread_id": "user-123"}}

agent.invoke({"messages": [("user", "트렌치 코트 재고 있어?")]}, config=config)
agent.invoke({"messages": [("user", "그거 언제 입고돼?")]}, config=config)
# → 같은 thread_id라 "그거"가 트렌치 코트임을 기억함!
```

### 비유

`thread_id`는 **상담 기록 노트의 고객 이름표**임. 노트에는 여러 손님의 대화가 적혀 있는데, 이름표(thread_id)로 "지금 응대하는 손님의 페이지"를 찾음. 김 손님(thread A)과 이 손님(thread B)의 대화는 같은 노트에 있어도 페이지가 분리돼 섞이지 않음.

### 원리

`thread_id`는 체크포인터가 상태를 **저장·조회하는 기본 키**임. 호출 시 `config`에 담아 넘기면, 체크포인터는 그 키에 해당하는 대화 기록을 불러와 이어붙이고, 끝나면 다시 그 키로 저장함. 그래서:

- 같은 `thread_id` → 같은 페이지 → 맥락 유지
- 다른 `thread_id` → 다른 페이지 → 완전히 새 대화

이 덕분에 한 에이전트로 **여러 사용자·여러 세션을 동시에, 서로 섞이지 않게** 응대할 수 있음.

### 📌 참고
> `thread_id`를 안 넘기거나 매번 바꾸면, 에이전트가 매 호출을 새 대화로 취급해 기억이 안 이어짐("앞 대화를 잊는" 버그의 흔한 원인). 보통 사용자 ID나 세션 ID를 thread_id로 씀.

### 실무 포인트
- 실제 서비스에서는 사용자별·세션별로 고유한 thread_id를 부여함(예: `f"user-{user_id}-{session_id}"`).
- "다른 사용자 대화가 섞인다" → 같은 thread_id를 공유한 것. "한 사용자의 맥락이 안 이어진다" → thread_id가 매번 달라진 것. 이 둘을 구분해 점검함.

### 3줄 요약
1. `thread_id`는 "이 대화가 어느 대화인지" 알려주는 열쇠임.
2. 같은 thread_id면 맥락 유지, 다르면 별개 대화로 분리됨.
3. `config={"configurable": {"thread_id": ...}}`로 호출 시 넘김.

## 10-4. 멀티턴 실전 — 맥락이 이어지는 에이전트

### 개념

체크포인터 + thread_id를 합치면, 드디어 "아까 그 상품"이 통하는 에이전트가 됨. Part 9의 상담 에이전트에 메모리를 붙여 자연스러운 멀티턴 대화를 완성함.

```python
agent = create_agent(
    model=model,
    tools=[get_stock, search_policy],
    system_prompt="너는 리리마켓 상담원이야.",
    checkpointer=InMemorySaver(),
)
cfg = {"configurable": {"thread_id": "session-1"}}

agent.invoke({"messages": [("user", "베이지 니트 재고 있어요?")]}, config=cfg)
# AI: "네, 12개 있습니다!"
agent.invoke({"messages": [("user", "그거 반품 되나요?")]}, config=cfg)
# AI: "베이지 니트는 수령 후 7일 이내 반품 가능합니다" ← "그거"=베이지 니트로 이해!
```

### 비유

이제 상담원이 **노트를 펼쳐 보며 대화하는 베테랑**이 됨. "그거", "아까 말한 거", "방금 그 상품" 같은 자연스러운 말을 알아듣고, 앞 대화를 근거로 매끄럽게 이어감. 손님은 매번 상품명을 반복할 필요가 없음.

### 원리

매 호출마다 체크포인터가 그 thread의 이전 messages를 자동으로 불러와 새 메시지 앞에 붙임. 그래서 모델은 "베이지 니트 재고 → 12개 → 그거 반품?"이라는 **전체 흐름**을 보고 답함. 우리가 한 일은 `checkpointer`를 붙이고 같은 `thread_id`를 넘긴 것뿐임 — 나머지는 자동임.

### 📌 참고
> 대화가 아주 길어지면 매번 불러오는 messages가 커져 토큰(비용)·속도 부담이 생김. 그래서 오래된 메시지를 잘라내거나 요약하는 전략을 함께 씀(미들웨어로 메시지를 트리밍하는 방식 등). 지금은 "메모리로 맥락이 이어진다"를 확실히 익히면 됨.

### 실무 포인트
- 멀티턴이 핵심인 상담·비서 에이전트는 이 패턴(checkpointer + thread_id)이 기본 골격임.
- 대화 길이가 길어지는 서비스는 메시지 관리(트리밍·요약) 전략을 처음부터 함께 설계하는 것이 좋음.

### 3줄 요약
1. checkpointer + 같은 thread_id면 "아까 그 상품"이 통함.
2. 체크포인터가 이전 messages를 자동으로 불러와 이어붙임.
3. 긴 대화는 트리밍·요약 전략을 함께 고려함.

## 10-5. 상태(state)란 무엇인가 — 고급으로 가는 다리

### 개념

지금까지 "메모리"라고 부른 것은 사실 **LangGraph의 상태(state)**임. Part 0·6·9에서 "에이전트는 LangGraph 런타임 위에서 돈다"고 했음. 그 런타임이 들고 도는 것이 바로 **상태** — 지금까지의 messages를 비롯한 "대화의 현재 모습" 전체임.

### 비유

상태는 **보드게임의 현재 판**과 같음. 말의 위치, 점수, 남은 턴 — 게임의 모든 현재 상황이 판에 담겨 있음. 매 턴(에이전트 호출)마다 이 판을 보고 다음 수를 두고, 판을 갱신함. 체크포인터는 그 판을 **사진 찍어 저장**해 두었다가, 다음에 그 판에서 게임을 이어가게 해주는 것임.

### 원리

LangGraph에서 에이전트의 한 걸음 한 걸음(모델 호출, 도구 실행)은 모두 **상태를 읽고 → 갱신하는** 과정임. messages는 그 상태의 한 부분일 뿐임. 체크포인터는 이 상태를 thread_id별로 저장·복원함. 즉:

> **메모리 = 상태를 저장·복원하는 것 = LangGraph의 상태 관리.**

지금은 `create_agent`가 이 상태(messages 중심)를 알아서 관리해줬음. 하지만 **고급(Part 12~)에서 LangGraph를 직접 다루면, 이 상태에 우리가 원하는 정보(예: 현재 단계, 누적 결과, 승인 여부)를 직접 정의하고 제어**하게 됨. 오늘 배운 메모리가 그 큰 개념의 첫 입구임.

### 📌 참고
> 그래서 이 파트는 중급의 마지막이자 고급의 예고편임. "에이전트가 기억한다"는 편리한 기능 뒤에는 "LangGraph가 상태를 들고 돈다"는 구조가 있고, 그 구조를 직접 손에 쥐는 것이 고급의 핵심임.

### 실무 포인트
- `create_agent`만 쓸 땐 상태를 의식할 필요가 적지만, 복잡한 흐름(분기·반복·여러 단계)을 직접 설계하려는 순간 "상태를 어떻게 정의할까"가 핵심 질문이 됨(Part 13).
- "에이전트가 무엇을 기억하고 무엇을 잊는가"는 결국 "상태에 무엇을 담는가"의 문제임.

### 3줄 요약
1. "메모리"는 사실 LangGraph의 상태(state)를 저장·복원하는 것임.
2. 상태는 messages를 포함한 "대화의 현재 모습" 전체임.
3. 고급에서는 이 상태를 직접 정의·제어함 — 오늘이 그 입구임.

## Part 10 마무리 — 중급의 끝, 고급의 문 앞에서

Part 10에서는 에이전트에 기억을 주고, 그 기억의 정체(상태)를 밝혔음. 정리하면:

- 에이전트도 매 호출이 독립적이라 기억이 필요함 (10-1)
- 체크포인터(`InMemorySaver`)가 메모리 장치 (10-2)
- `thread_id`가 대화를 구분하는 열쇠 (10-3)
- checkpointer + thread_id로 "아까 그 상품"이 통하는 멀티턴 완성 (10-4)
- 메모리 = LangGraph의 상태, 고급으로 가는 다리 (10-5)

**중급 전체 회고**: 우리는 체인과 에이전트의 차이(6) → 도구 정의(7) → RAG(8) → create_agent로 결합(9) → 메모리(10)를 거치며, **도구·지식·기억을 갖춘 진짜 에이전트**를 손에 쥐었음. 이제 남은 건 마지막 통합 — **Part 11**에서 이 모든 것을 합쳐 **중급 프로젝트**(RAG와 여러 도구를 쓰는 기억하는 리서치/상담 에이전트)를 처음부터 끝까지 완성함. 그리고 그 프로젝트의 한계를 마주하며, "분기·반복·사람 개입·멀티에이전트"를 직접 제어하는 **고급(LangGraph)**으로 넘어가는 문을 엶.

> 함께 제공되는 **Part 10 실습 노트북**에서는 Part 9의 상담 에이전트에 `InMemorySaver`를 붙여 "아까 그 상품"이 통하게 만들고, **같은 thread_id에서는 맥락이 이어지고 다른 thread_id에서는 분리되는** 것을 직접 비교하며, 한 에이전트로 여러 사용자를 섞이지 않게 응대하는 실습을 Gemini로 진행함.
