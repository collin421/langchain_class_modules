LangChain 에이전트 마스터 · Part 17

배포와 신뢰성

시스템을 실제 세상으로 내보냄. 프로토타입은 "한 번 잘 돌면" 충분하지만, 프로덕션은 다름 — 재시작에도 대화가 유지되고, 실패해도 견디고, 무한 루프와 비용 폭주를 막아야 함. 한 번 됨 ≠ 항상 안정.

🔴 고급 · 17/18 프로덕션 예상 40분
17-1

프로토타입 vs 프로덕션

지금까지 만든 건 대부분 프로토타입(한 번 잘 돎). 프로덕션은 다른 요구를 가짐 — 여러 사용자가 동시에, 며칠에 걸쳐, 실패·재시작 속에서도 안정적이어야 함. 요리 연습과 식당 운영의 차이임.

인터랙티브 · 무엇이 달라지는가

각 측면을 눌러, 프로토타입과 프로덕션이 어떻게 다른지 봄.

측면프로토타입프로덕션
상태메모리(소실)영속 DB ↗17-2
실패한 번 되면 됨재시도로 견딤 ↗17-3
폭주신경 안 씀상한 방어 ↗17-4
비용무시최적화 ↗17-5
행을 눌러 자세히 보세요.
3줄 요약
  1. 프로토타입(한 번 됨)과 프로덕션(항상 안정)은 다른 문제임.
  2. 프로덕션은 영속 상태·실패 대비·폭주 방어·비용을 요구함.
  3. 신뢰성은 기능이 아니라 운영의 품질임.
17-2

영속 상태 — 재시작에도 살아남기

InMemorySaver는 상태를 RAM에 둠 — 재시작 시 사라짐(포스트잇). 프로덕션은 SqliteSaver/PostgresSaver로 DB에 저장해 재시작에도 이어감(장부). 교체는 거의 한 줄.

인터랙티브 · 재시작 시뮬레이션

저장 방식을 고르고 "서버 재시작"을 눌러, 대화가 사라지는지 유지되는지 봄.

📌 참고  인터페이스가 같아 교체는 거의 한 줄(InMemorySaver → SqliteSaver). 단, PostgresSaver 등은 최초 setup()(테이블 생성)이 필요하고, 누적되는 체크포인트는 주기적 정리(TTL)가 필요함. 상태엔 큰 파일 대신 참조(URL)만 담음.
3줄 요약
  1. InMemorySaver는 재시작 시 소실 — 프로덕션은 영속 체크포인터로.
  2. 교체는 거의 한 줄(setup·저장소 선택만 추가).
  3. 상태엔 참조만, 오래된 체크포인트는 주기적으로 정리.
17-3

오류 처리 — 실패에 견디기

프로덕션에서 외부 의존(모델·API·네트워크)은 반드시 가끔 실패함. 신뢰성은 "실패가 없는 것"이 아니라 "실패해도 무너지지 않는 것" — 재시도·타임아웃·대체로 흡수함.

인터랙티브 · 가드 있을 때 vs 없을 때

토글해서, 같은 실패가 가드 없이는 시스템을 무너뜨리고 가드 있으면 흡수되는 걸 봄.

⚡ 모델 API 일시적 오류 (rate limit)
❌ 가드 없음: 예외 발생 → 요청 실패, 사용자에게 에러
✅ max_retries=3: 잠시 후 자동 재시도 → 대개 성공
⏱️ 도구가 응답 없이 멈춤
❌ 가드 없음: 요청이 영원히 매달림 → 시스템 점유
✅ timeout=30: 30초 후 중단 → 다른 요청 보호
💥 도구 내부 예외 (외부 API 다운)
❌ 가드 없음: 예외가 터져 그래프 전체 중단
✅ try/except: "조회 실패" 메시지로 변환 → 에이전트가 대응
📌 참고  무엇이든 재시도해선 안 됨. 비가역 행동(결제·발송)을 무턱대고 재시도하면 중복 실행 위험 → 멱등성(여러 번 해도 같은 결과)을 보장하거나, 재시도 전 확인(HITL·Part 14)을 둠.
3줄 요약
  1. 외부 의존은 반드시 가끔 실패함 — 예상하고 견딤.
  2. 재시도·타임아웃·대체·우아한 저하로 흡수함.
  3. 비가역 행동의 재시도는 멱등성·확인 장치와 함께만.
17-4

안전장치 — 폭주를 막다

자동화의 큰 위험은 폭주 — 무한 루프, 비용 치솟음. recursion_limit·단계 카운터·비용 상한·HITL 가드로 막음. 전기의 차단기처럼, 위험 수위에서 회로를 끊음.

인터랙티브 · recursion_limit 차단기

"한 단계씩"을 눌러, 종료 조건 없는 루프가 돌다 recursion_limit(10)에서 차단되는 걸 봄.

node → node → node ... (종료 조건 없음)
단계: 0
recursion_limit = 10
📌 참고  recursion_limit 도달 시 LangGraph가 GraphRecursionError를 자동으로 냄 → 잡아서 run을 "error"로 표시하고 우아하게 안내(17-3). 이중 안전장치: recursion_limit + 상태 단계 카운터(Part 13)를 함께 둠. 비가역 행동엔 HITL(Part 14).
3줄 요약
  1. 자동화의 큰 위험은 무한 루프·비용 폭주·위험 행동임.
  2. recursion_limit·단계 카운터·비용 상한·HITL로 막음.
  3. 기본값을 믿지 말고 상한을 명시 — 무제한은 사고의 씨앗.
17-5

비용·지연 관리

프로덕션은 비용·속도가 사업적으로 중요. 모델 선택·메시지 트리밍·캐싱 등으로 최적화함 — 식당의 원가·회전율 관리처럼. 단, 측정(Part 16) 후에 최적화함.

인터랙티브 · 최적화 레버

레버를 켜며, 비용·지연 게이지가 줄어드는 걸 봄.

모델 차등 — 라우팅엔 작은 모델, 핵심 추론에만 큰 모델
메시지 트리밍 — 오래된 대화를 잘라/요약해 컨텍스트 경량화
캐싱 — 반복 요청 결과 재사용
호출 절감 — 불필요한 도구·과한 핸드오프 줄이기
비용·지연100%
📌 참고  최적화는 측정 후에 함(Part 16). 트레이스로 "어느 단계가 느리고 비싼지" 먼저 보고 거기를 고침. 측정 없는 최적화는 엉뚱한 곳을 깎기 쉬움 — "추측하지 말고 측정하라".
3줄 요약
  1. 프로덕션은 비용·지연이 사업적으로 중요해짐.
  2. 모델 차등·트리밍·캐싱·호출 절감으로 최적화함.
  3. 최적화는 측정(트레이스) 후에 — 추측하지 말고 측정함.
