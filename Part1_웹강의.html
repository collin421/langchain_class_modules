첫 LLM 호출

지도 위에 디디는 첫 발임. LangChain의 기본 블록 ChatModel을 손에 쥐고, 대화의 단위인 메시지 세 종류를 익히며, 모델을 조율하는 인수와 실행 메서드까지 다룸.

🟢 초급 · 1/5 gemini-3-flash 예상 30분
1-1

ChatModel — LangChain의 심장

LangChain에서 LLM을 다루는 기본 블록은 ChatModel임. Gemini를 쓸 때 이름은 ChatGoogleGenerativeAI이며 langchain-google-genai 패키지에 들어 있음.

from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(model="gemini-3-flash")

ChatModel은 전화기와 같음. 전화기를 장만하는 일(ChatGoogleGenerativeAI(...))과 전화를 거는 일(.invoke())은 분리됨.

인터랙티브 · 어떤 모델을 쓸까

모델을 눌러 속도·비용·성능의 균형을 비교해봄. 무료 등급은 Flash 계열만 쓸 수 있음.

📌 참고  무료 등급 권장 기본값은 gemini-3-flash임. Pro는 2026년부터 유료라 무료 등급에서 못 씀. 모델 이름은 자주 바뀌니 실행이 안 되면 Google AI Studio에서 현재 모델명을 확인하면 됨.
3줄 요약
  1. 기본 블록은 ChatModel, Gemini는 ChatGoogleGenerativeAI임.
  2. 블록 생성과 호출은 분리됨(전화기 장만 → 통화 버튼).
  3. 무료 등급 기본 모델은 gemini-3-flash임.
1-2

메시지의 세 종류 — System · Human · AI

ChatModel과 대화하는 단위는 문자열이 아니라 역할(role)이 붙은 메시지임. SystemMessage(규칙)·HumanMessage(질문)·AIMessage(답) 세 종류가 핵심임.

연극으로 보면 SystemMessage는 役 지시문(무대 뒤), HumanMessage는 관객의 질문, AIMessage는 배우의 대사임. 役 지시문이 바뀌면 같은 질문에도 답의 분위기가 완전히 달라짐.

인터랙티브 · 메시지 블록 조립

블록을 추가해 메시지 리스트를 만들고 "모델 호출"을 눌러봄. SystemMessage 유무에 따라 답이 어떻게 달라지는지 관찰함.

※ 실제 응답은 노트북에서, 여기선 동작 시뮬레이션임
📌 참고  세 메시지 모두 langchain_core.messages에서 가져오며, content가 텍스트 인수임. 모델의 말투·역할·금지사항은 거의 전부 SystemMessage에서 정해짐.
3줄 요약
  1. 대화 단위는 역할이 붙은 메시지임.
  2. 세 종류는 System(규칙)·Human(질문)·AI(답)임.
  3. 정체성·말투는 대부분 SystemMessage에서 결정됨.
1-4

모델을 조율하는 손잡이 — temperature

모델을 만들 때 넣는 인수는 행동을 조율하는 손잡이임. 가장 중요한 것이 temperature — 답변의 무작위성·창의성을 조절함.

요리사의 자유도에 비유됨. 0은 레시피를 글자 그대로 지키는 요리사(매번 같은 결과), 1은 기분대로 변주하는 요리사(매번 다른 결과)임.

인터랙티브 · temperature 슬라이더

"리리마켓 가을 세일 슬로건"을 같은 값으로 3번 요청했다고 가정함. 슬라이더를 움직여 결과가 얼마나 흩어지는지 봄.

temperature0.0
0 · 일관0.51.0 · 다양
📌 참고  추출·분류·JSON 생성은 0에 가깝게, 카피·아이디어는 0.7~1.0으로. 일부 최신 모델은 기본값이 1.0이라 무작정 0으로 낮추면 성능이 떨어질 수 있음.
3줄 요약
  1. 인수는 모델 행동을 조율하는 손잡이임.
  2. temperature는 일관성↔다양성을 조절함.
  3. 추출·분류는 낮게, 창작은 높게가 기본 감각임.
1-3

대화는 메시지의 리스트다

여러 차례 이어지는 대화는 메시지를 리스트에 쌓아 표현함. 그리고 가장 중요한 사실 하나 — 모델은 기억력이 없음. 매 호출은 독립적임.

"그거 말고"가 통하는 이유는 모델이 기억해서가 아니라, 우리가 이전 대화를 리스트에 담아 매번 다시 보내기 때문임.

인터랙티브 · 맥락을 넣을까 뺄까

"그거 말고 캐주얼한 걸로"라는 2번째 질문을 보낼 때, 이전 대화를 함께 보내는 경우와 빼는 경우를 비교함.

📌 참고  멀티턴에서는 모델의 답(AIMessage)도 빠짐없이 리스트에 추가해야 맥락이 유지됨. 이 번거로움을 자동화한 것이 나중의 메모리(Part 10)이며, 그 메모리는 LangGraph 런타임이 상태로 관리함.
3줄 요약
  1. 멀티턴 대화는 메시지 리스트로 표현함.
  2. 모델은 기억력이 없고, 매번 리스트 전체를 다시 읽음.
  3. 맥락 유지는 이전 메시지를 계속 담아 보내기 때문에 가능함.
1-5

호출 메서드 — invoke · stream · batch

모델을 실행하는 핵심 메서드는 셋임. 모두 Runnable 규격이라 모든 LangChain 블록이 동일하게 가지고 있음.

인터랙티브 · 세 메서드의 출력 방식

탭을 눌러 각 메서드가 결과를 어떻게 내보내는지 애니메이션으로 비교함.

.invoke()
.stream()
.batch()
📌 참고  세 메서드 모두 비동기 버전(.ainvoke() 등)이 있음. .invoke()가 돌려주는 것은 문자열이 아니라 AIMessage 객체이며, 본문은 .content, 토큰 사용량은 .usage_metadata에 있음.
3줄 요약
  1. 핵심 메서드는 invoke(한 번에)·stream(실시간)·batch(대량)임.
  2. .invoke()는 AIMessage 객체를 돌려주며 본문은 .content임.
  3. 챗봇 UI는 stream, 대량 처리는 batch가 정석임.
