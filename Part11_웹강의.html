도구(7)·RAG(8)·메모리(10)를 하나로 합쳐 일하는 비서를 완성함. 보고서 개요를 짜고, 섹션을 쓰고, PPT를 구성하고, 자료에서 근거를 찾고, "아까 그 보고서에 추가"처럼 이어 작업함. 마지막엔 "정해진 절차"의 한계를 마주하며 고급의 문을 엶.

11-1

설계 — 문서 작성 비서

"○○ 주제로 보고서 만들어줘"라고 하면, 에이전트가 스스로 도구를 골라 개요·초안·구성·요약을 만들고, 대화를 이어가며 다듬어 주는 비서임. 출력이 요청마다 달라지므로 여러 도구를 갖춰 에이전트가 고르게 하는 것이 핵심임.

인터랙티브 · 프로젝트 설계도

자유로운 자연어 입력을, 요청에 맞는 문서 산출물로. 중급 셋이 모두 엔진으로 들어감.

입력
"생성형 AI 도입
보고서 짜줘"
(자연어)
→
출력 (요청에 따라)
📋 보고서 개요
✍️ 섹션 초안
📊 슬라이드 구성
📝 요약
🔧 도구 (7) 📕 RAG (8) 🧵 메모리 (10)
📌 참고  여기서 "문서 자동화"는 에이전트가 내용·구조를 생성하는 데 초점을 둠. 실제 .docx·.pptx 렌더링은 별개의 후처리(구조화된 결과를 파일 생성 코드에 넘김)임.
3줄 요약
  1. 요청에 따라 문서를 만들어 주는 작성 비서 에이전트임.
  2. 출력이 요청마다 달라 여러 도구를 갖춰 고르게 함.
  3. 도구·RAG·메모리가 모두 엔진으로 들어감.
11-2

문서 작업 도구들

문서 작업을 단일 기능 도구로 나눔(목차·초안·구성·요약). docstring의 "언제 쓰는지"가 라우팅을 좌우함(Part 7).

인터랙티브 · 요청 → 도구 선택

요청을 눌러, 비서가 어떤 작업 도구를 고르는지 봄.

✍️

📊

📝
summarize_text
3줄 요약
  1. 문서 작업을 단일 기능 도구로 나눔.
  2. docstring "언제 쓰는지"가 라우팅을 좌우함.
  3. 도구가 작은 체인을 품어 구조화 결과를 낼 수 있음.
11-3

RAG로 근거 붙이기 — 도구의 협력

근거가 필요한 대목엔 search_reference(RAG)로 자료를 찾아, 그 사실을 작성 도구에 반영함. 검색 도구(근거 수집) + 작성 도구(글쓰기)가 협력하는 것이 사실 기반 문서의 핵심임.

인터랙티브 · "데이터 근거로 단락 써줘"

"다음 단계"를 누르며, 에이전트가 자료를 먼저 검색하고 그 근거로 글을 쓰는 협력 과정을 따라감.

1🔍 search_reference("생성형 AI 도입 효과") 호출 — 근거가 필요하다고 판단
↓
2👁 자료 검색 결과: "도입률 2배 증가, 반복 업무 30% 단축"
↓
3✍️ draft_section — 그 수치를 인용해 단락 작성
↓
4📄 "2025년 도입률은 2배 증가했고, 반복 업무 시간이 30% 단축되었다 [자료 근거]"
📌 참고  system_prompt에 "수치·사실은 반드시 search_reference로 확인하라"를 명시하면 추측을 막음. 근거 출처를 표기하게 하면 문서 신뢰도가 올라감.
3줄 요약
  1. RAG를 도구로 붙이면 에이전트가 자료에서 근거를 가져옴.
  2. 검색 도구와 작성 도구가 협력해 사실 기반 문서를 만듦.
  3. "사실은 검색으로 확인"을 지침에 명시해 추측을 막음.
11-4

메모리로 이어 작업하기

문서 작업은 한 번에 안 끝남. 같은 thread로 "개요 → 섹션 추가 → 슬라이드"를 이어가면, 메모리 덕에 "거기에", "이 보고서를" 같은 후속 지시가 통함(Part 10).

인터랙티브 · 누적되는 문서 작업

"다음 턴"을 누르며, 같은 thread에서 문서가 점점 완성되어 가는 과정을 봄.

thread_id: "report-1"
3줄 요약
  1. 메모리로 이어지는 문서 편집이 가능해짐.
  2. "거기에", "이 보고서를" 같은 후속 지시가 통함.
  3. 도구·RAG·메모리가 한 흐름에서 함께 작동하는 완성형임.
11-5

한계 — 자유 비서 vs 결재 라인

완성한 에이전트는 흐름을 모델의 자유 판단에 맡김. 대부분의 일엔 충분하지만, 회사의 결재 라인처럼 "반드시 거쳐야 하는 정해진 절차"가 있는 일에는 버거워함.

인터랙티브 · 이 요구를 자유 에이전트로 보장할 수 있을까

요구를 바꿔, 자유 판단(에이전트)으로 보장되는지 판정해봄.

📌 참고  에이전트(자유 판단)와 그래프(명시적 제어)는 대립이 아니라 계층임. 큰 절차는 LangGraph로 짜고, 각 단계 안의 자유 판단만 에이전트에 맡김. 오늘 만든 비서는 버려지지 않고 그래프의 한 노드가 됨.
3줄 요약
  1. 완성한 에이전트는 흐름을 모델의 자유 판단에 맡김.
  2. "정해진 절차·반복·사람 개입"은 보장하기 어려움.
  3. 그때 필요한 것이 LangGraph로 흐름을 직접 그리는 것(고급).
