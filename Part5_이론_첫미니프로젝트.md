# Part 5. 첫 미니 프로젝트 — 상품 마케팅 카드 생성기

> **이 파트의 목표**
> Part 1~4에서 배운 네 부품 — **모델 · 프롬프트 · LCEL · 파서** — 을 하나로 합쳐, 실제로 작동하는 체인을 처음부터 끝까지 만들어 봄. 새 개념을 배우는 파트가 아니라, 배운 것을 **통합하고 손에 익히는** 졸업 작품임. 그리고 마지막에 "이 체인이 못 하는 것"을 마주하며, 왜 중급에서 **에이전트**가 필요해지는지 그 다리를 놓음.

---

## 5-1. 무엇을 만들 것인가 — 프로젝트 설계

### 개념

만들 것은 **상품 마케팅 카드 생성기**임. 리리마켓에 신상품이 들어올 때마다, 상품명만 넣으면 마케팅에 바로 쓸 수 있는 **구조화된 카드**가 나오는 체인임.

| 항목 | 내용 |
|---|---|
| **입력** | 상품명 (예: `"베이지 케이블 니트"`) |
| **출력** | 헤드라인 · 설명 · 타겟 고객 · 해시태그 · 가격대 (구조화된 객체) |
| **쓰는 부품** | ChatPromptTemplate(2) · LCEL 파이프(3) · with_structured_output(4) |

### 비유

이 프로젝트는 **자판기 조립**과 같음. Part 1~4에서 부품(모터·동전 인식기·진열대·배출구)을 하나씩 만들었다면, 이제 그 부품들을 한 상자에 넣고 배선을 연결해 "동전을 넣으면 음료가 나오는" 완성된 기계로 만드는 단계임.

### 원리 — 설계가 먼저다

코드를 짜기 전에 **입력과 출력을 먼저 못 박는 것**이 핵심임. "무엇이 들어가고 무엇이 나오는가"가 명확하면, 그 사이를 잇는 체인은 자연스럽게 따라옴. 우리의 설계는 이렇게 정리됨.

```
상품명(str)
   │
   ▼
[ ChatPromptTemplate ]  ── 상품명을 마케팅 카드 요청 프롬프트로
   │
   ▼
[ model.with_structured_output(MarketingCard) ]  ── 구조 강제 + 검증
   │
   ▼
MarketingCard 객체(headline, description, target, hashtags, price_range)
```

### 📌 참고
> 좋은 프로젝트의 90%는 "출력 구조를 어떻게 정의하느냐"에서 갈림. 출력이 명확하면 프롬프트도 모델도 그 틀에 맞춰짐. 그래서 다음 절에서 가장 먼저 **출력 구조(Pydantic 모델)**부터 만듦.

### 실무 포인트
- 실무에서도 "입력/출력 명세 → 그 사이 파이프라인" 순서로 설계함. 중간 구현부터 손대면 길을 잃기 쉬움.
- 입력은 단순하게(상품명 하나), 출력은 풍부하게(여러 필드) 잡으면 "적은 입력에서 많은 가치"라는 LLM 앱의 전형이 됨.

### 3줄 요약
1. 만들 것은 상품명 → 구조화된 마케팅 카드 생성기임.
2. 코드보다 입력·출력 설계를 먼저 못 박음.
3. 출력 구조가 명확하면 프롬프트와 모델이 그에 맞춰짐.

## 5-2. 1단계 — 출력 구조 설계 (Pydantic)

### 개념

가장 먼저 "카드가 어떤 칸으로 이뤄지는가"를 Pydantic 모델로 정의함(Part 4). 이것이 프로젝트의 **설계도**가 됨.

```python
from pydantic import BaseModel, Field

class MarketingCard(BaseModel):
    headline: str = Field(description="시선을 끄는 한 줄 카피")
    description: str = Field(description="상품 설명 2~3문장")
    target: str = Field(description="핵심 타겟 고객층")
    hashtags: list[str] = Field(description="SNS 해시태그 3~5개")
    price_range: str = Field(description="추천 가격대 (예: 5만원대)")
```

### 비유

Pydantic 모델은 **빈 카드 양식**임. 헤드라인 칸, 설명 칸, 해시태그 칸이 미리 그어져 있고, 모델은 이 칸을 채우기만 하면 됨. 양식이 정해져 있으니 어떤 상품이 들어와도 결과 모양이 일정함.

### 원리

각 필드의 `Field(description=...)`은 단순한 주석이 아니라 **모델에게 주는 지시**임. `with_structured_output`이 이 설명들을 모델에게 전달해, 각 칸을 무엇으로 채울지 알려줌. 설명이 구체적일수록(예: "해시태그 3~5개") 결과가 의도에 가까워짐.

### 📌 참고
> 필드 타입을 잘 고르면 출력이 더 안전해짐. 예를 들어 카드 톤을 "감성/실용/유머" 셋 중 하나로 제한하고 싶으면 `Literal["감성","실용","유머"]`를 쓰면 됨(Part 4). 리스트가 필요한 곳엔 `list[str]`을 써서 해시태그가 항상 리스트로 오게 함.

### 실무 포인트
- 필드명은 영어로(코드에서 다루기 쉬움), 설명은 한국어로(모델이 한국어 맥락을 이해) 쓰는 조합이 실무에서 무난함.
- 처음부터 필드를 너무 많이 만들지 말 것. 핵심 5개 정도로 시작해, 필요하면 늘리는 편이 안정적임.

### 3줄 요약
1. 프로젝트의 첫 단계는 출력 구조를 Pydantic 모델로 정의하는 것임.
2. `Field(description=...)`은 모델에게 각 칸을 무엇으로 채울지 알려주는 지시임.
3. 타입(`Literal`, `list[str]`)을 잘 고르면 출력이 더 안전해짐.

## 5-3. 2단계 — 체인 조립

### 개념

설계도(출력 구조)가 있으니, 이제 프롬프트와 모델을 파이프로 이어 체인을 완성함. Part 1~4가 한 줄에 모두 모임.

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_google_genai import ChatGoogleGenerativeAI

model = ChatGoogleGenerativeAI(model="gemini-3-flash", temperature=0.7)

prompt = ChatPromptTemplate.from_messages([
    ("system", "너는 리리마켓의 베테랑 마케터야. 20~40대 여성 고객을 잘 안다."),
    ("human", "다음 상품의 마케팅 카드를 만들어줘: {product}"),
])

# 핵심 한 줄 — 프롬프트 | 구조화된 모델
chain = prompt | model.with_structured_output(MarketingCard)

card = chain.invoke({"product": "베이지 케이블 니트"})
print(card.headline)
print(card.hashtags)
```

### 비유

이 한 줄(`prompt | model.with_structured_output(...)`)이 **자판기의 배선을 연결하는 순간**임. 따로 놀던 부품들이 하나의 흐름으로 묶여, "상품명 투입 → 카드 배출"이 작동하기 시작함.

### 원리

데이터는 이렇게 흐름.

1. `{"product": "베이지 케이블 니트"}`가 `prompt`로 들어가 System+Human 메시지가 됨.
2. 그 메시지가 `with_structured_output`이 입혀진 모델로 흘러감.
3. 모델은 `MarketingCard` 구조에 맞춰 답하고, 검증된 **`MarketingCard` 객체**가 체인의 결과로 나옴.

여기엔 Part 2(프롬프트)·Part 3(파이프)·Part 4(구조화 출력)가 모두 녹아 있음. 초급 전체가 이 한 체인으로 수렴함.

### 📌 참고
> `temperature`를 마케팅 카피처럼 다양성이 필요한 작업이라 `0.7` 정도로 둠(Part 1). 만약 카드를 매번 똑같이 뽑고 싶다면 `0`으로 내리면 됨. 작업 성격에 맞춘 손잡이 조절이 여기서 실전으로 쓰임.

### 실무 포인트
- 체인을 한 번 조립(`chain = ...`)해 두고, 상품마다 `.invoke()`로 재사용함.
- 결과가 객체라 `card.headline`, `card.hashtags`처럼 바로 코드에서 다룰 수 있음 — 화면 출력, DB 저장, 다른 API 전달이 자유로움.

### 3줄 요약
1. `prompt | model.with_structured_output(MarketingCard)` 한 줄이 프로젝트의 심장임.
2. 입력 상품명이 프롬프트→모델을 거쳐 검증된 카드 객체로 나옴.
3. 이 한 체인에 Part 2·3·4가 모두 녹아 있음.

## 5-4. 3단계 — 확장: 배치 처리와 병렬 변형

### 개념

체인 하나가 작동하면, Part 3에서 배운 `.batch()`와 `RunnableParallel`로 손쉽게 확장됨. 상품 수십 개를 한꺼번에 처리하거나, 한 상품에 대해 여러 버전의 카드를 동시에 뽑을 수 있음.

```python
# (1) 여러 상품 한꺼번에 — batch
products = [{"product": "베이지 니트"}, {"product": "트렌치 코트"}, {"product": "린넨 셔츠"}]
cards = chain.batch(products)   # 카드 객체 리스트

# (2) 한 상품, 두 톤을 동시에 — parallel
from langchain_core.runnables import RunnableParallel

감성_chain = ChatPromptTemplate.from_template("{product}의 감성적 헤드라인 한 줄") | model | StrOutputParser()
실용_chain = ChatPromptTemplate.from_template("{product}의 실용적 장점 한 줄") | model | StrOutputParser()

variants = RunnableParallel(감성=감성_chain, 실용=실용_chain)
result = variants.invoke({"product": "울 코트"})
# → {'감성': '...', '실용': '...'}
```

### 비유

기본 체인이 자판기 한 대라면, `.batch()`는 **여러 동전을 한 번에 넣어 음료를 줄줄이 받는 것**이고, `RunnableParallel`은 **한 동전으로 여러 진열대에서 동시에 음료를 뽑는 것**임. 부품을 새로 만들지 않고도 기능이 확장됨.

### 원리

`.batch()`는 같은 체인에 여러 입력을 병렬로 흘려 결과 리스트를 돌려줌 — 반복문보다 빠름. `RunnableParallel`은 같은 입력을 여러 체인에 동시에 흘려 결과를 딕셔너리로 모음. 둘 다 "체인은 다시 Runnable"이라는 Part 3의 성질 덕분에 가능함. 즉 **새 개념 없이, 배운 도구의 조합만으로** 프로젝트가 커짐.

### 📌 참고
> 실무 마케팅 자동화라면 여기에 결과를 표(스프레드시트)나 DB로 저장하는 단계가 붙음. 카드가 객체라 `card.model_dump()`로 딕셔너리화해 손쉽게 저장됨. 이런 "뒷단 연결"은 도구(Tool)와 에이전트를 배우는 중급에서 본격화됨.

### 실무 포인트
- 상품 카탈로그 전체에 카드를 다는 작업은 `chain.batch(전체상품리스트)` 한 줄로 끝남.
- A/B 테스트용으로 여러 톤의 카피가 필요하면 `RunnableParallel`로 한 번에 뽑아 비교함.

### 3줄 요약
1. `.batch()`로 여러 상품을, `RunnableParallel`로 여러 버전을 한꺼번에 처리함.
2. 새 개념 없이 Part 3의 도구 조합만으로 프로젝트가 확장됨.
3. 결과가 객체라 저장·전달이 쉬워, 자동화로 자연스럽게 이어짐.

## 5-5. 초급 졸업 — 그리고 체인의 한계

### 개념

축하함. 여기까지 왔다면 LangChain 초급의 모든 것을 갖췄음. 우리가 만든 체인은 강력하지만, 한 가지 본질적인 특징이 있음 — **항상 정해진 길로만 흐름.**

```
입력 → 프롬프트 → 모델 → 파서 → 출력
       (언제나 똑같은 순서, 똑같은 경로)
```

### 비유 — 체인은 "직진 도로"임 (Part 0 회상)

Part 0에서 LCEL을 "직진 도로"에 비유했음. 우리의 마케팅 카드 체인은 어떤 상품이 들어와도 같은 길을 감 — 프롬프트를 거쳐 모델을 거쳐 카드를 냄. 갈림길이 없으니 빠르고 예측 가능함. 대부분의 작업에는 이걸로 충분함.

### 원리 — 그런데 이런 상황이라면?

다음 요구가 들어왔다고 해보자.

> "마케팅 카드를 만들되, **상품이 신상이면 경쟁사 가격을 웹에서 검색**해서 가격대를 정하고, **재고가 없으면 카드 대신 입고 알림 문구**를 만들어줘."

우리의 체인으로는 이걸 깔끔히 풀 수 없음. 왜냐하면 이건 **"상황을 보고 다음 행동을 그때그때 골라야 하는"** 일이기 때문임.

- 검색을 할지 말지 → 모델이 **판단**해야 함
- 카드를 만들지 알림을 만들지 → **분기**가 필요함

정해진 길(체인)로는 이런 동적 판단을 담을 수 없음. 여기서 필요한 것이 바로 **에이전트** — 도구를 보고 스스로 다음 행동을 고르는 실행 단위임(Part 0의 "내비게이션을 든 운전자"). 이것이 중급의 문임.

### 📌 참고
> 체인이 부족해서 에이전트를 쓰는 게 아님. **순서가 고정된 일은 여전히 체인이 더 빠르고 안전함.** 에이전트는 "동적 판단·도구 사용·분기"가 필요한 순간에만 꺼내는 도구임. 이 경계 감각이 중급에서 가장 중요한 배움임.

### 실무 포인트
- 새 기능 요구가 들어오면 먼저 자문할 것: "이건 정해진 순서인가, 매번 판단이 필요한가?" 전자면 체인, 후자면 에이전트.
- 초급에서 만든 체인은 버려지지 않음 — 중급의 에이전트도 내부에서 이런 체인들을 도구처럼 활용함.

### 3줄 요약
1. 우리가 만든 체인은 강력하지만 항상 정해진 길로만 흐름(직진 도로).
2. "검색할지 판단·분기"가 필요한 일은 체인으로 담을 수 없음.
3. 그때 필요한 것이 에이전트이며, 그것이 중급의 시작임.

## Part 5 마무리 — 초급의 끝, 중급의 문 앞에서

Part 5에서는 배운 모든 것을 하나로 합쳤음. 정리하면:

- 입력·출력을 먼저 못 박는 프로젝트 설계 (5-1)
- Pydantic으로 출력 구조(설계도)를 정의 (5-2)
- `프롬프트 | 모델.with_structured_output(...)` 한 줄로 체인 완성 (5-3)
- `.batch()`·`RunnableParallel`로 확장 (5-4)
- 체인의 한계 → 에이전트의 필요성 (5-5)

**초급 전체 회고**: 우리는 모델(1) → 프롬프트(2) → LCEL(3) → 파서(4) → 통합 프로젝트(5)를 거치며, "LLM에 말을 걸고, 질문을 설계하고, 블록을 잇고, 결과를 다듬어, 작동하는 앱을 만드는" 전 과정을 손에 익혔음. 이제 우리가 만든 모든 것은 **정해진 길을 가는 체인**이었음.

**중급(Part 6~)에서는** 그 길에서 벗어나, 스스로 판단하는 흐름 — **에이전트**로 들어감. 첫 발인 **Part 6**은 "체인 vs 에이전트"로, 오늘 5-5에서 마주한 그 경계를 정면으로 다룸. 무엇이 정해진 길이고 무엇이 동적 판단인지, 그 결정적 차이를 손에 잡히게 익힘.

> 함께 제공되는 **Part 5 실습 노트북**에서는 마케팅 카드 생성기를 처음부터 끝까지 직접 만들어 봄 — Pydantic 카드 정의 → 체인 조립 → 여러 상품 배치 처리 → 두 톤 병렬 생성까지. 초급의 모든 부품을 한 프로젝트에서 다시 만나며, 마지막에 "이 체인으로는 풀기 어려운 요구"를 직접 마주하며 중급으로 넘어감.
