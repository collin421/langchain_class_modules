# Part 12. 왜 LangGraph인가

> **이 파트의 목표 (고급의 시작)**
> Part 11 마지막에서 "흐름을 모델의 자유 판단에 맡기는" 에이전트의 한계를 마주했음. 이제 흐름의 **운전대를 우리가 직접 쥠**. LangGraph는 노드와 엣지로 흐름을 직접 그리는 도구임 — 사실 우리가 만든 에이전트도 이미 그 위에서 돌고 있었음. 이 파트는 아직 그래프를 직접 코딩하지 않음(그건 Part 13). 대신 **"언제 `create_agent`를 넘어 직접 그래프를 그려야 하는가"**의 판단 전환을 확실히 함.

---

## 12-1. create_agent의 편리함과 그 천장

### 개념

`create_agent`는 강력하고 편리함. 모델·도구·프롬프트만 주면, 알아서 "생각→행동→관찰" 루프를 돌려줌(Part 9). 대부분의 에이전트 작업은 이걸로 충분함. 하지만 그 편리함에는 **천장**이 있음 — **흐름을 우리가 정밀하게 제어할 수 없음.**

### 비유

`create_agent`는 **자동 모드만 있는 드론**과 같음. "저쪽으로 가"라고 목표만 주면 알아서 날아감 — 대부분의 비행엔 편리함. 하지만 "이 지점에서 반드시 멈춰 사진을 찍고, 배터리가 30% 이하면 무조건 복귀하고, 착륙 전엔 조종사 승인을 받아라" 같은 **정밀한 비행 계획**을 강제하려면, 자동 모드만으로는 부족함. 수동 조종(직접 흐름 설계)이 필요해짐.

### 원리

`create_agent`의 흐름은 고정된 한 가지 패턴임 — "모델이 도구를 고르고, 도구 결과를 보고, 다시 고르고… 답이 되면 멈춘다." 이 패턴 자체는 못 바꿈. 그래서:

- "이 단계 다음엔 반드시 저 단계" 같은 **고정 순서**를 강제하기 어려움
- "조건 A면 이쪽, B면 저쪽" 같은 **명시적 분기**를 직접 제어하기 어려움
- "여기서 멈추고 사람을 기다려라" 같은 **개입 지점**을 박기 어려움

즉 `create_agent`는 "무엇을 할지"를 모델에 맡기는 데 최적화돼 있고, "어떤 순서·조건으로 흐를지"를 우리가 못 박는 데는 한계가 있음.

### 📌 참고
> 이건 `create_agent`가 부족하다는 뜻이 아님. 대부분의 작업은 자유 판단이 더 낫고 빠름. 천장에 부딪히는 건 "정해진 절차·복잡한 흐름"이 필요한 특정 순간임. 그 순간을 알아보는 것이 이 파트의 핵심임.

### 실무 포인트
- `create_agent`로 시작하는 것이 정석임. 천장에 부딪히기 전까지는 그래프가 과함.
- "에이전트가 가끔 절차를 건너뛴다 / 순서가 꼬인다 / 멈춰야 할 때 안 멈춘다"가 반복되면, 천장에 닿았다는 신호임.

### 3줄 요약
1. `create_agent`는 편리하지만 흐름은 고정된 한 패턴임.
2. 고정 순서·명시적 분기·개입 지점을 강제하기 어려움.
3. 대부분엔 충분하나, 정해진 절차가 필요한 순간 천장에 닿음.

## 12-2. LangGraph란 — 흐름을 직접 그리다

### 개념

**LangGraph**는 작업의 흐름을 **노드(node)와 엣지(edge)로 직접 그리는** 도구임. 노드는 "하나의 작업 단계", 엣지는 "단계 사이의 연결"임. 우리가 그래프를 그리면, 실행은 그 그림을 따라 흐름.

```
   [시작] → (초안 작성) → (검토) → 분기 → (발행)
                            └→ (수정) → 다시 (검토)로
```

### 비유

LangGraph는 **순서도(flowchart)를 그려 그대로 실행시키는 것**과 같음. 우리가 종이에 "이 일 다음 저 일, 여기서 갈라지고, 저기서 되돌아온다"를 그리면, 그 순서도가 실제로 작동함. 흐름이 우리 손에 그려진 그대로 보장됨 — 모델의 기분에 따라 바뀌지 않음.

### 원리 — 이미 그 위에서 돌고 있었다

놀라운 사실: **우리가 만든 에이전트는 처음부터 LangGraph 위에서 돌고 있었음**(Part 0·6·9·10). `create_agent`는 사실 "agent 노드 ↔ tools 노드를 잇고, 조건에 따라 반복하는" **미리 만들어진 그래프**임. 메모리(체크포인터)도 LangGraph의 상태 관리였음(Part 10).

즉 우리는 LangGraph를 **간접적으로 써 왔음.** 고급에서 하는 일은 그 미리 만들어진 그래프를 쓰는 대신, **노드와 엣지를 우리가 직접 정의**하는 것임 — 흐름을 손수 그리는 것.

### 📌 참고
> 그래서 LangGraph는 `create_agent`의 "경쟁자"가 아니라 그 **토대**임. create_agent = LangGraph로 만든 편리한 기성품, 직접 그래프 작성 = 맞춤 제작. 둘 다 같은 엔진(LangGraph 런타임) 위에 있음.

### 실무 포인트
- "에이전트의 내부 루프"를 이해하면 그래프 설계가 쉬워짐 — 이미 그 패턴을 써 봤기 때문임.
- 복잡한 시스템은 보통 "큰 흐름은 직접 그린 그래프, 일부 노드는 create_agent"로 섞음(12-4).

### 3줄 요약
1. LangGraph는 노드(작업)·엣지(연결)로 흐름을 직접 그리는 도구임.
2. 그린 그대로 실행이 보장됨 — 모델 기분에 안 흔들림.
3. 우리 에이전트도 이미 LangGraph 위에 있었음 — create_agent는 기성 그래프임.

## 12-3. 그래프가 필요한 네 가지 신호

### 개념

언제 `create_agent`를 넘어 직접 그래프를 그려야 할까? 다음 **네 신호** 중 하나라도 강하게 나타나면 LangGraph를 고려함.

| 신호 | 설명 | 예시 |
|---|---|---|
| **① 반복(루프)** | 조건을 만족할 때까지 단계를 되풀이 | "검토 통과까지 수정→재검토 반복" |
| **② 조건 분기** | 상황에 따라 명시적으로 다른 길 | "재고 있으면 주문, 없으면 알림" |
| **③ 장기 상태** | 여러 단계에 걸쳐 정보를 누적·관리 | "초안·검토의견·승인여부를 단계마다 갱신" |
| **④ 사람 개입** | 특정 지점에서 반드시 멈추고 사람 확인 | "발행 전 반드시 승인 받기" |

### 비유

이 네 신호는 **"자동 모드로는 안 되겠다"는 경고등**임. 한두 개 깜빡이는 정도면 에이전트로 버틸 수 있지만, 여러 개가 강하게 켜지면 수동 조종(직접 그래프)으로 전환할 때임.

### 원리

이 신호들의 공통점은 **"흐름 자체를 보장해야 한다"**는 것임. 자유 판단(에이전트)은 "대체로" 잘하지만 "반드시"를 보장하진 못함. 반복이 정확히 멈춰야 하고, 분기가 규칙대로 갈라져야 하고, 상태가 빠짐없이 누적돼야 하고, 사람 개입이 절대 생략되면 안 되는 일 — 이런 "반드시"가 필요한 순간이 그래프의 영역임. Part 13~14에서 이 네 신호를 각각 그래프로 구현함.

### 📌 참고
> 네 신호는 독립적이지 않고 함께 나타나는 경우가 많음. 예를 들어 "초안→검토→(통과 분기/반려 루프)→발행 전 승인"은 ②분기 ①루프 ④개입 ③상태가 모두 얽힌 전형적인 그래프 사례임(Part 14의 human-in-the-loop).

### 실무 포인트
- 새 요구를 받으면 네 신호를 체크리스트처럼 점검함. 강하게 켜지면 그래프, 아니면 에이전트.
- "반드시", "절대", "순서대로", "~할 때까지"라는 말이 요구에 들어 있으면 그래프 신호일 때가 많음.

### 3줄 요약
1. 그래프 신호: ①반복 ②조건 분기 ③장기 상태 ④사람 개입.
2. 공통점은 "흐름 자체를 반드시 보장해야 함"임.
3. 여러 신호가 강하게 켜지면 직접 그래프로 전환함.

## 12-4. 에이전트 vs 그래프 — 그리고 섞기

### 개념

핵심 대비를 정리함.

| | create_agent (자유 판단) | LangGraph (명시적 제어) |
|---|---|---|
| 흐름을 정하는 주체 | 모델 | 우리(설계자) |
| 강점 | 유연함, 빠른 구축 | 보장된 흐름, 정밀 제어 |
| 약점 | 절차 보장 어려움 | 설계 비용, 경직될 수 있음 |
| 적합 | 열린 작업, 도구 자유 선택 | 정해진 절차, 복잡한 흐름 |

### 비유

에이전트는 **숙련된 즉흥 연주자**, 그래프는 **악보를 따르는 오케스트라**임. 즉흥 연주는 유연하고 창의적이지만 매번 다름. 악보는 정확히 같은 곡을 보장하지만 자유롭진 않음. 그리고 **현실의 좋은 음악은 둘을 섞음** — 정해진 악보(그래프) 안에 즉흥 솔로(에이전트) 구간을 두는 식임.

### 원리 — 둘은 계층으로 섞인다

가장 중요한 실무 통찰: **에이전트와 그래프는 대립이 아니라 계층**임. 잘 설계된 시스템은 보통 이렇게 섞음.

```
[큰 흐름 = 직접 그린 그래프]
  ├─ 노드 A: 정해진 처리 (체인)
  ├─ 노드 B: 자유 판단이 필요한 부분 (create_agent를 노드로!)
  ├─ 분기: 조건에 따라 길 나눔
  └─ 노드 C: 사람 개입 지점
```

즉 **그래프의 한 노드 안에 에이전트를 넣을 수 있음.** Part 11에서 만든 문서 비서가 버려지지 않고, "검토 절차 그래프"의 한 노드("초안 작성" 노드)가 되는 식임. 큰 절차는 그래프로 보장하고, 그 안의 열린 작업은 에이전트에 맡김.

### 📌 참고
> 이 "계층적 결합"이 고급의 핵심 설계 철학임. 초급(체인)·중급(에이전트)·고급(그래프)은 위계가 아니라 **도구 상자**임 — 상황에 맞게 꺼내 조합함. 그래프 안에 에이전트, 에이전트 안에 체인(도구), 이런 중첩이 자연스러움.

### 실무 포인트
- "전부 그래프" 또는 "전부 에이전트"는 대개 나쁜 설계임. 부분마다 맞는 도구를 씀.
- 그래프로 흐름의 뼈대를 잡되, 판단·생성이 필요한 노드는 에이전트/체인으로 채우는 것이 실전 패턴임.

### 3줄 요약
1. 에이전트=자유 판단(유연), 그래프=명시적 제어(보장).
2. 둘은 대립이 아니라 계층 — 그래프의 노드에 에이전트를 넣음.
3. 큰 절차는 그래프로 보장, 열린 작업은 에이전트로 — 섞어 씀.

## 12-5. 언제 무엇을 — 판단 기준과 그래프 남용 경계

### 개념

중급에서 "에이전트 남용 경계"를 배웠듯(Part 6), 고급에서도 **"그래프 남용 경계"**가 똑같이 중요함. 그래프는 강력하지만 공짜가 아님 — 설계·유지 비용이 크고, 잘못 쓰면 시스템이 경직됨.

**그래프가 맞는 신호** (12-3 재확인)
- 흐름을 반드시 보장해야 함(반복·분기·상태·개입)
- 규정 준수·승인·감사 추적이 중요함
- 여러 단계가 복잡하게 얽혀 있음

**그래프가 과한 신호**
- 단순 변환이거나(→ 체인), 열린 작업이거나(→ 에이전트)
- 흐름이 "대체로 맞으면" 충분함
- 빠른 프로토타입이 필요함

### 비유

그래프를 모든 일에 쓰는 것은 **간단한 메모에도 정식 결재 시스템을 도입하는 것**과 같음. 절차가 필요한 일엔 결재 시스템이 안전을 주지만, 모든 일에 적용하면 느리고 답답해짐. **도구의 무게를 일의 무게에 맞추는 것**이 좋은 설계임.

### 원리 — 올라가는 사다리

실무 정석은 **가장 가벼운 것부터 시작해, 필요할 때만 올리는 것**임.

```
체인(정해진 변환)  →  에이전트(자유 판단 필요)  →  그래프(흐름 보장 필요)
       가벼움  ─────────────────────────────────→  무거움
```

체인으로 시작 → 동적 판단이 필요해지면 에이전트 → 흐름을 반드시 보장해야 하면 그래프. 거꾸로 "일단 그래프부터"는 대개 과설계임. 각 단계는 앞 단계를 품을 수 있으므로(그래프 안에 에이전트, 에이전트 안에 체인), 위로 올라가도 아래를 버리지 않음.

### 📌 참고
> "이 시스템 왜 이렇게 복잡해요?"라는 질문에 "흐름을 반드시 보장해야 해서요"라고 답할 수 있어야 함. "최신이라서", "멋있어서"는 위험 신호임 — 중급의 에이전트 남용 경계와 같은 원칙임.

### 실무 포인트
- 면접·설계 리뷰에서 "왜 그래프인가"를 네 신호(반복·분기·상태·개입)로 설명할 수 있어야 함.
- 의심스러우면 더 가벼운 도구로 시작하고, 천장에 부딪힐 때 올림. 미리 올리지 말 것.

### 3줄 요약
1. 그래프도 남용 경계가 있음 — 설계·유지 비용이 큼.
2. 흐름 보장이 꼭 필요할 때만 쓰고, 아니면 체인/에이전트로.
3. 사다리: 체인→에이전트→그래프, 가벼운 것부터 필요할 때만 올림.

- 에이전트 vs 그래프, 그리고 계층적 섞기 (12-4)
- 판단 기준과 그래프 남용 경계, 가벼운 것부터의 사다리 (12-5)

이제 우리는 **언제·왜** 직접 그래프를 그려야 하는지 알게 됐음. **Part 13**에서는 드디어 **`StateGraph`로 그래프를 직접 짬** — 상태(state)를 정의하고, 노드를 만들고, 엣지로 잇고, 조건 분기를 넣어 컴파일하고 실행함. Part 10에서 본 "상태"를 이제 우리 손으로 설계하며, 12-3의 네 신호 중 ①반복 ②분기 ③상태를 직접 구현함.

> 함께 제공되는 **Part 12 실습 노트북**에서는 같은 작업("초안→검토→통과 시 발행, 반려 시 수정 반복")을 `create_agent`로 시켰을 때 흐름이 보장되지 않는 것을 관찰하고, 그래프로 그렸을 때 흐름이 보장되는 차이를 개념적으로 비교함. 본격적인 `StateGraph` 코딩은 Part 13에서 시작하므로, 이 노트북은 "왜 그래프가 필요한가"를 체감하는 데 집중함.
