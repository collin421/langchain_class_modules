LangChain 에이전트 마스터 · Part 13

StateGraph 직접 짜기

Part 12에서 "왜"를 다뤘다면, 이제 그래프를 손으로 짬. 상태를 정의하고, 노드를 만들고, 엣지로 잇고, 조건 분기를 넣어 컴파일·실행함. 네 신호 중 ①반복 ②분기 ③상태를 직접 구현함.

🔴 고급 · 13/18 StateGraph 예상 50분
13-1

상태 정의 — 덮어쓰기 vs 누적

그래프의 출발점은 상태(state)임 — 여러 부서를 거치는 결재 서류 봉투. TypedDict로 칸을 정의함. 핵심 구분: 노드가 키를 갱신할 때 기본은 덮어쓰기, 누적하려면 리듀서(Annotated[list, operator.add])를 붙임.

class State(TypedDict):
    draft: str                              # 덮어쓰기
    history: Annotated[list, operator.add]  # 누적(리듀서)
인터랙티브 · 같은 갱신, 다른 결과

"단계 실행"을 누를 때마다 노드가 같은 갱신을 보냄. 리듀서 유무에 따라 결과가 어떻게 달라지는지 봄.

📌 참고  상태는 작고 명확하게. 단계 간 꼭 전달할 정보만 담음. 누적이 필요한 것(기록·메시지·결과)만 리듀서를 붙이고, 나머지는 덮어쓰기로 둠. 메시지엔 전용 리듀서 add_messages를 씀.
3줄 요약
  1. 상태는 그래프가 들고 다니며 갱신하는 데이터(TypedDict)임.
  2. 기본은 덮어쓰기, 누적은 리듀서(Annotated[list, operator.add]).
  3. 상태는 작고 명확하게 — 꼭 필요한 정보만.
13-2

노드 — 상태 받아 부분 갱신

노드는 "하나의 작업 단계" — 상태를 받아 갱신할 부분만 딕셔너리로 돌려주는 함수임. 결재 봉투를 받은 담당자가 자기 칸만 채워 넘기듯, 안 돌려준 키는 그대로 유지됨.

인터랙티브 · 노드가 하는 일

"실행"을 누르면 review_node가 입력 상태를 받아, review·history만 갱신해 돌려주는 과정을 봄.

입력 state
topic: "AI 도입"
draft: "...초안..."
review: ""
history: [초안]
→
review
_node
→
반환 (갱신할 키만)
📌 참고  노드 안에서 무엇을 하든 자유 — 모델 호출, 도구 실행, 계산, 심지어 create_agent 에이전트 호출도 가능(Part 12 "노드 안 에이전트"). "상태→부분 갱신" 규칙만 지키면 됨. 상태를 직접 수정하지 않는 순수 함수라 테스트가 쉬움.
3줄 요약
  1. 노드는 "상태 받아 갱신할 부분을 딕셔너리로 반환"하는 함수임.
  2. 갱신할 키만 반환하면 나머지는 유지됨(부분 갱신).
  3. 순수 함수라 예측·테스트가 쉬움.
13-3

엣지로 잇기 — 흐름 그리기

add_node로 등록, add_edge(A, B)로 "A 다음 항상 B"를 고정 연결. START·END가 진입·종료점. 이렇게 이은 흐름은 무조건 그 순서 — 흐름 보장의 실체임.

인터랙티브 · 그래프 골격 그리기

"엣지 추가"를 누르며 START → 초안 → 검토 → END를 하나씩 연결해, 그래프가 완성되는 걸 봄.

📌 참고  여기까지는 일직선 흐름. 진짜 힘은 다음 절의 조건 분기와 루프에서 나옴 — "검토 결과에 따라 갈라지고, 필요하면 되돌아가는" 흐름임.
3줄 요약
  1. add_node 등록, add_edge로 "A 다음 B" 고정 연결.
  2. START/END가 진입점·종료점.
  3. 이은 흐름은 무조건 그 순서 — 흐름 보장의 실체임.
13-4

조건 분기 — 상태 따라 길 나누기

add_conditional_edges로 상태에 따라 다른 노드로 감. 검토팀이 봉투에 찍는 도장처럼 — "통과" 도장이면 발행팀, "반려"면 수정팀. 판단 규칙(route 함수)을 우리가 명시적으로 정함.

인터랙티브 · 라우팅 함수 돌려보기

검토 결과와 시도 횟수를 바꿔, route_after_review가 어느 노드로 분기하는지 봄.

📌 참고  라우팅 함수는 상태만 보고 결정하는 단순 함수로 유지함. END를 돌려주면 그래프가 끝남. 분기 목적지가 이전 노드가 되면 루프(다음 절)가 됨.
3줄 요약
  1. add_conditional_edges로 상태에 따라 분기함.
  2. 라우팅 함수가 상태 보고 다음 노드 결정(코드가 명시적).
  3. 규칙대로 반드시 갈라짐 — 분기 보장임.
13-5

컴파일·실행, 그리고 루프

compile()로 실행 그래프를 만들고 invoke로 실행. 조건 분기의 목적지를 이전 노드로 두면 루프(반복)가 완성됨 — 통과 도장을 받을 때까지 검토-수정을 빙빙 돎. 단, 종료 조건(횟수 제한)은 필수임.

인터랙티브 · 전체 그래프 실행

"한 단계씩"을 눌러, 초안→검토→(반려→수정→다시 검토)→통과→발행이 흐르는 과정을 따라감. 반려가 반복되면 루프가 돎.

📌 참고  루프는 무한 반복 위험이 있어 반드시 종료 조건(횟수 제한 등)을 둠. compile(checkpointer=InMemorySaver())로 직접 만든 그래프에도 메모리를 붙일 수 있음(Part 10) — Part 14의 사람 개입(멈췄다 재개)이 여기서 이어짐.
3줄 요약
  1. compile()로 실행 그래프를 만들고 invoke로 실행.
  2. 뒤→앞 엣지로 루프를 만들며 종료 조건을 반드시 둠.
  3. 컴파일은 연결을 검사하고, 체크포인터로 메모리도 붙임.
