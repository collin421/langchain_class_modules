# Part 16. 관측·평가 — LangSmith

> **이 파트의 목표**
> Part 15까지 체인부터 멀티에이전트까지 시스템을 만드는 법을 모두 익혔음. 그런데 만든 것이 **잘 작동하는지 어떻게 알까?** 에이전트가 엉뚱한 도구를 부르고, 멀티에이전트가 잘못 분배하고, 답이 틀려도 — **보지 못하면 고칠 수 없음.** 이 파트에서는 시스템의 내부를 들여다보는 **관측(observability)**과, 품질을 체계적으로 재는 **평가(evaluation)**를 배움. 도구는 LangChain 팀이 만든 **LangSmith**임.

---

## 16-1. 왜 관측·평가인가 — 블랙박스를 열다

### 개념

에이전트는 **블랙박스**가 되기 쉬움. 질문을 넣으면 답이 나오지만, 그 사이에 "어떤 도구를 왜 불렀는지, 어디서 시간이 걸렸는지, 왜 틀렸는지"가 보이지 않음. **관측**은 그 내부를 들여다보는 것이고, **평가**는 "얼마나 잘하는지"를 수치로 재는 것임. 둘 다 없으면 시스템을 개선할 수 없음.

### 비유

관측·평가가 없는 에이전트는 **계기판 없는 자동차**와 같음. 달리긴 하는데 속도도, 연료도, 엔진 상태도 모름 — 고장 나도 어디가 문제인지 알 수 없음. 관측은 **계기판과 블랙박스(주행 기록)**를 다는 것이고, 평가는 **정기 성능 검사**임. 이게 있어야 안심하고 운행하고, 문제를 미리 잡음.

### 원리

소프트웨어 공학의 기본 — **측정할 수 없으면 개선할 수 없음.** 에이전트는 특히 비결정적(같은 입력에도 다른 출력)이라, "느낌"으로는 품질을 알 수 없음. 그래서 두 가지가 필요함.

- **관측(추적)**: 매 실행의 모든 단계를 기록 → "이번에 무슨 일이 있었나"를 봄(디버깅).
- **평가**: 정해진 기준으로 여러 사례를 점수화 → "전반적으로 얼마나 잘하나"를 봄(품질 관리).

이 둘은 프로토타입을 **프로덕션급 시스템**으로 끌어올리는 분기점임.

### 📌 참고
> Part 9·10에서 messages를 펼쳐 흐름을 추적한 것이 관측의 가장 기본 형태였음. LangSmith는 이를 자동·체계적으로 — 모든 단계를 시각적 트리로 기록하고, 지연·비용·오류까지 함께 보여줌.

### 실무 포인트
- "에이전트가 가끔 이상해요"라는 모호한 문제는 관측 없이는 못 고침. 트레이스를 봐야 원인이 보임.
- 출시 전 "이 시스템이 충분히 좋은가?"의 답은 평가 점수로 말할 수 있어야 함 — 느낌이 아니라.

### 3줄 요약
1. 에이전트는 블랙박스가 되기 쉬워, 보지 못하면 못 고침.
2. 관측은 내부를 들여다보고, 평가는 품질을 수치화함.
3. 측정 없이는 개선 없음 — 프로덕션급의 분기점임.

## 16-2. 트레이싱 — 모든 단계를 추적하기

### 개념

**트레이싱(tracing)**은 한 번의 실행에서 일어난 **모든 단계를 기록**하는 것임 — 모델 호출, 도구 실행, 핸드오프, 각 단계의 입출력·지연·비용까지. LangSmith는 이를 **트리 구조**로 보여줘, 에이전트가 무슨 일을 했는지 한눈에 펼쳐 볼 수 있음.

```python
from langsmith import traceable

@traceable
def my_assistant(question: str) -> str:
    context = get_context(question)   # 이 함수도 traceable이면 하위 단계로 기록됨
    return model.invoke(...).content
```

LangChain/LangGraph로 만든 에이전트는 **환경변수 하나**로 트레이싱이 켜짐.

```python
import os
os.environ["LANGSMITH_TRACING"] = "true"
os.environ["LANGSMITH_API_KEY"] = "..."   # smith.langchain.com에서 발급
# 이제 모든 에이전트·체인 실행이 자동으로 LangSmith에 기록됨
```

### 비유

트레이싱은 **수술 중 모든 과정을 녹화하는 것**과 같음. 결과(환자 상태)만 보면 성공/실패만 알지만, 녹화를 보면 "어느 단계에서 무슨 일이 있었는지" 전부 드러남. 문제가 생겼을 때 그 지점을 정확히 짚어 고칠 수 있음.

### 원리

트레이스는 **루트 실행(전체 요청) 아래에 하위 실행(각 단계)이 트리로 매달린** 구조임. 에이전트라면: 루트(에이전트 호출) → 모델 호출 → 도구 호출 → 도구 결과 → 다시 모델 호출 … 각 노드에 입출력·시간·토큰이 기록됨.

- `@traceable`: 임의의 함수를 추적 단위로 만듦(중첩되면 하위 단계로).
- 환경변수 방식: LangChain/LangGraph 객체는 자동으로 모든 내부 단계를 추적.

이 트리를 보면 "왜 이 도구를 불렀나, 어디서 느렸나, 어느 단계에서 틀렸나"가 즉시 보임 — Part 9의 messages 추적을 자동화·시각화한 것임.

### 📌 참고
> 트레이싱은 **개발 디버깅**과 **운영 모니터링** 둘 다에 쓰임. 개발 중엔 "왜 이렇게 동작했지"를 파고들고, 운영 중엔 "실사용자 요청들이 어떻게 처리되는지"를 지켜봄(16-5).

### 실무 포인트
- 에이전트가 이상하게 동작하면, 먼저 트레이스를 열어 단계별 입출력을 확인하는 것이 디버깅의 출발임.
- `@traceable`로 직접 만든 함수(도구 내부 로직 등)도 추적 대상에 넣으면 더 세밀하게 볼 수 있음.

### 3줄 요약
1. 트레이싱은 한 실행의 모든 단계를 트리로 기록함.
2. `@traceable` 또는 환경변수 하나로 켜짐(LangChain/LangGraph 자동).
3. 트레이스를 보면 도구 선택·지연·오류 원인이 즉시 보임.

## 16-3. 무엇을 평가할까 — 평가 기준

### 개념

평가의 첫걸음은 **"무엇을 잴 것인가"**를 정하는 것임. 에이전트 품질은 한 가지가 아니라 여러 측면임. 대표적 기준:

| 기준 | 측정하는 것 |
|---|---|
| **정확성(correctness)** | 답이 사실·정답에 맞는가 |
| **도구 선택** | 올바른 도구를 적절히 불렀는가 |
| **관련성/완결성** | 질문에 충분히·관련되게 답했는가 |
| **지연(latency)** | 얼마나 빨리 답하는가 |
| **비용(cost)** | 토큰·호출이 얼마나 드는가 |
| **안전성** | 해선 안 될 행동을 피했는가 |

### 비유

평가 기준을 정하는 것은 **시험 채점 기준표를 만드는 것**과 같음. "정답 여부"만 볼 수도 있지만, 실무에선 "정확성·풀이 과정·시간"을 함께 봄. 무엇을 중요하게 볼지(채점 항목과 배점)를 먼저 정해야, 일관되게 채점하고 비교할 수 있음.

### 원리

기준이 중요한 이유는, **개선의 방향이 기준에 따라 달라지기** 때문임. 정확성만 보면 느리고 비싼 모델을 택할 수 있고, 비용만 보면 품질이 떨어질 수 있음. 보통 **여러 기준을 함께** 보며 균형을 맞춤. 에이전트 특유의 기준인 **"도구 선택이 옳았나"**(Part 7의 docstring 품질과 직결)도 중요한 평가 대상임 — 답은 맞아도 엉뚱한 도구를 거쳐 왔다면 개선 여지가 있음.

### 📌 참고
> 평가 기준은 **그 시스템의 목적**에서 나옴. 고객 상담 봇은 정확성·안전성이, 대량 처리 파이프라인은 비용·지연이 더 중요할 수 있음. "이 시스템에서 무엇이 가장 중요한가"가 기준 선택의 출발임.

### 실무 포인트
- 기준을 너무 많이 잡으면 평가가 무거워짐. 그 시스템에 핵심적인 2~4개로 시작함.
- 정성적 기준(정확성·관련성)은 사람 또는 LLM 심판으로, 정량적 기준(지연·비용)은 자동 측정으로 잼(16-4).

### 3줄 요약
1. 평가는 "무엇을 잴 것인가"(기준)를 먼저 정하는 데서 시작함.
2. 정확성·도구 선택·관련성·지연·비용·안전성 등 여러 측면이 있음.
3. 기준은 시스템 목적에서 나오며, 핵심 2~4개로 시작함.

## 16-4. 평가셋과 evaluator — 체계적으로 재기

### 개념

품질을 체계적으로 재려면 **평가셋(dataset)**과 **evaluator(채점기)**가 필요함. 평가셋은 "입력 + 기대 출력" 예시 모음(테스트 케이스)이고, evaluator는 "에이전트의 답을 기준에 따라 점수 매기는" 함수임.

```python
from langsmith import Client

client = Client()

# 1) 평가셋 만들기 (입력 + 기대 출력)
dataset = client.create_dataset("문서비서_평가")
client.create_examples(
    dataset_id=dataset.id,
    examples=[
        {"inputs": {"q": "반품 기간?"}, "outputs": {"answer": "7일 이내"}},
        {"inputs": {"q": "환불은 며칠?"}, "outputs": {"answer": "3영업일"}},
    ],
)

# 2) evaluator (채점기) — 답이 기대와 맞는지
def correctness(outputs: dict, reference_outputs: dict) -> bool:
    return reference_outputs["answer"] in outputs["answer"]

# 3) 평가 실행 — 에이전트를 평가셋 전체에 돌려 점수화
client.evaluate(my_agent, data="문서비서_평가", evaluators=[correctness])
```

### 비유

평가셋은 **표준 시험지(문제 + 모범답안)**이고, evaluator는 **채점 기준에 따라 점수를 매기는 채점관**임. 매번 다른 문제로 "느낌상 잘하네"가 아니라, **같은 시험지를 반복**해서 "이번 버전은 80점, 새 버전은 85점"처럼 객관적으로 비교함.

### 원리

핵심은 **반복 가능한 비교**임. 프롬프트를 바꾸거나 모델을 교체했을 때, 같은 평가셋에 돌려 점수를 비교하면 "정말 나아졌는지/어디가 망가졌는지(회귀)"를 알 수 있음. evaluator는 세 종류로 나뉨.

- **코드 evaluator**: 규칙으로 채점(정답 포함 여부, 형식, 지연 등) — 빠르고 객관적.
- **LLM 심판(LLM-as-judge)**: 다른 LLM이 "이 답이 좋은가"를 판단 — 정성적 기준에.
- **사람 피드백**: 사람이 직접 점수 — 가장 신뢰도 높지만 느림.

평가셋은 처음엔 작게(5~10개) 시작해, **실제 실패 사례가 나올 때마다 추가**하며 키움 — 테스트 스위트처럼.

### 📌 참고
> "프롬프트를 고쳤더니 7번 예시가 깨졌다"를 출시 전에 잡는 것이 평가의 가치임. 평가셋은 한 번 만들고 끝이 아니라, 시스템과 함께 자라는 **살아있는 테스트 스위트**임.

### 실무 포인트
- 평가셋은 작게 시작하되, 운영에서 발견된 실패 사례를 계속 추가해 현실을 반영하게 함.
- 새 버전을 배포하기 전 평가셋에 돌려 회귀(점수 하락)를 확인하는 습관이 사고를 막음.

### 3줄 요약
1. 평가셋(입력+기대출력) + evaluator(채점기)로 체계적으로 평가함.
2. 같은 평가셋 반복 → 버전 간 객관적 비교, 회귀 탐지.
3. evaluator는 코드·LLM 심판·사람 — 평가셋은 작게 시작해 키움.

## 16-5. 운영 모니터링 — 프로덕션에서 지켜보기

### 개념

개발 중 평가가 "출시 전 검사"라면, **운영 모니터링**은 "출시 후 상시 감시"임. 실제 사용자 요청이 어떻게 처리되는지 지속적으로 관측하고, **비용·지연·오류율**을 지켜보다 이상 징후에 대응함.

### 비유

운영 모니터링은 **병원의 환자 모니터**와 같음. 수술(개발)이 끝났다고 끝이 아니라, 회복실에서 심박·혈압을 계속 지켜보며 이상이 생기면 즉시 대응함. 시스템도 출시 후 실사용 데이터에서 새 문제가 드러나므로, 지켜보는 일이 계속됨.

### 원리

운영에서는 모든 실사용 요청이 트레이스로 쌓임(16-2). 이를 집계해 지켜봄:

- **비용·토큰**: 갑자기 토큰이 3배 튀면 — 우연이 아니라 신호(루프 폭주·프롬프트 비대 등).
- **지연**: 응답이 느려지면 사용자 이탈로 이어짐.
- **오류율**: 실패·예외가 늘면 즉시 알아야 함.
- **품질 표류**: 실사용 트레이스에서 나쁜 사례를 발견하면 → 평가셋에 추가(16-4) → 다음 개선에 반영.

이렇게 **운영 관측 → 실패 사례 수집 → 평가셋 보강 → 개선**의 순환이 돌아감. 임계치를 정해 알림을 걸면, **사용자가 불평하기 전에** 문제를 잡음.

### 📌 참고
> 운영 모니터링과 개발 평가는 **하나의 루프**임. 운영에서 발견한 실패가 평가셋이 되고, 개선된 버전이 다시 운영에 나감. 이 순환이 시스템을 점점 견고하게 만듦.

### 실무 포인트
- 비용·지연·오류율에 임계치와 알림을 설정해, 이상을 사용자보다 먼저 감지함.
- 운영 트레이스에서 나쁜 사례를 주기적으로 골라 평가셋에 추가하는 루틴을 만듦.

### 3줄 요약
1. 운영 모니터링은 실사용 요청을 상시 관측하는 것임.
2. 비용·지연·오류율을 지켜보고 임계치 알림으로 선제 대응함.
3. 운영 관측 → 실패 수집 → 평가셋 보강 → 개선의 순환이 핵심임.

Part 16에서는 시스템을 들여다보고 품질을 재는 법을 익혔음. 정리하면:

- 관측·평가 — 블랙박스를 열어야 고칠 수 있음 (16-1)
- 트레이싱 — 모든 단계를 트리로 추적(LangSmith) (16-2)
- 평가 기준 — 정확성·도구 선택·지연·비용 등 (16-3)
- 평가셋 + evaluator — 반복 가능한 객관적 비교 (16-4)
- 운영 모니터링 — 상시 감시와 개선 순환 (16-5)

이제 우리는 시스템을 **만들고(1~15) 측정할(16)** 수 있음. 남은 것은 이 시스템을 실제 세상에 **안정적으로 내보내는 것**임. 영속적 상태 저장, 오류 처리와 재시도, 안전장치, 비용 관리 — 프로토타입과 프로덕션을 가르는 마지막 요소들임. **Part 17**에서는 **배포와 신뢰성(deployment & reliability)**을 다뤄, 지금까지 만든 에이전트를 실제 서비스로 운영하기 위한 핵심 고려사항을 정리함.

> 함께 제공되는 **Part 16 실습 노트북**에서는 `@traceable`로 함수에 추적을 다는 법과, 작은 평가셋 + 코드 evaluator로 에이전트의 정확성·도구 선택을 점수화하는 흐름을 다룸. LangSmith는 계정·API 키가 필요하므로, 키 없이도 개념을 체험할 수 있게 **로컬에서 evaluator로 채점하는 미니 평가**를 함께 구현해, 평가의 원리를 Gemini로 직접 확인함.
