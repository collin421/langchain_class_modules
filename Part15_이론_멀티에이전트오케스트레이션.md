# Part 15. 멀티에이전트 오케스트레이션

> **이 파트의 목표**
> Part 14까지 단일 그래프·단일 에이전트로 흐름을 완전히 제어하는 법을 익혔음. 그런데 한 에이전트가 너무 많은 일을 떠맡으면 도구가 넘쳐 판단이 흐려짐(Part 7의 경고). 이 파트에서는 일을 **여러 전문 에이전트로 나누고**, **감독자(supervisor)**가 작업을 적절한 전문가에게 분배·조율하는 **멀티에이전트** 구조를 배움. 한 명의 만능 직원 대신, 전문가 팀과 그 팀장을 두는 것임.

## 15-1. 왜 멀티에이전트인가 — 한 에이전트의 한계

### 개념

단일 에이전트는 강력하지만, 맡는 일이 늘수록 한계가 드러남. Part 7에서 봤듯 **도구가 10개를 넘어가면 모델이 무엇을 고를지 헷갈리고**, 서로 다른 전문성이 필요한 작업(리서치·코딩·작성·검토)을 한 에이전트가 다 하면 **각 영역의 깊이가 얕아짐.**

### 비유

단일 에이전트는 **모든 일을 혼자 하는 만능 직원**과 같음. 작은 일은 효율적이지만, 회사가 커지면 한 사람이 영업·개발·회계·디자인을 다 하기 어려움 — 각 분야의 전문성이 떨어지고, "지금 무슨 일부터 해야 하지"라며 헤맴. 이때 필요한 것이 **전문가들로 이뤄진 팀**임.

### 원리

멀티에이전트는 이 문제를 **분업**으로 푼다. 큰 일을 영역별로 나눠, 각 영역에 **좁고 깊은 전문 에이전트**를 둠.

- 도구가 분산됨 → 각 에이전트는 자기 영역 도구만 → 선택이 명확해짐
- 전문성이 깊어짐 → 각 에이전트의 프롬프트·도구가 한 영역에 집중됨
- 책임이 분리됨 → 디버깅·개선이 영역 단위로 쉬워짐

다만 분업에는 **조율 비용**이 따름 — 누가 무슨 일을 할지 정하고, 결과를 모아야 함. 그 조율을 맡는 것이 다음 절의 감독자임.

### 📌 참고
> 멀티에이전트가 항상 더 낫지는 않음. 단일 에이전트로 충분한 일을 굳이 쪼개면 복잡성·비용만 늘어남(Part 6·12의 남용 경계와 같은 원칙). "한 에이전트가 도구 과다·전문성 희석으로 버거워질 때"가 전환 신호임.

### 실무 포인트
- 도구가 너무 많거나, 한 에이전트의 system_prompt가 여러 상반된 역할을 욱여넣고 있으면 분리 신호임.
- 영역 경계가 자연스럽게 나뉘는 일(리서치 vs 작성 vs 검토)이 멀티에이전트에 적합함.

### 3줄 요약
1. 단일 에이전트는 도구 과다·전문성 희석의 한계가 있음.
2. 멀티에이전트는 일을 영역별 전문 에이전트로 분업해 해결함.
3. 분업엔 조율 비용이 따르며, 단일로 충분하면 쪼개지 않음.

## 15-2. 전문 에이전트 나누기 — 좁고 깊게

### 개념

멀티에이전트의 첫 단계는 일을 **역할별로 나눠 각 전문 에이전트를 만드는** 것임. 각 에이전트는 `create_agent`로 만들되(Part 9), **자기 영역에 집중한 도구와 프롬프트**만 가짐.

```python
from langchain.agents import create_agent

research_agent = create_agent(
    model=model,
    tools=[search_web, search_reference],     # 리서치 도구만
    system_prompt="너는 리서치 전문가야. 자료를 찾아 사실을 정리해.",
    name="researcher",
)

writer_agent = create_agent(
    model=model,
    tools=[draft_section, make_outline],      # 작성 도구만
    system_prompt="너는 글쓰기 전문가야. 주어진 자료로 문서를 작성해.",
    name="writer",
)

reviewer_agent = create_agent(
    model=model,
    tools=[check_facts, check_grammar],       # 검토 도구만
    system_prompt="너는 검토 전문가야. 사실·문법 오류를 잡아내.",
    name="reviewer",
)
```

### 비유

전문 에이전트를 나누는 것은 **회사를 부서로 나누는 것**과 같음. 리서치팀·집필팀·검토팀이 각자 자기 일에만 집중함. 리서치팀은 검색 도구를, 집필팀은 작성 도구를 가짐 — 자기 영역 연장만 있으니 무엇을 쓸지 헷갈리지 않고, 그 일에 능숙해짐.

### 원리

각 전문 에이전트는 **그 자체로 완전한 에이전트**임 — Part 6~10에서 배운 모든 것(도구·RAG·메모리·루프)을 그대로 가짐. 차이는 **범위가 좁다**는 것뿐임. 좁은 범위 덕에:

- 도구 수가 적어 선택이 정확함
- system_prompt가 한 역할에 집중돼 행동이 일관됨
- 독립적으로 테스트·교체가 가능함

각 에이전트에 **`name`을 주는 것이 중요**함 — 감독자가 이 이름으로 "누구에게 일을 넘길지" 지정하기 때문임(15-3).

### 📌 참고
> 전문 에이전트를 너무 잘게 쪼개면 조율 비용이 커짐. "리서치/작성/검토"처럼 의미 있는 큰 역할로 나누는 것이 보통 적절함. 한 에이전트가 여전히 도구 과다면 더 쪼개고, 너무 잘아 핸드오프만 잦으면 합침.

### 실무 포인트
- 각 전문 에이전트는 "이 부서는 무슨 일을 하나"가 한눈에 드러나게 좁게 설계함.
- 에이전트의 `name`은 감독자가 참조하는 식별자이므로 명확하게(`researcher`, `writer`).

### 3줄 요약
1. 일을 역할별로 나눠 각 전문 에이전트를 `create_agent`로 만듦.
2. 각자 자기 영역 도구·프롬프트만 가져 선택이 정확하고 전문적임.
3. 감독자가 참조하도록 각 에이전트에 명확한 `name`을 줌.

## 15-3. 감독자(supervisor) 패턴 — 작업 분배

### 개념

전문 에이전트들을 조율하는 것이 **감독자(supervisor)**임. 감독자는 사용자 요청을 보고 **"이 일은 누가 해야 하는가"**를 판단해 적절한 전문 에이전트에게 넘기고(핸드오프), 그 결과를 받아 다음을 정하거나 사용자에게 답함. `langgraph-supervisor`의 `create_supervisor`로 만듦.

```python
from langgraph_supervisor import create_supervisor

supervisor = create_supervisor(
    agents=[research_agent, writer_agent, reviewer_agent],
    model=model,
    prompt=(
        "너는 문서 제작 팀의 감독자야.\n"
        "- 자료 조사가 필요하면 researcher에게 넘겨.\n"
        "- 글 작성은 writer에게, 검토는 reviewer에게 넘겨.\n"
        "- 각 결과를 받아 다음 단계를 정하고, 끝나면 사용자에게 종합해 답해."
    ),
).compile()

result = supervisor.invoke({"messages": [("user", "생성형 AI 도입 보고서를 자료 조사부터 작성·검토까지 해줘")]})
```

### 비유

감독자는 **프로젝트 팀장**임. 클라이언트 요청을 듣고 "먼저 리서치팀, 그다음 집필팀, 마지막 검토팀"으로 일을 분배하고, 각 팀의 결과를 받아 조율하며, 완성되면 클라이언트에게 종합 보고함. 팀장 자신은 실무를 직접 하기보다 **누가·언제·무엇을 할지 결정**함.

### 원리 — 핸드오프(handoff)

감독자가 일을 넘기는 메커니즘이 **핸드오프(handoff)**임. 감독자는 내부적으로 "이 에이전트에게 넘긴다"는 **핸드오프 도구**를 가짐. 감독자가 `researcher`에게 넘기기로 하면, 그 도구가 호출되어 제어와 맥락(지금까지의 메시지)이 리서치 에이전트로 전달됨. 리서치가 끝나면 다시 감독자로 돌아와, 감독자가 다음 행동(다른 에이전트에게 넘김 / 종료)을 정함.

즉 흐름은 **감독자 ↔ 전문 에이전트**를 오가는 형태임. 감독자가 중앙에서 모든 통신을 통제하므로, 모든 에이전트가 서로를 직접 부르는 "네트워크"식 혼란을 피함.

### 📌 참고
> 핸드오프 시 기본적으로 **전체 메시지 이력**이 다음 에이전트에 전달됨 — 그래서 전문 에이전트가 맥락을 알고 일함. 토큰을 아끼려면 핸드오프 시 전달 범위를 조절하거나, 감독자가 워커의 답을 그대로 사용자에게 전달하는 forwarding 도구를 쓰기도 함.

### 실무 포인트
- 감독자의 `prompt`에 "어떤 일은 누구에게"라는 분배 가이드를 명확히 적음 — 도구 docstring(Part 7)과 같은 역할임.
- 감독자가 엉뚱한 에이전트에게 넘기면, 분배 가이드(prompt)와 각 에이전트의 역할 설명을 점검함.

### 3줄 요약
1. 감독자는 요청을 보고 적절한 전문 에이전트에게 작업을 분배함.
2. 핸드오프로 제어·맥락을 넘기고, 결과를 받아 다음을 정함.
3. 감독자가 중앙 통제 → 모든 에이전트가 서로 부르는 혼란을 피함.

## 15-4. 에이전트 간 정보 전달 — 공유 상태

### 개념

전문 에이전트들이 협업하려면 **결과를 주고받아야** 함. 리서처가 찾은 자료를 라이터가 써야 하고, 라이터의 글을 리뷰어가 봐야 함. 이 정보 전달은 **공유 상태(shared state)**를 통해 일어남 — Part 13의 상태(state)가 멀티에이전트 단위로 커진 것임.

### 비유

공유 상태는 **팀이 함께 보는 작업 게시판**과 같음. 리서치팀이 조사 결과를 게시판에 붙이면, 집필팀이 그걸 보고 글을 써 다시 붙이고, 검토팀이 그 글을 봄. 모두가 같은 게시판(상태)을 공유하므로, 한 팀의 결과물이 다음 팀에게 자연스럽게 전달됨.

### 원리

멀티에이전트 그래프도 하나의 **상태**를 공유함. 보통 `messages`를 중심으로, 각 에이전트가 자기 결과를 상태에 더함(Part 13의 리듀서로 누적). 핸드오프 시 이 상태(맥락)가 다음 에이전트에게 전달되므로:

- 리서처가 상태에 "조사 결과"를 남김 →
- 핸드오프로 라이터가 그 상태를 받아 "글"을 추가 →
- 다시 리뷰어가 받아 "검토 의견"을 추가

즉 **상태가 에이전트들을 거치며 결과를 누적**해 감. Part 13에서 한 그래프 안 노드들이 상태를 갱신하던 것과 같은 원리이며, 여기서는 노드가 "전문 에이전트"인 것임.

### 📌 참고
> 공유 상태에 무엇을 담을지 설계하는 것이 멀티에이전트 품질의 핵심임. 메시지만 공유할 수도 있고, "조사결과·초안·검토의견"처럼 구조화된 필드를 두어 에이전트 간 전달을 명확히 할 수도 있음(Part 13의 커스텀 State).

### 실무 포인트
- 에이전트가 "앞 단계 결과를 못 본다"면, 공유 상태에 그 결과가 제대로 누적되는지(리듀서·핸드오프 전달 범위) 점검함.
- 전달 정보가 너무 많으면 토큰 비용이 커짐 — 필요한 것만 공유하도록 상태를 설계함.

### 3줄 요약
1. 전문 에이전트들은 공유 상태로 결과를 주고받음.
2. 상태가 에이전트들을 거치며 결과를 누적함(Part 13 상태의 확장).
3. 공유 상태에 무엇을 담을지 설계가 멀티에이전트 품질의 핵심임.

## 15-5. 언제 멀티에이전트인가 — 복잡성 비용 경계

### 개념

멀티에이전트는 강력하지만 가장 무거운 구조임. **단일 에이전트보다 느리고 비싸고 복잡함** — 감독자가 판단하는 추가 단계, 에이전트 간 핸드오프, 더 많은 모델 호출이 듦. 그래서 멀티에이전트도 **남용 경계**가 분명함.

**멀티에이전트가 맞는 신호**
- 한 에이전트의 도구가 과다해 선택이 흐려짐
- 서로 다른 전문성이 뚜렷이 필요한 영역들이 있음
- 영역별 독립 개발·테스트·교체가 중요함

**멀티에이전트가 과한 신호**
- 단일 에이전트로 충분히 잘 됨
- 영역 경계가 모호해 핸드오프만 잦아짐
- 속도·비용이 민감한데 작업이 단순함

### 비유

멀티에이전트는 **전문가 팀을 꾸리는 것**임. 큰 프로젝트엔 팀이 효율적이지만, 간단한 일에 팀을 꾸리면 회의·조율 비용만 늘어남 — "이 정도 일은 한 사람이 하는 게 빠른데" 싶은 상황임. 일의 규모에 조직의 규모를 맞춰야 함.

### 원리 — 사다리의 맨 위

Part 12의 사다리(체인 → 에이전트 → 그래프)에 멀티에이전트는 **가장 위**에 놓임.

```
체인  →  단일 에이전트  →  그래프/HITL  →  멀티에이전트
가벼움 ──────────────────────────────────→  무거움
```

정석은 여전히 **가벼운 것부터, 필요할 때만 올리는 것**임. 단일 에이전트의 천장(도구 과다·전문성 희석)에 부딪힐 때 멀티에이전트로 올림. 그리고 멀티에이전트도 내부에서 각 전문 에이전트가 그래프·HITL·RAG를 품을 수 있음 — 계층은 그대로 중첩됨(Part 12).

### 📌 참고
> 멀티에이전트는 비용이 큰 만큼, 도입 전 "단일 에이전트로 안 되는가?"를 반드시 자문함. 면접·설계 리뷰에서 "왜 멀티에이전트인가"를 "도구 과다/전문성 분리가 필요해서"로 답할 수 있어야 함 — "멋있어서"는 위험 신호(Part 6·12와 같은 원칙).

### 실무 포인트
- 단일 에이전트로 시작해, 도구·역할이 비대해질 때 분리하는 것이 안전함.
- 감독자 패턴(중앙 조율)이 대부분의 경우 적절함. 에이전트들이 서로 자유롭게 부르는 "네트워크"식은 통제가 어려워 신중히.

### 3줄 요약
1. 멀티에이전트는 가장 무거운 구조 — 느리고 비싸고 복잡함.
2. 도구 과다·전문성 분리가 필요할 때만 쓰고, 단순하면 단일로.
3. 사다리의 맨 위 — 가벼운 것부터, 천장에 닿을 때만 올림.

## Part 15 마무리 — 다음으로

Part 15에서는 여러 전문 에이전트를 조율하는 멀티에이전트를 익혔음. 정리하면:

- 단일 에이전트의 한계(도구 과다·전문성 희석)와 분업의 해법 (15-1)
- 전문 에이전트 나누기 — 좁고 깊게, `name` 부여 (15-2)
- 감독자(supervisor) 패턴 — 핸드오프로 작업 분배 (15-3)
- 공유 상태로 에이전트 간 결과 전달 (15-4)
- 복잡성 비용 경계 — 사다리의 맨 위, 필요할 때만 (15-5)

이제 우리는 체인부터 멀티에이전트까지 **모든 구조를 손에 쥐었음.** 시스템을 만드는 법은 익혔지만, 만든 것이 **잘 작동하는지 어떻게 알까?** 에이전트가 엉뚱한 도구를 부르고, 멀티에이전트가 잘못 분배하고, 답이 틀려도 — 그것을 보지 못하면 고칠 수 없음. **Part 16**에서는 시스템의 내부를 들여다보고 품질을 측정하는 **관측(observability)과 평가(evaluation)**를 배움 — LangSmith로 에이전트의 모든 단계를 추적하고, 답의 품질을 체계적으로 평가하는 법임.

> 함께 제공되는 **Part 15 실습 노트북**에서는 리서치·작성·검토 세 전문 에이전트를 만들고, `create_supervisor`로 감독자를 두어 "보고서를 조사부터 검토까지" 요청을 처리함. 감독자가 작업을 각 전문가에게 분배하고, 공유 상태로 결과가 전달되며, 핸드오프가 일어나는 과정을 messages로 추적하는 것을 Gemini로 확인함.
