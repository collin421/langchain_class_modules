LangChain 에이전트 마스터 · Part 10 · 중급 마무리

메모리와 상태

Part 9 에이전트는 대화를 기억 못 했음 — "아까 그 상품"을 못 이어감. 메모리를 더해 멀티턴을 기억하게 만들고, 그 기억이 사실 LangGraph의 상태(state)임을 밝히며 고급으로 가는 다리를 놓음.

🟡 중급 · 10/18 checkpointer · thread_id 예상 35분
10-1

에이전트도 매번 잊는다

Part 1에서 "모델은 기억력이 없다"고 배웠음. 에이전트도 마찬가지임. 메모리가 없으면 매 호출이 독립적이라, 방금 한 말도 잊는 상담원이 됨.

인터랙티브 · 메모리 있을 때 vs 없을 때

"베이지 니트 재고?" → "그거 반품 돼?"라고 이어 물었을 때, 메모리 유무에 따라 답이 어떻게 달라지는지 봄.

📌 참고  여기서 다루는 건 단기 메모리(한 대화 내 맥락 유지)임. 세션을 넘어 사용자 선호를 기억하는 장기 메모리는 별도 개념으로 더 복잡한 저장소를 씀.
3줄 요약
  1. 에이전트도 매 호출이 독립적이라 기본적으로 기억 못 함.
  2. 메모리는 "이전 대화를 저장했다 불러오는" 일을 자동화함.
  3. 단기 메모리(한 대화 내 맥락)가 이 파트의 주제임.
10-2

체크포인터 — 상담 기록 노트

메모리를 주는 장치가 체크포인터임. InMemorySaver()를 create_agent의 checkpointer에 넣으면 끝. 상담원 옆에 둔 기록 노트처럼, 대화를 적어뒀다 다시 펼침.

from langgraph.checkpoint.memory import InMemorySaver

agent = create_agent(
    model=model, tools=[get_stock],
    checkpointer=InMemorySaver(),   # ← 메모리 장착
)
인터랙티브 · 노트에 기록되는 과정

"기록하기"를 누를 때마다 에이전트가 대화를 노트(체크포인터)에 저장하는 모습을 봄.

🤖
에이전트
✏️→
📒 체크포인터 (상담 기록 노트)
아직 기록 없음
📌 참고  InMemorySaver는 프로그램이 끝나면 기억이 사라짐(학습·프로토타입용). 실무는 SqliteSaver·PostgresSaver 같은 영속 체크포인터로 서버 재시작에도 대화를 유지함. 인터페이스는 거의 같음.
3줄 요약
  1. 체크포인터는 에이전트에 메모리를 주는 장치임.
  2. InMemorySaver()를 checkpointer에 넣으면 장착됨.
  3. 호출 시점 상태를 저장했다 다시 불러와 대화를 이어줌.
10-3 · 핵심

thread_id — 대화를 구분하는 열쇠

체크포인터를 붙였으면, 이 대화가 어느 대화인지를 thread_id로 알려줘야 함. 같은 thread면 기억이 이어지고, 다른 thread면 별개 대화로 분리됨 — 기록 노트의 고객 이름표와 같음.

인터랙티브 · thread를 바꿔 보기

먼저 thread A에서 "베이지 니트"를 얘기한 뒤, 같은/다른 thread에서 "그거 반품 돼?"를 물어보며 기억이 이어지는지 봄.

🧵 Thread A
thread_id: "user-A"
🧵 Thread B
thread_id: "user-B"
📌 참고  config={"configurable": {"thread_id": ...}}로 넘김. thread_id를 안 넘기거나 매번 바꾸면 기억이 안 이어짐("앞 대화를 잊는" 흔한 버그). 보통 사용자 ID·세션 ID를 thread_id로 씀.
3줄 요약
  1. thread_id는 "이 대화가 어느 대화인지"의 열쇠임.
  2. 같은 thread면 맥락 유지, 다르면 별개 대화로 분리됨.
  3. 체크포인터가 상태를 저장·조회하는 기본 키임.
10-4

여러 사용자를 섞이지 않게

thread_id별로 분리되므로, 같은 에이전트 하나로 여러 손님을 동시에 응대해도 섞이지 않음. 이것이 실제 서비스의 기본 구조임.

인터랙티브 · 두 손님 동시 응대

"동시 응대"를 누르면, 손님A·손님B가 번갈아 "그거"라고 물어도 각자의 '그거'를 정확히 구분하는 걸 봄.

🧵 손님A (베이지 니트)
🧵 손님B (트렌치 코트)
3줄 요약
  1. thread_id별 분리로 한 에이전트가 여러 사용자를 동시에 응대함.
  2. 각자의 "그거"가 섞이지 않고 정확히 구분됨.
  3. 사용자 ID·세션 ID를 thread_id로 부여하는 것이 실무 기본임.
10-5

"메모리"의 정체 — LangGraph의 상태

지금까지 "메모리"라 부른 것은 사실 LangGraph의 상태(state)임. 에이전트는 LangGraph 위에서 돌고(Part 0·9), 그 런타임이 들고 도는 것이 상태 — messages를 포함한 "대화의 현재 모습" 전체임. 보드게임의 현재 판과 같음.

인터랙티브 · thread의 상태 들여다보기

agent.get_state(...)로 특정 thread의 상태를 꺼내봄. 이 "messages를 담은 판"이 고급에서 직접 다룰 state임.

▶ "상태 조회"를 눌러보세요.
📌 참고  이 파트는 중급의 마지막이자 고급의 예고편임. "에이전트가 기억한다"는 편리함 뒤에 "LangGraph가 상태를 들고 돈다"는 구조가 있고, 그 상태를 직접 정의·제어하는 것이 고급(Part 12~)의 핵심임.
3줄 요약
  1. "메모리"는 사실 LangGraph의 상태를 저장·복원하는 것임.
  2. 상태는 messages를 포함한 "대화의 현재 모습" 전체임.
  3. 고급에서는 이 상태를 직접 정의·제어함 — 오늘이 입구임.
