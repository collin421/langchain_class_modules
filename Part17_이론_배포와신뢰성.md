> **이 파트의 목표**

## 17-1. 프로토타입 vs 프로덕션 — 무엇이 다른가

### 개념

지금까지 만든 것은 대부분 **프로토타입**이었음 — `InMemorySaver`로 메모리에 상태를 두고, 키만 넣으면 한 번 잘 돌았음. 하지만 **프로덕션**(실제 서비스)은 전혀 다른 요구를 가짐: 여러 사용자가 동시에, 며칠에 걸쳐, 실패·재시작·악의적 입력 속에서도 안정적으로 돌아야 함.

### 비유

프로토타입과 프로덕션의 차이는 **요리 연습과 식당 운영**의 차이임. 집에서 한 접시 잘 만드는 것(프로토타입)과, 매일 수백 명에게 일관된 품질로, 재료가 떨어지거나 주문이 몰려도 안정적으로 내는 것(프로덕션)은 완전히 다른 일임. 후자는 재고 관리·실패 대비·위생 규정 같은 "운영"이 핵심임.

### 원리 — 프로덕션이 추가로 요구하는 것

같은 코드라도, 프로덕션에서는 다음이 추가로 필요함.

| 측면 | 프로토타입 | 프로덕션 |
|---|---|---|
| **상태** | 메모리(재시작 시 소실) | 영속 DB(재시작에도 유지) — 17-2 |
| **실패** | 한 번 되면 됨 | 모델·API 실패에 견딤(재시도) — 17-3 |
| **폭주** | 신경 안 씀 | 무한 루프·비용 상한 방어 — 17-4 |
| **규모** | 나 혼자 | 동시 다수 사용자 |
| **비용** | 무시 | 모델·토큰·지연 최적화 — 17-5 |

핵심은 **"한 번 되는 것"과 "항상 안정적으로 되는 것"은 다른 문제**라는 인식임. 신뢰성은 기능이 아니라 **운영의 품질**임.

### 📌 참고
> 모든 프로젝트가 프로덕션급일 필요는 없음. 내부 도구·실험은 프로토타입으로 충분함. 하지만 실사용자를 받는 순간, 이 파트의 고려사항들이 "있으면 좋은 것"에서 "없으면 사고 나는 것"으로 바뀜.

### 실무 포인트
- "이 시스템에 실사용자가 붙는가?"가 프로덕션 고려를 시작할 기준임.
- 프로토타입에서 프로덕션으로 갈 때, 코드 변경은 의외로 작음(체크포인터 한 줄 교체 등) — 차이는 주로 "운영 관점"을 더하는 것임.

### 3줄 요약
1. 프로토타입(한 번 됨)과 프로덕션(항상 안정)은 다른 문제임.
2. 프로덕션은 영속 상태·실패 대비·폭주 방어·규모·비용을 요구함.
3. 신뢰성은 기능이 아니라 운영의 품질임.

## 17-2. 영속적 상태 — 재시작에도 살아남기

### 개념

`InMemorySaver`(Part 10·14)는 상태를 **메모리(RAM)**에 둠 — 프로그램이 멈추면 모든 대화가 사라짐. 프로덕션에서는 상태를 **데이터베이스**에 저장하는 영속 체크포인터로 바꿔, 서버가 재시작돼도 대화·작업이 이어지게 함.

```python
# 프로토타입
from langgraph.checkpoint.memory import InMemorySaver
checkpointer = InMemorySaver()   # RAM — 재시작 시 소실

# 프로덕션 (SQLite: 단일 서버·소규모)
from langgraph.checkpoint.sqlite import SqliteSaver        # pip install langgraph-checkpoint-sqlite

# 프로덕션 (PostgreSQL: 대규모·다중 서버)
from langgraph.checkpoint.postgres import PostgresSaver    # pip install langgraph-checkpoint-postgres
with PostgresSaver.from_conn_string("postgresql://...") as checkpointer:
    checkpointer.setup()                  # 최초 1회: 테이블 생성
    graph = builder.compile(checkpointer=checkpointer)
```

### 비유

`InMemorySaver`는 **포스트잇 메모**, 영속 체크포인터는 **장부에 기록하는 것**과 같음. 포스트잇은 바람 한 번(재시작)에 날아가지만, 장부는 보관되어 다음 날에도, 다른 직원이 와도 그대로 남아 있음. 식당이 매일 같은 장부를 이어 쓰듯, 서비스도 영속 저장으로 대화를 이어감.

### 원리

체크포인터의 **인터페이스는 동일**함 — `compile(checkpointer=...)`에 무엇을 넣느냐만 다름. 그래서 프로토타입에서 프로덕션으로 가는 변경은 거의 **한 줄 교체**임(InMemorySaver → SqliteSaver/PostgresSaver). 단, 영속 저장소는 몇 가지 운영 고려가 따름.

- **최초 설정**: PostgresSaver 등은 `setup()`으로 테이블을 한 번 만들어야 함(배포 스크립트에서 처리, 앱 런타임 아님).
- **저장소 선택**: SQLite(단일 서버·간단), PostgreSQL/Redis(대규모·다중 서버·고성능).
- **이력 관리**: 매 단계가 체크포인트 행을 만들어 누적됨 → 오래된 thread를 주기적으로 정리(TTL).

이 덕분에 Part 14의 HITL("발행 전 멈췄다 며칠 뒤 재개")도 진짜로 가능해짐 — 상태가 DB에 안전히 남아 있기 때문임.

### 📌 참고
> 상태(state)에 큰 파일을 직접 담지 말 것. 매 단계마다 복사 저장되어 DB가 비대해짐. 파일은 외부 저장소에 두고 상태엔 **URL/참조만** 담음. 상태는 작고 가볍게(Part 13의 원칙) — 프로덕션에서 더 중요해짐.

### 실무 포인트
- 개발은 `InMemorySaver`/`SqliteSaver`, 운영은 `PostgresSaver`(또는 Redis)가 일반적 조합임.
- 체크포인트 누적으로 DB가 커지므로, 오래된 thread 정리(나이 N일 초과 삭제) 잡을 처음부터 계획함.

### 3줄 요약
1. `InMemorySaver`는 재시작 시 소실 — 프로덕션은 영속 체크포인터로.
2. 인터페이스가 같아 교체는 거의 한 줄(setup·저장소 선택만 추가).
3. 상태엔 참조만 담고, 오래된 체크포인트는 주기적으로 정리함.

## 17-3. 오류 처리와 재시도 — 실패에 견디기

### 개념

프로덕션에서 외부 의존(모델 API·도구·네트워크)은 **반드시 가끔 실패함** — 일시적 오류, 타임아웃, 속도 제한(rate limit). 신뢰성 있는 시스템은 이를 **예상하고 견딤**: 재시도(retry), 타임아웃(timeout), 대체(fallback)로 일시적 실패를 흡수함.

```python
from langchain_google_genai import ChatGoogleGenerativeAI

model = ChatGoogleGenerativeAI(
    model="gemini-3-flash",
    max_retries=3,      # 일시적 실패 시 자동 재시도
    timeout=30,         # 30초 내 응답 없으면 중단
)
```

도구 안에서도 실패를 잡아 모델이 이해할 결과로 돌려줌.

```python
@tool
def fetch_data(query: str) -> str:
    """외부 API에서 데이터를 가져온다."""
    try:
        return call_external_api(query)
    except TimeoutError:
        return "데이터 조회 시간 초과 — 잠시 후 다시 시도해 주세요."   # 모델이 읽고 대응
```

### 비유

오류 처리는 **식당의 비상 대비**와 같음. 한 재료가 떨어졌다고 영업을 멈추지 않음 — 대체 재료를 쓰거나(fallback), 잠시 후 다시 확인하거나(retry), 손님에게 양해를 구함(graceful degradation). 좋은 운영은 "실패가 없는 것"이 아니라 "실패해도 무너지지 않는 것"임.

### 원리

프로덕션 신뢰성의 핵심은 **"실패는 예외가 아니라 정상"**이라는 전제임. 대응 도구:

- **재시도(retry)**: 일시적 실패(네트워크·rate limit)는 잠깐 뒤 다시 하면 대개 성공. `max_retries`로 자동화. 보통 지수 백오프(점점 간격 늘림)를 씀.
- **타임아웃(timeout)**: 응답이 영영 안 오는 것을 막아, 한 요청이 시스템을 붙잡지 않게 함.
- **대체(fallback)**: 주 모델/도구가 실패하면 대체 경로로(예: 다른 모델, 캐시된 답).
- **우아한 저하(graceful degradation)**: 완전 실패 대신 "부분 결과 + 양해"를 돌려줌.

도구 내부에서 예외를 잡아 **모델이 읽을 메시지로 변환**하면, 에이전트가 "실패했으니 다시/다르게 시도"라고 판단할 수 있음(에이전트의 자기 복구).

### 📌 참고
> 무엇이든 재시도해선 안 됨. **비가역 행동(결제·발송)**을 무턱대고 재시도하면 중복 실행 위험이 있음 — 멱등성(idempotency, 여러 번 해도 결과가 같음)을 보장하거나, 재시도 전 확인(HITL·Part 14)을 둠.

### 실무 포인트
- 모델 호출엔 `max_retries`·`timeout`을 기본으로 설정함.
- 도구는 예외를 그대로 터뜨리지 말고, 잡아서 "무엇이 왜 실패했는지" 모델/사용자가 알 메시지로 돌려줌.
- 비가역 행동의 재시도는 멱등성·확인 장치와 함께만.

### 3줄 요약
1. 프로덕션에선 외부 의존이 반드시 가끔 실패함 — 예상하고 견딤.
2. 재시도·타임아웃·대체·우아한 저하로 일시적 실패를 흡수함.
3. 비가역 행동의 재시도는 멱등성·확인 장치와 함께만.

## 17-4. 안전장치 — 폭주를 막다

### 개념

자동화 시스템의 큰 위험은 **폭주**임 — 무한 루프로 영원히 돌거나, 비용이 통제 없이 치솟는 것. 프로덕션에서는 **명시적 상한과 가드**로 이를 막아야 함.

```python
# 무한 루프 방지: 최대 단계 수 상한
graph.invoke(input, config={
    "configurable": {"thread_id": "..."},
    "recursion_limit": 25,      # 25단계 넘으면 GraphRecursionError
})
```

### 비유

안전장치는 **차단기와 과속 방지턱**과 같음. 전기 시스템에 차단기가 없으면 합선 하나가 건물 전체를 태움 — 차단기는 위험 수위에서 회로를 끊음. recursion_limit·비용 상한은 시스템의 차단기로, "여기까지"를 강제해 폭주가 전체로 번지는 것을 막음.

### 원리 — 막아야 할 폭주들

1. **무한 루프**: 조건 분기가 모델로 되돌아가고, 모델이 같은 도구를 부르고, 그 도구가 같은 식으로 실패하면 — 영원히 돎(Part 13). 막는 법:
   - `recursion_limit`: 그래프의 최대 단계 수. 넘으면 `GraphRecursionError` 발생 → 잡아서 "오류"로 처리.
   - 상태의 **단계 수 카운터**(Part 13의 `attempts` 제한): 라우팅에서 N회 넘으면 강제 종료.

2. **비용 폭주**: 토큰이 통제 없이 늘어남(긴 메시지 누적, 잦은 호출). 막는 법:
   - 메시지 트리밍(17-5), 호출 횟수 상한, 예산 임계 알림(Part 16의 모니터링).

3. **위험 행동**: 비가역·고위험 행동은 가드를 둠 — Part 14의 HITL(사람 승인)이 곧 안전장치임.

핵심은 **"기본값을 신뢰하지 말고, 상한을 명시적으로 정하는 것"**임. 무한·무제한은 프로덕션에서 사고의 씨앗임.

### 📌 참고
> `recursion_limit`을 넘으면 LangGraph가 `GraphRecursionError`를 자동으로 냄. 이를 try/except로 잡아 run을 "error"로 표시하고 사용자에게 우아하게 안내함(17-3의 우아한 저하와 연결).

### 실무 포인트
- 모든 그래프 invoke에 `recursion_limit`을 명시하고, 상태에 단계 카운터도 함께 둠(이중 안전장치).
- 비가역 행동엔 HITL 가드(Part 14), 비용엔 모니터링 임계 알림(Part 16)을 결합함.

### 3줄 요약
1. 자동화의 큰 위험은 무한 루프·비용 폭주·위험 행동임.
2. `recursion_limit`·단계 카운터·비용 상한·HITL 가드로 막음.
3. 기본값을 믿지 말고 상한을 명시적으로 정함 — 무제한은 사고의 씨앗.

## 17-5. 비용·지연 관리 — 실전 운영 최적화

### 개념

프로덕션은 **비용과 속도**가 사업적으로 중요해짐. 같은 기능도 어떻게 운영하느냐에 따라 비용·지연이 몇 배 차이남. 모델 선택·메시지 관리·캐싱 등으로 최적화함.

### 비유

비용·지연 관리는 **식당의 원가·회전율 관리**와 같음. 같은 요리도 재료를 어디서 사고(모델 선택), 미리 손질해 두고(캐싱), 안 쓰는 재료를 쟁여두지 않으면(메시지 트리밍) 원가와 제공 속도가 크게 달라짐. 맛(품질)을 지키면서 원가를 낮추는 것이 운영의 묘임.

### 원리 — 주요 레버

- **모델 선택**: 모든 단계에 최강·최비싼 모델을 쓸 필요 없음. 간단한 분류·라우팅엔 작고 빠른 모델(예: Flash), 복잡한 추론엔 큰 모델 — **작업 난이도에 모델을 맞춤.**
- **메시지 트리밍/요약**: 대화가 길어지면 매번 보내는 messages가 커져 토큰·지연이 늘어남(Part 10). 오래된 메시지를 잘라내거나 요약해 컨텍스트를 가볍게 유지함.
- **캐싱**: 같은(또는 비슷한) 요청의 결과를 재사용. 반복 질문·고정 프롬프트에 효과적.
- **불필요한 호출 줄이기**: 도구를 안 써도 되는 걸 쓰거나, 멀티에이전트가 과한 핸드오프를 하면 비용↑(Part 15). 구조를 적정 무게로(사다리 원칙).
- **비동기·백그라운드**: 오래 걸리는 작업은 HTTP 연결을 붙잡지 말고 백그라운드로 돌리고, 결과를 폴링/구독하게 함(대규모의 흔한 패턴).

### 📌 참고
> 최적화는 **측정 후에** 함(Part 16). 트레이스로 "어느 단계가 느리고 비싼지"를 먼저 보고, 거기를 고침. 측정 없는 최적화는 엉뚱한 곳을 깎기 쉬움 — "추측하지 말고 측정하라"가 원칙임.

### 실무 포인트
- 작업 난이도별로 모델을 차등 적용(라우팅엔 작은 모델, 핵심 추론엔 큰 모델).
- 긴 대화 서비스는 메시지 트리밍/요약 전략을 처음부터 설계함(Part 10에서 예고).
- 최적화 전 반드시 트레이스로 병목을 측정함(Part 16).

### 3줄 요약
1. 프로덕션은 비용·지연이 사업적으로 중요해짐.
2. 모델 차등·메시지 트리밍·캐싱·호출 절감·비동기로 최적화함.
3. 최적화는 측정(트레이스) 후에 — 추측하지 말고 측정함.

## Part 17 마무리 — 다음으로

Part 17에서는 시스템을 실제 세상으로 내보내는 신뢰성 요소들을 정리했음. 정리하면:

- 프로토타입 vs 프로덕션 — 한 번 됨 ≠ 항상 안정 (17-1)
- 영속 상태 — InMemory → SQLite/Postgres, 재시작에도 유지 (17-2)
- 오류 처리·재시도 — 실패를 예상하고 견딤 (17-3)
- 안전장치 — recursion_limit·상한·가드로 폭주 방어 (17-4)
- 비용·지연 관리 — 측정 후 최적화 (17-5)

이제 우리는 시스템을 **만들고(1~15) · 측정하고(16) · 안정적으로 운영(17)**할 수 있음. 커리큘럼의 모든 조각이 갖춰졌음. 마지막 **Part 18**에서는 이 모든 것을 하나로 묶는 **최종 프로젝트**를 만듦 — 초급의 체인부터 고급의 그래프·HITL·멀티에이전트·관측·신뢰성까지, 배운 전부를 결합한 완성형 시스템을 설계하고 구현하며 커리큘럼을 졸업함.

> 함께 제공되는 **Part 17 실습 노트북**에서는 지금까지의 그래프에 프로덕션 요소를 더해 봄 — `InMemorySaver`를 `SqliteSaver`로 바꿔 재시작에도 상태가 유지되는 것을 확인하고, 모델에 `max_retries`·`timeout`을 설정하고, 도구에 예외 처리를 넣고, `recursion_limit`으로 무한 루프를 막는 안전장치를 적용함. "한 번 되는 코드"를 "안정적으로 도는 코드"로 바꾸는 과정을 Gemini로 체험함.
