# Part 14. Human-in-the-loop — 사람 개입

> **이 파트의 목표**
> Part 12의 네 신호 중 셋(반복·분기·상태)을 Part 13에서 구현했음. 마지막 하나가 **④사람 개입**임. 에이전트·그래프가 아무리 똑똑해도, 되돌릴 수 없는 행동(발송·결제·삭제·발행)은 **사람의 확인을 거쳐야** 안전함. 이 파트에서는 그래프를 특정 지점에 **멈추고(`interrupt`)**, 사람의 결정을 받아 **이어가는(`Command(resume=...)`)** Human-in-the-loop(HITL)를 배움. Part 13에서 만든 체크포인터(상태 저장·재개)가 그 토대임.

## 14-1. 왜 사람 개입인가 — 자동화의 안전벨트

### 개념

에이전트가 실제 도구(이메일 발송, 결제, DB 삭제, 문서 발행)를 쥐면, 그 행동은 **되돌릴 수 없음**. 모델이 옳다고 판단해 실행한 것이 틀렸다면 — 잘못된 메일이 나가고, 엉뚱한 결제가 되고, 데이터가 지워짐. **Human-in-the-loop**는 이런 **고위험·비가역 행동 직전에 사람을 끼워 넣어**, "정말 실행할까요?"를 확인받는 안전장치임.

### 비유

HITL는 **중요 서류의 최종 결재란**과 같음. 실무자가 아무리 일을 잘해도, 거액 지출이나 외부 발송 같은 중대한 일은 **결재권자의 도장**을 거쳐야 진행됨. 도장을 받기 전까지 서류는 책상에서 **멈춰 기다림**. HITL는 그래프에 이 "결재 대기" 지점을 박는 것임.

### 원리

자동화의 가치는 사람의 손을 더는 것이지만, **모든 일을 자동화하는 것이 항상 옳지는 않음**. 특히:

- **비가역적**: 한 번 하면 되돌릴 수 없는 행동
- **고위험**: 틀리면 피해가 큰 행동(금전·평판·법적)
- **규제·감사**: 사람 승인 기록이 요구되는 행동

이런 지점에 사람을 두면, 자동화의 속도와 사람의 판단을 **둘 다** 얻음. 핵심은 "사람을 어디에 두느냐"임 — 너무 많이 두면 자동화의 의미가 없고, 안 두면 위험함.

### 📌 참고
> HITL는 비용임 — 사람의 시간·지연·조율이 듦. 그래서 **되돌릴 수 있는 단계에는 두지 않음.** 모든 단계에 개입을 걸면, 사람이 무심코 "승인"만 누르는 습관이 생겨 오히려 안전장치가 무력해짐. 꼭 필요한 지점에만 둠.

### 실무 포인트
- "이 행동이 틀렸을 때 되돌릴 수 있나? 피해가 큰가?"가 개입 여부의 판단 기준임.
- 읽기·조회·초안 작성 같은 안전한 행동엔 개입을 두지 않고, 발송·결제·삭제·발행 같은 행동에만 둠.

### 3줄 요약
1. HITL는 비가역·고위험 행동 직전에 사람 확인을 끼우는 안전장치임.
2. 자동화의 속도와 사람의 판단을 둘 다 얻는 방법임.
3. 모든 단계가 아니라 꼭 필요한 지점에만 둠(남용은 무력화).

## 14-2. interrupt — 그래프를 멈추기

### 개념

그래프를 특정 지점에 멈추는 함수가 **`interrupt`**임. 노드 안에서 `interrupt(...)`를 호출하면, 그래프 실행이 **그 자리에서 일시정지**되고, 호출자에게 "사람의 입력이 필요하다"는 신호와 함께 제어가 돌아감.

```python
from langgraph.types import interrupt

def approval_node(state: State) -> dict:
    # 사람에게 보여줄 정보를 담아 멈춤
    decision = interrupt({
        "question": "이 문서를 발행할까요?",
        "draft": state["draft"],
    })
    # ↑ 여기서 그래프가 멈춤. decision은 사람이 재개할 때 채워짐(14-3)
    return {"approved": decision == "승인"}
```

### 비유

`interrupt`는 **"결재 대기" 도장을 찍고 서류를 멈추는 것**임. 담당자가 서류에 "검토 요망: 이 내용 발행할까요?"라고 적어 결재함에 올려두고 손을 뗌. 결재권자가 도장을 찍어 돌려주기 전까지, 서류는 그 자리에서 움직이지 않음.

### 원리 — 멈추려면 상태를 저장해야 한다

`interrupt`가 작동하려면 **체크포인터가 필수**임(Part 13). 멈춘다는 것은 "지금까지의 상태를 안전하게 얼려 두었다가, 나중에 그 지점부터 다시 시작한다"는 뜻이기 때문임. 체크포인터가 그 동결 상태를 thread_id별로 저장함.

```python
graph = builder.compile(checkpointer=InMemorySaver())   # ← 멈춤·재개의 토대
```

`interrupt`에 넘긴 값(`{"question": ..., "draft": ...}`)은 **사람에게 보여줄 정보**로 호출자에게 전달됨. 사람은 그 정보를 보고 결정을 내림.

### 📌 참고
> `interrupt`로 멈춘 그래프를 실행하면, 결과에 "인터럽트가 발생했다"는 정보가 담겨 옴(어디서·왜 멈췄는지). 호출자(앱)는 이를 받아 사람에게 보여줄 UI를 띄우고, 사람의 답을 받아 재개함.

### 실무 포인트
- `interrupt` 노드는 보통 "위험 행동 직전"에 둠 — 발송·결제·발행 노드 바로 앞.
- 멈출 때 사람이 판단하기에 충분한 정보(무엇을·왜)를 함께 넘겨야 함. 정보가 부족하면 사람이 제대로 결정 못 함.

### 3줄 요약
1. `interrupt(...)`는 노드 안에서 그래프를 그 자리에 멈춤.
2. 멈추려면 체크포인터가 필수임(상태를 얼려 두었다 재개).
3. `interrupt`에 넘긴 값은 사람에게 보여줄 정보가 됨.

## 14-3. Command(resume=...) — 사람 입력으로 재개하기

### 개념

멈춘 그래프를 다시 움직이는 것이 **`Command(resume=값)`**임. 사람이 내린 결정을 `resume`에 담아 그래프를 호출하면, **멈췄던 바로 그 지점부터** 실행이 이어짐. 이때 `resume`에 담은 값이 노드 안 `interrupt(...)`의 **반환값**이 됨.

```python
from langgraph.types import Command

# 1) 처음 실행 → interrupt에서 멈춤
config = {"configurable": {"thread_id": "doc-1"}}
graph.invoke(initial_state, config=config)
# (사람에게 "발행할까요?" 질문이 전달되어 멈춤)

# 2) 사람이 "승인"을 결정 → 그 값으로 재개
graph.invoke(Command(resume="승인"), config=config)
# ↑ approval_node의 interrupt(...)가 "승인"을 반환하며 거기서부터 이어짐
```

### 비유

`Command(resume=...)`는 **결재권자가 도장을 찍어 서류를 돌려보내는 것**임. "승인" 도장이 찍힌 서류가 결재함에서 내려와, 멈췄던 그 단계부터 다시 흐르기 시작함. 담당자는 처음부터 다시 일하는 게 아니라, **딱 멈춘 지점부터** 이어감.

### 원리 — 같은 thread_id가 핵심

재개의 핵심은 **처음 실행과 재개에 반드시 같은 `thread_id`를 써야** 한다는 것임. 체크포인터는 그 thread_id로 "얼려 둔 상태"를 찾아 복원하기 때문임. thread_id가 다르면 엉뚱한(혹은 빈) 상태에서 시작함.

흐름을 정리하면:

1. `graph.invoke(입력, config)` → 노드의 `interrupt`에서 멈춤 (상태 동결)
2. 앱이 사람에게 질문을 보여주고 결정을 받음
3. `graph.invoke(Command(resume=결정), config)` → 같은 thread_id로 재개
4. `interrupt(...)`가 `결정`을 반환하며 그 지점부터 이어 실행

### 📌 참고
> 사람이 영영 답하지 않을 수도 있음. 실무에서는 일정 시간 지난 미응답 thread를 자동 처리(만료·재알림)하는 장치를 함께 둠. 또 thread_id는 사용자·작업마다 고유해야 함 — 겹치면 한 사람의 승인이 다른 사람의 그래프를 재개하는 사고가 남.

### 실무 포인트
- 초기 호출과 재개에 **같은 config(thread_id)**를 쓰는 것을 잊지 말 것 — HITL 버그의 1순위 원인임.
- 재개 입력은 보통 사람의 결정값("승인"/"반려"/수정 텍스트 등)임. 이 값이 곧 `interrupt`의 반환값이 됨.

### 3줄 요약
1. `Command(resume=값)`로 멈춘 그래프를 그 지점부터 재개함.
2. `resume`에 담은 값이 노드 안 `interrupt`의 반환값이 됨.
3. 초기 실행과 재개에 같은 thread_id를 써야 상태가 복원됨.

## 14-4. 세 가지 개입 패턴 — 승인 · 수정 · 반려

### 개념

HITL의 사람 결정은 보통 세 가지 형태임. 같은 `interrupt`/`resume` 구조로 모두 구현됨.

| 패턴 | 사람의 결정 | 그래프의 반응 |
|---|---|---|
| **승인(Approve)** | "그대로 진행" | 제안된 행동을 실행 |
| **수정(Edit)** | "이렇게 고쳐서 진행" | 사람이 고친 값으로 실행 |
| **반려(Reject)** | "하지 마 / 다시 해" | 행동 취소, 또는 수정 루프로 |

```python
def approval_node(state: State) -> dict:
    decision = interrupt({"action": "발행", "draft": state["draft"]})

    if decision["type"] == "approve":
        return {"final": state["draft"], "next": "publish"}
    elif decision["type"] == "edit":
        return {"final": decision["edited_text"], "next": "publish"}  # 사람이 고친 값
    else:  # reject
        return {"next": "revise"}   # 수정 루프로 (Part 13의 분기·루프와 결합)
```

### 비유

세 패턴은 **결재권자가 할 수 있는 세 가지 행동**과 같음 — 도장 찍기(승인), 빨간펜으로 고쳐서 돌려주기(수정), 반려 도장 찍어 되돌리기(반려). 어느 쪽이든 서류는 그 결정에 따라 다음 경로로 흐름.

### 원리

세 패턴 모두 `interrupt`의 **반환값(사람의 결정)을 보고 분기**하는 것임 — Part 13의 조건 분기·루프와 자연스럽게 결합됨. 승인이면 행동 노드로, 수정이면 사람이 준 값으로 행동, 반려면 수정 루프로 되돌아감. 즉 **HITL는 별개 기능이 아니라, "사람의 입력을 받는 분기점"**일 뿐임. 네 신호(반복·분기·상태·개입)가 한 그래프에서 함께 작동하는 것임.

### 📌 참고
> "수정" 패턴은 특히 강력함 — 모델의 제안을 사람이 다듬어 더 나은 결과로 만들고 그것을 그대로 실행함. 모델과 사람의 협업이 가장 잘 드러나는 패턴임.

### 실무 포인트
- 어떤 결정 형태를 받을지(승인만? 수정도?)를 `interrupt`에 보여줄 정보와 함께 명확히 설계함.
- 반려 시 무한 반복을 막으려면 Part 13의 횟수 제한을 함께 둠.

### 3줄 요약
1. 개입은 승인·수정·반려 세 패턴으로 나뉨.
2. 셋 다 `interrupt` 반환값(사람 결정)을 보고 분기하는 것임.
3. HITL는 "사람 입력을 받는 분기점" — 네 신호가 함께 작동함.

## 14-5. 어디에 사람을 둘까 — 설계 원칙

### 개념

HITL의 성패는 **"어디에 개입을 두느냐"**에 달림. 너무 많으면 자동화가 무의미해지고, 너무 적으면 위험함. 원칙을 정리함.

**개입을 둬야 하는 곳**
- 비가역적 행동(삭제·발송·결제·발행)
- 고위험 행동(큰 금액·외부 노출·법적 영향)
- 규제·감사로 사람 승인 기록이 필요한 곳

**개입을 두지 말아야 하는 곳**
- 되돌릴 수 있는 행동(초안 작성·조회·내부 계산)
- 빈번해서 매번 멈추면 흐름이 끊기는 곳
- 사람이 판단할 정보가 없거나 의미 없는 곳

### 비유

개입 지점을 정하는 것은 **건물에 비상정지 버튼을 설치하는 것**과 같음. 위험한 기계 앞엔 반드시 두되, 모든 문마다 달면 아무도 신경 쓰지 않게 됨. **위험이 큰 곳에, 사람이 의미 있게 판단할 수 있는 곳에** 둠.

### 원리

좋은 HITL 설계의 핵심은 **"개입의 비용 ≤ 막아내는 위험"**임. 사람의 시간·지연이라는 비용을 치를 만큼, 그 지점의 행동이 위험할 때만 둠. 그리고 사람이 **무심코 승인하지 않도록** — 개입을 드물게, 그리고 충분한 정보와 함께 둬야 함. 개입이 흔해지면 사람은 내용을 안 보고 "승인"을 누르게 되어, 안전장치가 형식만 남음.

### 📌 참고
> 큰 시스템에서는 "위험도에 따라 차등 개입"을 둠 — 소액은 자동, 고액만 승인 요청처럼. 이는 그래프의 조건 분기(Part 13)로 "위험하면 interrupt 노드로, 아니면 바로 실행"을 구현하면 됨. 개입조차 조건부일 수 있음.

### 실무 포인트
- 새 행동을 추가할 때마다 "이건 개입이 필요한가?"를 비가역성·위험도로 점검함.
- 개입이 너무 잦다고 느껴지면, 정말 위험한 것만 남기고 조건부 개입(위험할 때만)으로 줄임.

### 3줄 요약
1. 개입은 비가역·고위험·규제 지점에만 두고, 안전한 곳엔 안 둠.
2. 원칙: 개입의 비용 ≤ 막아내는 위험.
3. 너무 잦으면 무심코 승인하게 되어 안전장치가 무력해짐.

## Part 14 마무리 — 다음으로

Part 14에서는 네 신호의 마지막인 사람 개입을 구현했음. 정리하면:

- HITL는 비가역·고위험 행동의 안전벨트 (14-1)
- `interrupt`로 그래프를 멈춤(체크포인터 필수) (14-2)
- `Command(resume=값)`로 같은 thread_id에서 재개 (14-3)
- 승인·수정·반려 세 패턴, 모두 분기로 구현 (14-4)
- 어디에 둘까 — 비가역·고위험에만, 남용 금지 (14-5)

이제 우리는 Part 12의 **네 신호(반복·분기·상태·개입)를 모두 그래프로 구현**할 수 있음. 단일 그래프 안에서 흐름을 완전히 제어하게 된 것임. 하지만 더 큰 시스템에서는 한 그래프·한 에이전트로는 벅찬 일이 있음 — 서로 다른 전문성이 필요한 여러 작업을 한 에이전트가 다 떠맡으면 도구가 너무 많아지고 판단이 흐려짐(Part 7의 경고). **Part 15**에서는 이를 풀기 위해 **여러 전문 에이전트를 오케스트레이션**하는 법을 배움 — 감독자(supervisor)가 작업을 적절한 전문 에이전트에게 분배하고 결과를 모으는, 멀티에이전트 구조임.

> 함께 제공되는 **Part 14 실습 노트북**에서는 Part 13의 문서 그래프에 **발행 전 승인 노드**를 추가해, `interrupt`로 "발행할까요?"에서 멈추고 `Command(resume=...)`로 승인/수정/반려를 받아 재개하는 HITL를 직접 구현함. 같은 thread_id로 멈춤→재개가 이어지는 것과, 세 가지 결정에 따라 그래프가 다르게 흐르는 것을 확인함.
