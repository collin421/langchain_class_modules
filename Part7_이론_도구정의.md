# Part 7. 도구(Tool) 정의

> **이 파트의 목표**
> Part 6에서 에이전트가 "스스로 도구를 골라 행동한다"는 것을 봤음. 그런데 도구가 없으면 에이전트는 판단만 할 뿐 아무것도 못 함. 이 파트에서는 에이전트의 **손과 발인 도구를 직접 정의**하는 법을 배움. 핵심은 충격적일 만큼 단순함 — **함수의 이름·docstring·타입 힌트가 그대로 모델에게 주는 사용 설명서가 됨.** 즉 "모델이 이 도구를 언제·어떻게 쓸지"는 우리가 쓰는 docstring 한 줄에 달려 있음.

---

## 7-1. 도구란 무엇인가 — 외부 세계와의 접점

### 개념

**도구(Tool)**는 에이전트가 호출할 수 있는 **하나의 기능**임 — 웹 검색, 계산, 데이터베이스 조회, 이메일 발송, 재고 확인 등. 모델 자체는 "글자를 생성"할 뿐 실제 행동(검색·계산·조회)은 못 함. 도구는 그 행동을 대신 수행해 결과를 모델에게 돌려줌.

### 비유

모델(LLM)은 **뛰어난 두뇌지만 몸이 없는 존재**임. 아무리 똑똑해도 직접 책을 펴거나 계산기를 두드릴 수 없음. **도구는 그 두뇌에 달아주는 손과 발**임. 검색 도구라는 손을 달면 정보를 집어 올 수 있고, 계산 도구라는 손을 달면 숫자를 정확히 처리할 수 있음. 어떤 손발을 달아주느냐가 에이전트가 할 수 있는 일의 범위를 정함.

### 원리

도구는 결국 **평범한 함수**임. "입력을 받아 일을 하고 결과를 돌려주는" 파이썬 함수에, 모델이 이해할 수 있는 설명을 붙인 것이 도구임. 에이전트는 가진 도구들의 **이름과 설명**을 보고 "지금 상황에 어떤 도구가 맞는지" 판단해 호출함(Part 6의 ReAct 루프).

### 📌 참고
> LangChain에는 이미 만들어진 도구(웹 검색, 위키백과 등)도 많지만, 실무에서 진짜 가치는 **내 시스템에 맞는 도구를 직접 만드는 것**에서 나옴. 리리마켓이라면 "재고 조회", "주문 등록", "배송 추적" 같은 자사 기능을 도구로 만들어 에이전트에 쥐여줌.

### 실무 포인트
- 에이전트의 능력 = 도구의 품질. 어떤 도구를 주느냐가 무엇을 할 수 있는지를 결정함.
- 도구 하나는 "한 가지 일"만 하게 작게 만드는 편이 모델이 고르기 쉬움(7-5에서 다룸).

### 3줄 요약
1. 도구는 에이전트가 호출하는 하나의 기능(검색·계산·조회 등)임.
2. 모델은 두뇌, 도구는 그 두뇌에 다는 손과 발임.
3. 도구는 결국 평범한 함수에 모델용 설명을 붙인 것임.

## 7-2. @tool 데코레이터 — 함수를 도구로

### 개념

평범한 함수를 도구로 바꾸는 가장 기본은 **`@tool` 데코레이터**임. 함수 위에 한 줄 붙이면, LangChain이 그 함수를 에이전트가 쓸 수 있는 도구로 포장해줌.

```python
from langchain_core.tools import tool

@tool
def get_stock(product: str) -> str:
    """상품명을 받아 현재 재고 수량을 돌려준다."""
    # 실제로는 DB를 조회하겠지만, 여기선 예시
    stock_db = {"베이지 니트": 12, "트렌치 코트": 0}
    qty = stock_db.get(product, "정보 없음")
    return f"{product}의 재고: {qty}"
```

이제 `get_stock`은 에이전트에 넘길 수 있는 도구임.

```python
agent = create_agent(model, tools=[get_stock])
```

### 비유

`@tool`은 **명함을 만들어 붙이는 것**과 같음. 평범한 직원(함수)에게 "이름: 재고조회, 하는 일: 상품 재고를 알려줌"이라는 명함을 달아주면, 에이전트(관리자)가 그 명함을 보고 "재고를 알아야 할 땐 이 직원을 부르면 되겠군"이라고 판단함.

### 원리

`@tool`은 함수를 검사해 **세 가지를 자동으로 추출**함.

1. **이름**: 함수 이름(`get_stock`)
2. **설명**: docstring(`"상품명을 받아 현재 재고 수량을 돌려준다."`)
3. **입력 명세**: 인수와 타입 힌트(`product: str`)

이 셋이 모델에게 전달되는 "도구 사용 설명서"가 됨. 에이전트는 이 설명서만 보고 도구를 고르고 호출함 — **함수 내부 코드는 모델이 보지 않음.**

### 📌 참고
> 도구의 반환값은 보통 **문자열**로 두는 것이 안전함(모델이 읽어야 하므로). 복잡한 객체를 돌려줘야 하면 문자열로 정리해 반환하는 편이 모델이 결과를 이해하기 쉬움.

### 실무 포인트
- `@tool` 한 줄이면 기존 함수를 거의 그대로 도구화할 수 있어, 자사 기능을 빠르게 에이전트에 붙임.
- 함수 이름은 동사+명사로 명확히(`get_stock`, `send_email`). 모델이 이름만으로도 용도를 짐작함.

### 3줄 요약
1. `@tool` 데코레이터 한 줄이 함수를 도구로 만듦.
2. 함수의 이름·docstring·타입 힌트가 자동으로 추출돼 도구 명세가 됨.
3. 모델은 명세만 보고 도구를 고름 — 내부 코드는 보지 않음.

## 7-3. docstring과 시그니처가 곧 명세 — 모델은 설명을 읽는다

### 개념

이 파트에서 가장 중요한 사실 — **모델이 도구를 고르는 근거는 docstring과 함수 시그니처임.** 코드가 무슨 일을 하든, 모델은 그 설명만 읽고 판단함. 그래서 **docstring을 잘 쓰는 것이 곧 도구 설계의 핵심**임.

### 비유

도구의 docstring은 **공구함에 붙은 라벨**임. 모델은 공구함을 열어 각 공구를 직접 써보는 게 아니라, **라벨만 읽고** "이 일엔 이 공구"라고 고름. 라벨이 "나사용 드라이버"라고 정확하면 제대로 고르고, "도구 1"처럼 모호하면 엉뚱한 걸 집음.

### 원리 — 같은 함수, 다른 docstring

다음 두 도구는 내부 코드가 같아도, 모델이 고르는 정확도가 완전히 다름.

```python
# ❌ 나쁜 예 — 모델이 언제 써야 할지 모름
@tool
def func1(x: str) -> str:
    """처리한다."""
    ...

# ✅ 좋은 예 — 모델이 정확히 판단함
@tool
def get_exchange_rate(currency: str) -> str:
    """원화 대비 환율을 조회한다. currency는 'USD', 'EUR' 같은 통화 코드.
    해외 상품 가격을 원화로 환산할 때 사용한다."""
    ...
```

모델은 `func1`의 "처리한다"만 보고는 무슨 일인지 알 수 없어, 환율이 필요한 질문에도 이 도구를 못 고르거나 엉뚱하게 부름. 반면 `get_exchange_rate`는 **무엇을·언제 쓰는지**가 명확해 모델이 정확히 호출함.

### 📌 참고
> docstring에 "**언제 쓰는지**"를 적는 것이 특히 강력함. "환율을 조회한다"보다 "해외 상품 가격을 원화로 환산할 때 사용한다"가 모델의 판단을 크게 돕는다. 모델은 사용자 의도와 docstring을 매칭하기 때문임.

### 실무 포인트
- 에이전트가 "엉뚱한 도구를 부른다 / 도구를 안 부른다"의 1순위 원인은 **docstring 부실**임.
- docstring은 모델에게 보내는 프롬프트의 일부라고 생각하고, 프롬프트 쓰듯 정성껏 작성함.

### 3줄 요약
1. 모델은 docstring과 시그니처만 보고 도구를 고름(내부 코드 아님).
2. docstring에 "무엇을·언제" 쓰는지 명확히 적는 것이 핵심임.
3. 도구 오작동의 1순위 원인은 부실한 docstring임.

## 7-4. 타입 힌트와 args_schema — 입력 구조를 명확히

### 개념

도구의 **인수와 타입 힌트**도 모델에게 전달되는 명세의 일부임. 타입을 명확히 하면 모델이 올바른 형식으로 값을 넣음. 인수가 복잡하면 Pydantic 모델로 입력 구조를 정의해(`args_schema`) 더 정밀하게 통제함.

```python
from langchain_core.tools import tool

# 타입 힌트 + 인수별 설명
@tool
def search_products(category: str, max_price: int) -> str:
    """카테고리와 최대 가격으로 상품을 검색한다.

    Args:
        category: 상품 카테고리 (예: '니트', '코트')
        max_price: 최대 가격 (원 단위 정수)
    """
    ...
```

더 정밀하게는 Pydantic으로 입력 스키마를 정의함.

```python
from pydantic import BaseModel, Field

class SearchInput(BaseModel):
    category: str = Field(description="상품 카테고리")
    max_price: int = Field(description="최대 가격(원)", gt=0)

@tool(args_schema=SearchInput)
def search_products(category: str, max_price: int) -> str:
    """조건에 맞는 상품을 검색한다."""
    ...
```

### 비유

타입 힌트는 **신청서 칸에 적힌 형식 안내**임. "가격(숫자만)", "카테고리(텍스트)"라고 칸마다 형식이 적혀 있으면, 작성자(모델)가 엉뚱한 형식으로 적는 실수가 줄어듦. Pydantic `Field`는 거기에 "0보다 커야 함" 같은 검증 규칙까지 더한 것임.

### 원리

`@tool`은 함수의 타입 힌트를 읽어 **각 인수가 어떤 타입인지**를 모델에게 알려줌. 모델은 도구를 호출할 때 이 명세에 맞춰 인수를 채움(예: `max_price`엔 숫자). `args_schema`로 Pydantic 모델을 주면, 각 인수의 설명(`Field(description=...)`)과 제약(`gt=0` 등)까지 전달돼 더 정확하고 안전한 호출이 됨 — Part 4의 구조화 출력과 같은 발상임.

### 📌 참고
> 인수의 설명은 Args 섹션(docstring) 또는 `Field(description=...)`로 적음. 인수가 1~2개로 단순하면 타입 힌트 + docstring Args로 충분하고, 인수가 많거나 검증이 필요하면 `args_schema`를 씀.

### 실무 포인트
- 모델이 인수를 자꾸 틀린 형식으로 넣으면, 타입 힌트와 인수 설명을 더 명확히 하면 대개 해결됨.
- 숫자 범위·필수 값 등 제약이 중요하면 Pydantic `Field`로 못 박아 안전성을 높임.

### 3줄 요약
1. 인수와 타입 힌트도 모델에게 전달되는 명세임.
2. 단순하면 타입 힌트+docstring Args, 복잡하면 `args_schema`(Pydantic).
3. 인수 설명·제약을 명확히 하면 호출 형식 오류가 줄어듦.

## 7-5. 좋은 도구 설계 — 에이전트가 헛돌지 않게

### 개념

도구를 잘 만드는 것은 에이전트 품질의 절반임. 핵심 원칙 몇 가지를 정리함.

**좋은 도구의 조건**
1. **한 가지 일만**: 도구 하나는 단일 기능. "검색하고 정리하고 메일도 보내는" 만능 도구는 모델이 언제 쓸지 혼란스러워함.
2. **이름이 곧 용도**: `get_stock`, `send_email`처럼 동사+명사로 명확히.
3. **docstring에 '언제'를 명시**: "~할 때 사용한다"를 적어 모델의 판단을 도움.
4. **반환값은 모델 친화적으로**: 모델이 읽고 다음 행동을 정할 수 있게 명확한 문자열로. 에러도 숨기지 말고 "재고 정보 없음"처럼 알려줌.
5. **부작용에 주의**: 메일 발송·결제처럼 되돌릴 수 없는 도구는 신중히. (고급의 human-in-the-loop·Part 14와 연결됨)

### 비유

좋은 도구함은 **잘 정리된 주방**과 같음. 칼은 칼, 국자는 국자로 용도가 분명하고 라벨이 붙어 있으면 요리사(모델)가 막힘없이 일함. 반대로 "다용도 만능 도구" 하나에 모든 기능을 욱여넣으면, 정작 필요할 때 어떻게 써야 할지 헷갈림.

### 원리

에이전트가 헛도는(엉뚱한 도구 호출, 무한 반복) 대부분의 원인은 **도구 설계 문제**로 거슬러 올라감. 도구가 너무 크거나, 설명이 모호하거나, 반환값이 불명확하면 모델이 올바른 판단을 못 함. 즉 **에이전트를 똑똑하게 만드는 일의 상당 부분은 모델 튜닝이 아니라 도구를 명확히 설계하는 일**임.

### 📌 참고
> 도구가 많아질수록(10개 이상) 모델이 고르기 어려워짐. 이럴 땐 도구를 묶거나, 단계를 나누거나, 여러 전문 에이전트로 분리하는 설계(멀티에이전트·Part 15)를 고려함. 지금은 "도구는 적고 명확하게"가 좋은 시작임.

### 실무 포인트
- 새 도구를 만들면, 모델 입장에서 "이 설명만 보고 언제 쓸지 알 수 있나?"를 자문함.
- 되돌릴 수 없는 행동(삭제·발송·결제)은 도구 안에서 함부로 실행하지 말고, 확인 절차나 human-in-the-loop를 둠.

### 3줄 요약
1. 좋은 도구는 단일 기능·명확한 이름·'언제' 쓰는지 명시·친화적 반환값임.
2. 에이전트가 헛도는 원인의 다수는 도구 설계 문제임.
3. 도구는 적고 명확하게 — 많아지면 분리·멀티에이전트를 고려함.

## Part 7 마무리 — 다음으로

Part 7에서는 에이전트의 손과 발을 만드는 법을 익혔음. 정리하면:

- 도구는 에이전트가 외부 세계와 만나는 접점(손과 발) (7-1)
- `@tool` 데코레이터로 함수를 도구로, 이름·docstring·타입이 자동 명세화 (7-2)
- 모델은 docstring과 시그니처만 읽고 도구를 고름 — 설명이 곧 설계 (7-3)
- 타입 힌트·`args_schema`로 입력 구조를 명확히 (7-4)
- 좋은 도구 설계 원칙 — 단일 기능·명확·친화적 (7-5)

이제 에이전트에 손발을 달 수 있게 됐음. 하지만 가장 강력하고 자주 쓰이는 도구가 아직 빠졌음 — **"우리 문서에서 찾아 답하는" 능력**임. 에이전트가 사내 매뉴얼·상품 DB·정책 문서를 근거로 답하려면, 그 문서들을 검색 가능한 형태로 만들어 두어야 함. **Part 8**에서는 이 핵심 기술인 **RAG(검색 증강 생성)**의 기초를 배움 — 문서 로더 → 청킹 → 임베딩 → 벡터스토어 → retriever로 이어지는, "에이전트에게 지식을 주는" 파이프라인임.

> 함께 제공되는 **Part 7 실습 노트북**에서는 리리마켓 도메인의 도구(재고 조회·환율 계산 등)를 `@tool`로 직접 만들어 에이전트에 붙이고, **docstring을 좋게/나쁘게 바꿔가며** 모델의 도구 선택이 어떻게 달라지는지 직접 관찰함. `args_schema`로 입력을 통제하는 실습까지 Gemini로 진행함.
