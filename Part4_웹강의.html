파서 — 출력의 마지막 공정

모델 출력은 텍스트뿐이라, 프로그램이 쓰려면 누군가 파싱(해석)해야 함. 출력 파서는 그 텍스트를 원하는 형태(문자열·딕셔너리·객체)로 바꾸는 마지막 블록임. 손글씨 주문서를 전산 폼으로 옮기는 사무원과 같음.

파서도 Runnable이라 파이프 끝에 | parser로 붙이면 됨.

  3. 파서도 Runnable이라 | parser로 이어 붙임.

StrOutputParser — 포장 벗기기

StrOutputParser는 AIMessage라는 포장을 벗기고 안의 본문 문자열만 손에 쥐여 줌. Part 3에서 (lambda m: m.content)로 하던 일을 정식 블록으로 만든 것임.

인터랙티브 · 포장 벗기기

상자(AIMessage)를 열어 알맹이(str)만 꺼내봄.

AIMessage 포장
📦 닫힌 상자
content="포근함을 입다"
📌 참고  결과가 str이라 이후 텍스트 가공(해시태그 변환 등)을 자연스럽게 이을 수 있음. 스트리밍에서도 chunk가 바로 출력할 문자열이 됨.
3줄 요약
  1. StrOutputParser는 AIMessage에서 본문 문자열만 추출함.
  2. 결과 타입이 str이라 이후 가공이 쉬움.
  3. 사람에게 보여줄 답을 만드는 체인의 기본 마무리임.
4-3 · 4-5

format_instructions — 앞뒤가 맞물린다

구조화 파서가 작동하려면 모델이 먼저 그 형식대로 답해야 함. get_format_instructions()는 파서가 모델에게 "이 형식으로 답하라"고 주는 설명서임 — 택배 송장 양식 안내문과 같음.

인터랙티브 · 양방향 흐름 따라가기

"단계 진행"을 누르며, 파서가 어떻게 앞에서는 형식을 주문하고 뒤에서는 해석하는지 봄.

1📤파서 → 모델: get_format_instructions()가 "JSON 스키마에 맞춰 답하라"는 지시문 생성 → 프롬프트에 끼움
↓
2🤖모델: 지시대로 JSON 텍스트로 답함 → '{"name":"니트", "codi":[...]}'
↓
3📥모델 → 파서: 파서가 그 JSON 텍스트를 해석해 파이썬 dict로 변환 ✅
📌 참고  with_structured_output을 쓰면 이 과정을 우리가 직접 챙기지 않아도 됨(모델 기능으로 내부 처리). 그래도 원리를 알면 형식이 깨질 때 디버깅이 쉬워짐.
3줄 요약
  1. format_instructions는 파서가 모델에게 주는 형식 설명서임.
  2. 프롬프트에 끼우면 모델이 형식을 맞추고 파서가 해석함(한 짝).
  3. 누락이 형식 깨짐의 1순위 원인임.
4-4

Pydantic · with_structured_output — 타입 보장

JSON으로 받아도 "가격은 숫자여야 하는데 문자열로 왔다" 같은 문제가 생김. Pydantic 모델로 구조·타입을 클래스로 정의하고 with_structured_output을 쓰면, 검증된 객체가 바로 손에 들어옴 — 입국 신고서 양식처럼 형식이 틀리면 반려됨.

인터랙티브 · dict vs 검증된 객체

두 방식이 어떻게 다른지 비교함. 같은 데이터라도 안전성이 다름.

JsonOutputParser → dict
with_structured_output → 객체
📌 참고  with_structured_output은 모델의 구조화 출력 기능을 직접 써서 사후 파싱보다 안정적임. Field(description=...)을 성의 있게 적을수록 모델이 각 칸을 정확히 채움 — 설명이 곧 지시임.
3줄 요약
  1. Pydantic 모델로 출력의 구조·타입을 클래스로 정의함.
  2. with_structured_output이 구조를 강제하고 검증된 객체를 돌려줌(1.0 권장).
  3. DB 저장·분기·API 연동 등 출력을 코드가 쓸 때 핵심임.
