0-1

LLM을 직접 부르면 생기는 문제

LLM(대규모 언어 모델)은 그 자체로는 "질문을 넣으면 글자를 뱉는 함수" 하나에 불과함. 직접 호출하면 프롬프트 조립, 응답 파싱, 모델 교체, 대화 기억, 행동 제어를 전부 손으로 해야 함.

LLM 직접 호출은 전기 부품을 낱개로 사서 직접 납땜하는 것과 같음. 작동은 하지만 부품 하나를 바꾸려면 회로 전체를 다시 손봐야 함. LangChain은 그 부품들을 표준 규격의 블록으로 만들어, 모델만 쏙 빼서 갈아 끼워도 나머지가 그대로 돌아가게 함.

인터랙티브 · 모델을 바꿔보기

"OpenAI로 만든 코드를 Gemini로 바꾼다"고 가정함. 방식을 눌러 무엇이 무너지는지 비교해봄.

    📌 참고  LangChain의 모든 부품은 Runnable이라는 공통 규격을 따름. "입력을 받아 출력을 내는 모든 것은 invoke()라는 똑같은 메서드를 가진다"는 약속임. 이 약속 덕분에 부품을 파이프(|)로 이을 수 있고, 그 문법이 곧 LCEL임.
    3줄 요약
    1. LLM은 그 자체로는 "글자를 뱉는 함수" 하나일 뿐임.
    2. 직접 호출하면 프롬프트·파싱·모델 교체·기억·행동을 전부 손으로 함.
    3. LangChain은 이를 표준 블록으로 묶어 갈아 끼우고 이어 붙이게 함.

    LangChain 1.0 — 무엇이 달라졌는가

    LangChain은 2025년 10월 1.0 버전으로 올라서며 구조가 정리됨. 이 강의는 1.0 기준임. 핵심은 한 문장으로 요약됨.

    "LangChain은 에이전트를 빠르게 조립하는 층,
    LangGraph는 그 에이전트를 안정적으로 돌리는 엔진이다."

    📌 참고  1.0의 가장 중요한 변화: 모든 LangChain 에이전트는 이제 LangGraph 런타임 위에서 실행됨. 중급의 create_agent()는 겉보기엔 LangChain 함수지만, 내부에서 LangGraph가 상태를 들고 도구 호출 루프를 돌림. 옛 자료의 AgentExecutor 방식은 1.0 이전 구형 패턴임.
    3줄 요약
    1. 이 강의는 2025년 10월 출시된 LangChain 1.0 기준임.
    2. 핵심은 "모든 에이전트가 LangGraph 런타임 위에서 돈다"임.
    3. 따라서 초·중·고급은 같은 엔진을 점점 깊이 보는 하나의 길임.

    세 개의 층 — LCEL · 에이전트 · LangGraph

    앞으로 배울 모든 것은 세 층 위에 있음. 이 강의의 초급/중급/고급이 정확히 이 세 층과 일대일로 대응함. 차이는 결국 "길 안내 방식"의 차이임.

    LCEL

    📌 참고  실무 규칙: LCEL로 시작하라. 그러다 "반복", "조건 분기", "여러 단계 상태 유지"라는 벽에 부딪히면 그때 LangGraph로 넘어가면 됨. 처음부터 LangGraph를 쓰는 것은 동네 마트 갈 때 트럭을 모는 것과 같음.
    3줄 요약
    1. LCEL(정해진 길) → 에이전트(판단) → LangGraph(직접 설계)가 곧 초·중·고급임.
    2. 경계는 "정해진 흐름이면 충분한가, 동적 판단이 필요한가"임.
    3. 항상 LCEL로 시작해 벽에 부딪히면 LangGraph로 올라가는 것이 정석임.

    이 강의를 다 들으면 만들 수 있는 것

    종착점은 "스스로 판단하고, 필요하면 사람의 승인을 받으며, 여러 전문가가 협업하고, 무슨 일을 했는지 기록으로 남기는 에이전트"임. 아래 로드맵의 단계를 눌러 전체 여정을 펼쳐봄.

    인터랙티브 · 전체 여정 로드맵
    📌 참고  각 파트는 이론 문서(개념·문법·동작 원리)와 실습 노트북(직접 돌려보는 Gemini 코드) 두 벌로 구성됨. 한 파트라도 건너뛰면 다음 파트의 비유와 코드가 허공에 뜸.
    3줄 요약
    1. 종착점은 판단·승인·협업·기록을 갖춘 완성형 에이전트임.
    2. 길은 초급(재료) → 중급(판단) → 고급(제어)으로 누적됨.
    3. 각 파트는 이론 문서 + Gemini 실습 노트북 두 벌로 진행됨.
